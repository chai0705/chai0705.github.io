3.20.1 U-Boot 启动方式

下面是关于 U-Boot 启动方式的解释：

U-Boot 是一种开源的引导加载程序，用于在嵌入式系统上启动操作系统。它具有广泛的硬件支持和灵活的配置选项。U-Boot 可以从多种存储介质启动，包括 NAND、NOR、SD 卡和 SDRAM。

  1. NAND 启动：
    NAND 闪存是一种非易失性存储介质，常用于嵌入式系统中。当 U-Boot 从 NAND 启动时，它会从 NAND 闪存芯片读取引导程序（bootloader）并执行。
  2. NOR 启动：
    NOR 闪存是另一种常见的非易失性存储介质，通常用于存储 U-Boot。在 NOR 启动方式下，U-Boot 位于 NOR 闪存中，并且系统会直接从 NOR 闪存中加载和执行 U-Boot。
  3. SD 卡启动：
    SD 卡是一种可移动存储介质，常用于嵌入式系统中。当 U-Boot 从 SD 卡启动时，它会从 SD 卡中读取引导程序并执行。
  4. SDRAM 启动：
    SDRAM 是系统内存，U-Boot 可以从 SDRAM 启动。在 SDRAM 启动方式下，U-Boot 首先加载到 SDRAM 中，然后执行。

在启动过程中，U-Boot 需要知道引导程序的存储位置和大小。根据您提供的信息，我理解以下是相关的地址范围和存储介质：

  • NAND：存储介质为 NAND 闪存。
  • NOR：存储介质为 NOR 闪存。
  • SD 卡：存储介质为 SD 卡。
  • SDRAM：存储介质为系统内存。

3.20.2 U-Boot生成文件的一般过程：

在 U-Boot 的编译过程中，会生成不同的文件，其中包括以下常见的文件格式：

  • ELF 格式：U-Boot 的主要可执行文件通常以 ELF（Executable and Linkable Format）格式存储。ELF 文件包含了可执行代码、数据和符号表等信息，可用于调试和符号解析。
  • BIN 格式：U-Boot 还可以使用 objcopy 工具将 ELF 格式的可执行文件转换为 BIN（Binary）格式。BIN 格式是一种简化的二进制文件格式，它只包含可执行代码，没有调试信息和符号表。
  • Symbol 文件：U-Boot 生成的符号文件（例如 u-boot.sym）包含了代码中定义的符号信息，这对于调试和符号解析非常有用。符号文件通常与 ELF 格式文件一起使用。
  • System.map 文件：U-Boot 在编译过程中还会生成 System.map 文件，它包含了 U-Boot 的符号表和地址映射信息。

3.20.3 uboot镜像依赖关系

解释如下：

  1. u-boot-init：这是一个初始文件或函数，可能包含一些启动代码或初始化代码。
  2. start.o：这是一个目标文件，位于 arch/arm/cpul/armv7 目录中。它可能包含与 ARMv7 架构相关的启动代码。
  3. u-boot-main：这是一个主要的 U-Boot 文件或函数，可能包含 U-Boot 的核心功能。
  4. built-in.o：这是一个目标文件，位于 xx/xxx 目录中。它可能包含 U-Boot 中的一些功能模块或驱动程序。
  5. u-boot.lds：这是 U-Boot 的链接脚本文件，用于指定链接器如何将目标文件组合成最终的可执行镜像。

3.20.4 uboot启动流程

U-Boot 的启动流程可以大致描述如下：

  1. 硬件上电或复位后，处理器会跳转到预定义的启动地址，通常是 SoC 内部的 ROM 或 Flash 存储器中的特定位置。
  2. 启动地址处的代码通常是一段汇编代码，用于设置处理器的初始状态和寄存器，然后跳转到 U-Boot 的入口点。
  3. 在 U-Boot 的入口点，首先执行的是 arch/arm/lib/vectors.S 文件中的 reset 函数。该函数主要完成一些处理器和中断控制器的初始化工作。
  4. reset 函数会调用 arch/arm/cpu/armv7/start.S 文件中的 mair_init 函数。mair_init 主要用于设置内存访问控制和缓存配置等相关设置。
  5. 接下来，mair_init 函数会调用 arch/arm/lib/crt0.S 文件中的 relocate_code 函数。relocate_code 函数负责将 U-Boot 的代码从其加载地址（通常是 Flash 存储器）复制到 RAM 中，并更新相关的地址。
  6. 在 relocate_code 函数执行完毕后，控制权将转移到 U-Boot 的入口函数（通常是 board_init_f 或 main_loop 函数），从而开始执行 U-Boot 的主要功能。

3.20.5 重定位之前准备工作

在 U-Boot 的重定位之前，通常会进行一些准备工作。根据您提供的信息，这些准备工作可能包括以下内容：

在 start.S 文件中：

  • 设置 CPU 为 SVC（Supervisor）模式。
  • 关闭缓存和关闭 MMU（Memory Management Unit）。
  • 设置关门狗（Watchdog）、屏蔽中断、设置时钟等初始化操作。
  • 初始化 SDRAM（同步动态随机存取存储器）。

在 crt0.S 文件中：

  • 初始化 C 语言运行环境，包括堆栈设置等。
  • 进行板级各种初始化操作，例如初始化 NAND、SDRAM 等外设。
  • 初始化全局结构体变量 GD（Global Data），并在 GD 中保存 U-Boot 的实际加载地址。
  • 调用 relocate_code 函数，将 U-Boot 镜像从 Flash 拷贝到内存中。
  • 从 Flash 跳转到 RAM 空间，继续执行 U-Boot 程序。
  • 清零 BSS 段（未初始化的全局变量）、进行板级初始化操作。
  • 最后进入 bootcmd 或 main_loop 循环，执行 U-Boot 的主要功能。
