第一章 可加载模块的概念

1.1 微内核和宏内核

微内核(Microkernel)架构:

  1. 设计理念:
    • 将操作系统的核心功能最小化,只实现最基础的机制,如进程管理、内存管理、中断处理等。
    • 将其他功能如文件系统、网络协议栈等作为独立的服务运行在用户空间。
  2. 优点:
    • 内核体积小,可靠性高。内核崩溃不会导致整个系统崩溃。
    • 可扩展性强,可以方便地添加新的功能作为服务。
    • 各个服务之间相对独立,可以灵活地组合和替换。
  3. 缺点:
    • 进程间通信开销大,各服务之间的交互效率较低。
    • 由于服务之间的依赖关系,系统启动和初始化较复杂。
  4. 代表系统:
    • Mach 内核
    • L4 微内核
    • QNX 操作系统

宏内核(Monolithic Kernel)架构:

  1. 设计理念:
    • 将操作系统的所有功能模块都集成到内核中,形成一个大而全的内核。
    • 各功能模块之间直接调用,无需进程间通信。
  2. 优点:
    • 功能全面,性能较高。各模块之间的交互效率高。
    • 系统启动和初始化较为简单。
  3. 缺点:
    • 内核体积较大,可靠性相对较低。内核崩溃会导致整个系统崩溃。
    • 扩展性较差,添加新功能需要修改内核源码。
  4. 代表系统:
    • Linux 内核
    • Windows NT 内核
    • FreeBSD 内核

​ ==程序的运行状态可以分为内核态和用户态，一般来说用户态和内核态的切换会造成很大的性能损耗，所以Linux的宏内核相较于微内核会有更高的性能==

​ 总的来说,微内核和宏内核是两种不同的操作系统内核设计思路,各有优缺点。微内核追求可靠性和可扩展性,而宏内核则更注重性能和便利性。实际应用中,不同的操作系统会根据具体需求选择适合的架构。

1.2 Linux内核的模块机制

  1. 内核模块化:
    • Linux 内核采用模块化设计,将不同的功能组件(如驱动程序、文件系统、网络协议栈等)封装为可动态加载和卸载的内核模块。
    • 这种模块化设计提高了内核的灵活性和可扩展性,使得内核能够适应不同的硬件平台和应用场景。
  2. 可定制和裁剪:
    • 由于内核采用模块化设计,用户或发行商可以根据需求选择性地编译和加载内核模块,裁剪掉不需要的功能。
    • 这样可以减小内核的体积,提高性能,同时也降低了维护和定制的成本。
  3. 支持不同硬件平台:
    • Linux 内核支持广泛的硬件架构,如x86、ARM、MIPS等。
    • 通过模块化设计,内核可以适配不同的硬件平台,加载对应的硬件驱动模块。
  4. 动态加载和卸载:
    • Linux 内核支持在运行时动态加载和卸载内核模块,无需重新编译和重启内核。
    • 用户可以使用 insmod、rmmod、modprobe 等命令来管理内核模块。
    • 内核还提供了编程接口,如 request_module()、try_module_get() 等,供内核其他部分动态加载模块。
  5. 模块依赖管理:
    • 内核模块之间可能存在依赖关系,比如某些模块需要依赖其他模块提供的符号和功能。
    • 内核的模块依赖管理机制可以自动解决模块之间的依赖关系,确保模块能够正确加载和卸载。
  6. 模块参数和配置:
    • 内核模块可以提供可配置的参数,用户在加载模块时可以通过命令行参数进行设置。
    • 模块内部通过 module_param() 等宏定义参数,并提供参数说明。

总的来说,Linux 内核的模块机制是其设计的核心之一,它使得内核具备了高度的可定制性和可扩展性,能够适应不同的硬件平台和应用场景,并且支持在运行时动态管理内核组件,大大提高了内核的灵活性和性能。

第2章 内核模块的构成

  1. 模块头文件:
    • 这个文件通常命名为 module.h,包含了内核提供的各种宏和函数声明。
    • 它定义了内核模块所需要的基本结构和接口,如模块初始化和退出函数、导出符号等。
  2. 模块初始化函数:
    • 这个函数在模块被加载时被内核调用,负责执行模块的初始化操作。
    • 通常使用 module_init() 宏来定义这个函数。
  3. 模块退出函数:
    • 这个函数在模块被卸载时被内核调用,负责执行模块的退出操作,如释放资源等。
    • 通常使用 module_exit() 宏来定义这个函数。
  4. 模块信息:
    • 这部分包含了模块的描述性信息,如模块名称、版本号、作者等。
    • 这些信息通常使用 MODULE_*() 系列宏来定义。
  5. 模块许可信息:
    • 这部分描述了模块的许可协议,如 GPL、MIT 等。
    • 使用 MODULE_LICENSE() 宏来定义。
  6. 模块依赖信息:
    • 这部分描述了模块所依赖的其他内核模块或符号。
    • 使用 MODULE_DEPEND() 或 MODULE_IMPORT_NS() 宏来定义。
  7. 模块参数:
    • 这部分定义了模块可配置的参数,可以在加载模块时传入参数值。
    • 使用 module_param() 或 module_param_array() 宏来定义。
  8. 模块实现:
    • 这部分包含了模块的具体实现代码,如初始化函数、中断处理函数、文件操作函数等。
    • 这部分代码需要与内核提供的 API 进行交互,实现模块的功能。

总的来说,内核模块的构成包括了初始化和退出函数、模块信息、许可信息、依赖信息、可配置参数以及具体的实现代码。这些部分共同组成了一个完整的内核模块,可以被内核加载和使用。

第3章 内核许可声明

内核许可声明详解

什么是内核许可声明？

内核许可声明是用于描述内核模块的许可权限定义。内核模块是可以加载到内核中的动态可插拔组件，能够扩展内核的功能。为了确保模块的合法性和兼容性，需要对其许可进行声明。

内核许可类型

  1. GPL (GNU General Public License):
    • 最常见的开源许可证之一，要求所有基于GPL代码的衍生作品也必须采用GPL许可证发布。
    • 这意味着如果你的模块是基于GPL代码编写的，你的模块也必须开源。
  2. GPL v2:
    • 内核主要采用GPL v2版本发布，明确规定了模块和内核的许可关系。
  3. Dual BSD/GPL:
    • 允许模块在BSD或GPL任意一种许可下发布，给予开发者更大的灵活性。
  4. Proprietary:
    • 私有模块，源代码不公开，通常为商业软件。
    • 这种模块加载时，内核会被标记为“tainted”状态。

许可声明的作用

  1. 明确许可权:
    • 用来描述内核的许可权限：内核以GPL发布。
    • 通过声明，用户和开发者可以清楚了解模块的法律使用范围。
  2. 避免污染内核:
    • 模块不声明LICENSE，内核会有(kernel tainted)警告。
    • 内核状态此时是受污染的（tainted），意味着加载了非GPL兼容的模块。
  3. 维护系统稳定性:
    • 内核受污染后，一些调试、打印功能可能会失效。
    • 通过许可证声明，可以避免加载不兼容的模块，从而保证系统的稳定性。

许可证声明的影响

  1. 内核状态标记:
    • 当加载非GPL兼容的模块时，内核会被标记为“tainted”，表示内核状态受污染，可能存在不稳定因素。
    • 这种标记会影响内核调试和错误报告功能。
  2. 模块发布与使用:
    • 开发者必须根据模块的实际使用情况和依赖关系，选择合适的许可证进行声明。
    • 如果模块是基于GPL代码开发的，那么必须使用GPL许可证声明，否则会违反GPL协议。
  3. 法律合规性:
    • 正确的许可声明可以帮助开发者和企业避免法律纠纷，确保软件发布和使用的合法性。

总结来说，内核许可声明对于内核模块的开发、发布和使用至关重要。通过合理的许可声明，可以确保内核的稳定性、兼容性和法律合规性。

第四章 内核污染

内核被污染的原因:

  1. 加载一些不开源、跟GPL不兼容的驱动:
    • 内核采用GPL开源许可,所以加载一些闭源的第三方驱动会导致内核被污染。这些驱动可能是一些硬件厂商提供的专有驱动。
  2. Staging内核模块:
    • Staging区是内核中一些新加入但还不稳定的驱动和子系统。这些模块虽然还在测试阶段,但也可能会导致内核被污染。
  3. 一些out-of-tree模块的加载运行:
    • 内核源码树之外的第三方内核模块,比如内核补丁、内核扩展等,也可能会导致内核被污染。
  4. 内核编译时依赖的内核版本和运行版本不一致:
    • 如果编译内核时使用的内核版本与运行时的内核版本不匹配,也会导致内核被污染。
  5. 支持SMP的内核在早期不支持多核的单核CPU上运行:
    • 早期的SMP内核版本在单核CPU上运行也会导致内核被污染。
  6. BIOS或UEFI中的一些bug:
    • BIOS或UEFI中的硬件相关bug也可能导致内核被污染。
  7. 内核启动时的一些机器检查或OOPS:
    • 内核启动时的硬件兼容性检查或内核OOPS也可能会导致内核被污染。

内核被污染后的影响:

  1. 一些调试功能、输出失效:
    • 当内核被污染后,内核中一些调试功能和输出信息可能会受到影响而无法正常工作。
  2. 一些API系统调用失效:
    • 内核API和系统调用可能会受到影响而无法正常使用。
  3. 社区一般不会处理tainted kernel下的bug:
    • 当内核被污染时,社区通常不会处理由此引起的bug和问题。开发者可能会要求先解决内核污染问题再提供进一步的技术支持。

总的来说,内核污染通常是由于加载不受支持的驱动或模块、硬件兼容性问题等原因导致的。解决内核污染问题的关键是要识别并移除导致内核被污染的因素,同时还要注意内核版本的一致性。社区也通常不会处理tainted kernel下的问题,所以尽量避免内核被污染是很重要的。

第7章 模块的Makefile分析

​ 在Linux内核的编译过程中,Makefile和kbuild是密切相关的两个部分:

  1. Makefile:
    • Makefile是传统的Make工具使用的构建脚本。
    • 它定义了编译内核的主要规则和依赖关系。
    • Makefile会包含一些全局变量,如ARCH、CROSS_COMPILE等,用于控制编译过程。
    • 在Makefile中,内核模块的编译规则通常是使用obj-y和obj-m变量来指定。
  2. kbuild:
    • kbuild是Linux内核特有的构建系统,是基于Makefile之上的一个抽象层。
    • kbuild为内核模块的编译提供了更加灵活和可配置的机制。
    • kbuild利用了Kconfig系统来管理内核的配置选项,并将其与Makefile中的编译规则关联起来。

    • EXPORT_SYMBOL只能用于导出全局符号,不能用于导出静态符号。
    • 过度导出符号可能会导致内核体积增大,影响性能,因此需要谨慎使用。
    • 在内核编译时,会自动生成一个Module.symvers文件,其中记录了所有被导出的符号信息。

这个Makefile的主要部分包括:

  1. 内核版本判断

    • 检查是否在内核编译环境中,通过判断 KERNELRELEASE 环境变量是否定义。

  2. 内核模块编译目标

    • 在内核编译环境中, obj-m := hello.o
    • 将 hello.o 作为可加载模块进行编译。

  3. 独立编译环境设置

    • 如果不在内核编译环境中:

      • EXTRA_CFLAGS += -DDEBUG


- 设置额外的符号文件路径。


      • export KBUILD_EXTRA_SYMBOLS


- 设置内核源码目录。


      • ARCH_ARGS := ARCH=arm CROSS_COMPILE=arm-linux-gnueabi-





    • clean:
	make $(ARCH_ARGS) -C $(KDIR) M=$(PWD) modules clean

    • 这些目标用于在独立编译环境中构建和清理内核模块。

总的来说,这个Makefile的设计目的是为了适应两种不同的编译环境:内核编译环境和独立编译环境。通过判断 KERNELRELEASE 环境变量,Makefile 可以自动切换到合适的构建规则,确保内核模块能够在不同环境下正确编译。同时,它还添加了一些额外的编译参数和符号文件路径,以满足特定的编译需求。

内核镜像的生成过程以及后续压缩成 Image.gz 的过程可以分为以下几个步骤：

Image 生成过程

  1. 配置内核
    • 使用 make menuconfig 或其他配置工具选择适当的内核选项。
    • 配置文件通常保存为 .config。
  2. 编译内核
    • 运行 make 命令，开始编译内核。
    • 编译过程主要包括：
      • 预处理：解析头文件和宏定义。
      • 编译：将 C 语言代码编译为汇编。
      • 汇编：将汇编代码转换为目标代码。
      • 链接：将目标文件链接为单个内核镜像 vmlinux。
  3. 生成 Image
    • vmlinux 是一个未压缩的 ELF 格式文件。
    • 通过 objcopy 工具，将 vmlinux 转换为可加载的 Image 格式（通常是裸数据格式）。

Image.gz 生成过程

  1. 压缩内核
    • 使用 gzip 工具压缩 Image 文件，生成 Image.gz。
    • 压缩后的文件更小，方便传输和存储。
  2. 安装内核（可选）
    • 将 Image.gz 复制到引导加载程序可访问的目录。
    • 更新引导加载程序配置以加载新的内核。

总结

  • Image 是内核的未压缩版本，适合直接加载。
  • Image.gz 是压缩版本，适合节省空间。

这种流程适用于大多数 Linux 内核的构建和压缩过程。具体步骤可能根据不同的硬件平台和配置选项有所调整。

H.264 解码器示例工作流程：

视频流首先由 MPI/MPP 层排队，MPP 将流发送到 codec 层，codec 层解析流头并生成协议标准输出。此输出将发送到 HAL 生成寄存器文件集并与硬件通信。硬件完成任务并发送回信息。MPP 通知 codec 硬件结果，codec 按显示顺序输出解码帧。

  1. 解码器支持三种内存使用模式：

    模式1：纯内部模式

    在这种模式下，用户不会调用 MPP_DEC_SET_EXT_BUF_GROUP 控制解码器，只需调用 MPP_DEC_SET_INFO_CHANGE_READY 让解码器继续运行。解码器会在内部创建缓冲区，用户需要释放每一帧获取到的缓冲区。

    优点：

    • 简单易用，快速获得示例。

    缺点：

    1. 解码器的缓冲区在解码器关闭前可能无法返回，可能导致内存泄漏或崩溃。
    2. 无法控制解码器的内存使用，解码器处于自由运行状态，会消耗所有可用内存。
    3. 实现零拷贝显示路径较为困难。

    模式2：半内部模式

    这种模式是当前 mpi_dec_test 代码使用的模式。用户需要根据返回的信息变化创建 MppBufferGroup，并使用 mpp_buffer_group_limit_config 限制解码器的内存使用。

    优点：

    1. 简单易用。
    2. 用户可以在解码器关闭后释放 MppBufferGroup，内存可以更安全地保留更长时间。
    3. 可以通过 mpp_buffer_group_limit_config 限制内存使用。

    缺点：

    1. 缓冲区限制仍不准确，内存使用是100%固定的。
    2. 实现零拷贝显示路径仍然困难。

    模式3：纯外部模式

    在这种模式下，用户需要创建空的 MppBufferGroup 并通过文件句柄从外部分配器导入内存。在 Android 上，surfaceflinger 将创建缓冲区，mediaserver 从 surfaceflinger 获取文件句柄并提交到解码器的 MppBufferGroup。

    优点：

    • 最有效的零拷贝显示方式。

    缺点：

    1. 学习和使用较为困难。
    2. 播放器工作流可能限制这种使用方式。
    3. 可能需要外部解析器来获取外部分配器所需的正确缓冲区大小。

    所需缓冲区大小计算：

    • 像素数据：hor_stride * ver_stride * 3 / 2
    • 额外信息：hor_stride * ver_stride / 2
    • 总计：hor_stride * ver_stride * 2

    对于 H.264/H.265 编码器，20+ 个缓冲区足够；对于其他编解码器，10 个缓冲区足够。

update-initramfs -u命令用于更新Linux系统的初始化内存文件系统（initramfs）。

initramfs是一个临时的文件系统，位于Linux启动过程的早期阶段。它包含了用于引导系统的必要文件、驱动程序和工具。initramfs在系统引导时被加载到内存中，并在完成引导过程之前提供必要的资源。

update-initramfs -u命令的作用是重新生成和更新initramfs文件。它会根据当前系统配置和安装的软件包，重新打包和生成initramfs文件，以确保其与当前系统环境保持同步。

在更新initramfs时，该命令会扫描系统上已安装的内核和相关的驱动程序，将其添加到initramfs文件中，以便在引导过程中正确加载所需的模块和驱动程序。

通常，在更新内核或修改了与引导相关的配置文件后，需要运行update-initramfs -u命令来确保initramfs文件与系统的变更保持同步。这样可以确保系统能够正确引导，并正确加载所需的模块和驱动程序。

编译过程：从源文件到汇编文件

​ 汇编的过程都做了什么呢？（从高级语言到低级语言的转变），先说一下原因呢？不论是何种高级语言编写的程序，都要经过预处理和汇编这一步，预处理这个步骤要做的不再解释，汇编就是从C程序转换为同样功能的汇编语言，不论是什么类型的芯片，使用什么语言来完成这一功能，他们最终的汇编语言必然都是差不多的，如果每一种语言都写一个编译器，那工作量可真的太大了，所以每个语言的汇编器不一样，这是肯定的，但他们全部必然有一个相同的编译器。

​ 上面也说了，汇编实际上就是将高级语言转变为低级的汇编语言，再具体可以细分为

  • 程序语句、函数转变为代码段
  • 变量、常量转变为数据段、BSS段、rodata段
  • 各种辅助信息也就是符号表和重定位表

​ 上面有一个地方认识错了，从C源文件编译为可执行文件的具体步骤是，预处理、编译、汇编、链接。

main.c main.i main.s main.o a.out

​ 预处理 编译 汇编 链接

我们先来分析编译这一步骤，又可以分为下面这一些零零散散的步骤：

​ 词法分析、语法分析、语义分析、中间代码生成、汇编代码生成、目标代码生成

​ 想去了解一下编译器，目前我的疑惑是同一架构的编译器，到底能不能通用。

==编译器==

​ 分解成一个又一个的token，是不可再分的最小单元，它可能是关键字、可能是符号、可能是字符串，标识符和标点符号。

​ 既然编译器是将高级语言编译成可执行程序的一种工具，可是编译器也是一种可执行程序呀，编译器是怎样编译出来的。

​ 最初的编译器通常是手工编写的，使用底层语言（如汇编语言）或其他更低级别的语言来实现。这些早期的编译器可能只支持语言的子集，但足以进行自举过程。

​ 当得到一个基本的编译器后，它可以用于编译更高级别的语言实现，例如用C语言编写的编译器。一旦拥有了使用高级语言编写的编译器，就可以更容易地进行编译器的改进和扩展。

​ 第二步骤为将最小的token整合成statement sequence，翻译一下就是语句序列

​ 分解成的token是线性结构，而token整合而来的statement squence 是一个abstract sysntax tree抽象语法树，尽可能的使语法保持完整

​ 每个语句的分类是不同的，就比如可以将变量分为varable-defintion，for循环有一个为for-statement while为while-statement，等等

在上一个步骤中已经将全部的token转换为了抽象语法树，这时候其实已经可以执行了，但并不是能让人很好的理解，更好的办法其实还是将其转换回线性结构，

词法分析

​ 从左到右。一个字符一个字符的读入源程序

​ 对源程序的字符流进行扫描，分解成一系列的记号 token(关键字、标识符、字面量、特殊字符、分界符等等)，判断是不是又错误的词法，就比如中文字符，圆角半角字符等等。

语法分析

​ 将前一阶段产生的token序列进行语法分析，看是否构成一个语法上正确的程序，分解成语短语，用语法树来表示，语法树不再是一个线性结构，而是树状结构，syntax 句法

语义分析

​ 检查语法分析输出的语句、程序、表达式有没有错误，语义分为动态语义和静态语义，例如不允许使用一个没有声明的变量，在运行期间检测出除数为零。其实我们大部分错误都是发生在这个地方。

生成中间代码

​ 将语法树转换成为中间代码，中间代码是内部表现形式，他非常接近目标代码，类似于伪代码形式，容易生成，容易将其翻译为目标代码。

生成汇编文件

​ 将中间代码翻译为汇编文件

汇编过程

​ 在上一个步骤中已经将经过预处理的原始代码，编译为了.S汇编文件，而汇编文件又会经过汇编器生成.o目标文件。

​ 汇编器动作的基本流程如下所示：

  1. 词法分析
  2. 语法分析
  3. 代码生成
  4. 赋值信息

​ 他的主要目录就是进行指令翻译，将汇编源文件，翻译为二进制指令，然后呢就是生成各种表的信息，就包括符号表和重定位表。

​ 指令生成

​ 我们编写的程序一般包括变量、函数、关键字等等，他们都会被放到不同的section段中，就包括代码段数据段、只读代码段、bss段，初始化的全局变量、静态变量放到数据段，未初始化的全局变量和静态变量放到bss段，而一些函数调用放到代码段，一些字符串放到只读数据段里面。

​ 符号表

​ readelf -s ~~~.o

​ 读取一个.o文件的符号表，符号表中的信息就是汇编语言中各个section，以及偏移地址

​ 重定位表

​ 当然只有符号表肯定是不行的，他们最终肯定会经过链接从而生成可执行文件，符号表只是将汇编程序中的一些符号进行了收集，那他们呢，要重定位到哪一个地址呢？

​ 总算是比较清楚了，也终于是知道了symtab是啥了。sym就是symbol tab 就是表，symtab就是符号表，而rel text就是重定位信息，可以通过readelf -r 来查看，而大S readelf -S可以查看各个段的信息，

符号表

​ 符号表：编译过程中用来来保存源程序中各种符号的信息，主要包括符号的地址值、类型、占用空间的大小。

ELF文件和BIN文件

​ 文件结构：
​ BIN文件即raw binary文件

​ 只包含机器码，纯粹的程序文件，即镜像文件

​ 类似的还有HEX文件

​ ELF文件除了机器码之外还有一些额外的信息

​ 段的加载地址、运行地址

​ 符号表、重定位表等等

链接过程：地址空间分配与链接脚本

​ 目的：将所有的可重定位目标文件合并组装成可执行目标文件

​ 主要步骤：地址空间分配，强符号和弱符号、重定位

地址空间的分配

​ 扫描所有的目标文件，从各个文件段表中获取各个文件代码段、数据段信息：大小、地址等

​ 从指定的链接地址开始，按照代码段、数据段、BSS段顺序将各个目标文件的同类型段合并，然后重新计算各个段的长度和位置

链接过程：符号解析 强符号和弱符号

​ 在地址空间分配的过程中，会创建全局符号表，收集目标文件中符号表中的符号，统一放到全局符号表之中，但是可能遇到不同模块中遇到相同符号的情况，那这种情况要怎样解决呢？这时候就有了强符号和弱符号的概念了

​ 强符号：函数名 初始化的全局变量

​ 弱符号：未初始化的全局变量

==强符号不允许多次定义，但是弱符号可以共存，强弱符号共存的情况，强符号会覆盖弱符号==

链接过程：重定位

本质就是修正指令中的符号地址

​ 是连接过程中最核心最重要的一步

​ 地址空间分配、符号解析都是为重定位服务的

修改符号地址

​ 读取各个目标文件中，各个段的重定位信息：重定位表

​ 进行符号重定位、修改指令中的符号地址

​ 确定各个重定位符号的新地址、形成新的符号表

地址修正

​ 重定位地址=新段基址 + 段内偏移

程序的运行

​ 加载器

​ 在操作系统环境下执行一个可执行文件，加载器会首先将可执行文件加载到内存中，加载器拷贝数据完毕之后，进行相关操作，然后会跳转到程序入口

​ 镜像加载地址

​ 程序头表：

​ 指定了可执行文件镜像加载到内存中的地址

​ 链接地址等于加载到内存中的地址

​ ELF文件头基本信息

​ 文件类型 运行平台 程序入口地址

Linux内存映像

​ 在Linux环境下，通过调用execve函数来启动加载器，可执行文件的运行通过进程的形式实现，当运行一个可执行文件时，主要工作有三步，创建一个独立的虚拟地址空间，读取可执行文件头，建立虚拟地址空间和可执行文件的映射关系，将PC指针设置为可执行文件的入口地址，启动运行

BSS段的处理

​ 特点：
​ 未初始化的全局变量、static局部变量放到BSS段

​ BSS段不占目标文件的空间

​ 可执行文件运行时，在内存中给BSS段分配空间

​ 设置BSS段的目的：早期计算机存储资源贵，节省磁盘空间

main函数入口

链接静态库

​ 归档文件

​ 目标归档文件

​ 制作静态库

​ ar rcs libtest.a x.o xx.o

​ 链接静态库

​ gcc main.c -l ./include -L ./lib -ltest

链接静态库带来的问题：
一般定义的函数都会引用到，所以链接的时候会全部组装在一起，就比如C静态库，我们使用了printf，将所有的模块全部连接上了，这就导致可执行文件的体积大大增加。

动态链接

​

静态链接的缺点：

​ 生成的可执行文件体积较大，包含相同的公共代码，浪费存储空间、运行时占用的内存较大

​ 程序运行时需要一次性加载到内存之中

​ 对于内存配置较小的系统可能装载不下，无法运行

动态链接：

​ 静态链接发生在链接的过程中，动态链接推迟到运行时再进行

​ 动态链接的目标文件也称为动态链接库，运行时除了可执行文件，这些库也要被加载到内存，进行重定位

好处

​ 解决了空间浪费问题，节省内存

​ 可以运行更大的程序

​ 升级更加容易方便

ldd 命令可以查看用到的共享库

​ 与地址无关的代码

​ 放到哪里，都可以执行，无需重定位，无需改变

​ 实现思想

​ 将指令中需要修改的部分分离出来，跟数据放在一起

​ 剩余指令就能做到与地址无关，被多个进程共享

​ 数据和需要被修改的指令在每个进程中都有一个副本，互不影响

​ gcc -fPIC -c main.c

共享库

​ 三个主要路径：
​ /lib 存放最关键和最基础的共享库，例如动态链接器、C库、数学库，这些库主要是/bin/sbin下程序运行以及系统启动所需要的库

​ /usr/lib 非系统运行所需要的关键性的共享库，比如开发时用到的一些共享库

​ /usr/local/lib 存放第三方应用程序所需要的一些共享库

​ 基本过程

​ ELF文件执行，动态链接器/bin/ld-linux-so.xvei加载成功，到.dynamic段里面找ELF文件依赖的共享库

​ 若该路径是绝对路径就到绝对路径下面寻找

​ 若该路径是相对路径，就到/lib /usr/lib /etc/ld.so.conf 配置文件指定的目录中查找共享库

​ 库目录缓存

​ /etc/ld.so.conf存放共享库的路径，为了避免每次查找，耗费时间，可以使用ldconfig生成一个缓存/etc/ld.so.cache

​ 新增、删除、或者更新共享库路径的时候，需要使用ldconfig更新缓存

​ 环境变量 LD_LIBRARY_PATH

​ 由若干个路径组成的环境变量，每个路径用冒号查找

​ 可以使用该变量临时更改程序依赖的共享库查找路径，而不影响系统中其他程序的运行。

插件

​ 什么是插件

​ 为了使软件方便扩展，具备通用性，可以采用插件机制

​ 主程序逻辑框架不变，各个业务以动态链接库的形式加载进来

​ 软件发布后，不用重新编译，通过插件形式更新功能、软件增值

​ 插件的本质是共享库，只不过组装形式复杂
