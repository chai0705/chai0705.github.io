该系统通过主电源与备用电池的切换，保障 RTC 模块在主电源（VCC3V3_SYS）断电时，由 CR1220 电池持续供电以维持时间信息。

工作原理：

  • 系统正常时，VCC3V3_SYS 经二极管 D2 为 RTC 供电，D3 反向截止避免主电源消耗电池；

  • 主电源失效时，D2 截止防止电池耗电，D3 导通，CR1220 为 RTC 供电确保其运行。

本章将介绍 RTC 的软件内容，具体包括 RTC 子系统框架、驱动源码分析以及 RX8010 驱动的移植方法。

上图中 RTC 子系统分为用户空间、设备驱动层和硬件层三个层次，其中设备驱动层包含 RTC 设备驱动和 RTC 核心层，各层功能如下：

用户空间是应用程序运行层，应用程序通过多种接口与系统交互，例如访问 /dev/xxx 设备节点读写 RTC 时间，或通过 sysfs、proc 文件系统获取 / 设置硬件状态。

设备驱动层是连接用户空间与硬件层的核心：

  • RTC 设备驱动负责硬件操作细节，通过与 RTC 芯片直接通信实现控制，将硬件设备以 device（设备文件）和 driver（驱动程序）形式暴露给上层，其中 /dev/xxx 是用户交互接口，driver 模块承担具体硬件控制；

  • RTC 核心层负责管理协调时间管理功能，确保系统正确读写时间，并保障时钟在断电时持续运行。

硬件层为实际的 RTC 硬件，通过 I2C、SPI 等硬件接口与设备驱动层通信，提供实际计时功能。

i2c_register_driver() 函数中，i2c_driver 结构体的 is_registered 变量会在驱动注册成功后赋值为 true。I2C 核心框架通过 i2c_init() 函数初始化，该函数使用 postcore_initcall 宏指定初始化时机。

因此，只有 I2C 总线注册成功后，才能通过 i2c_add_driver()（宏封装 i2c_register_driver()）注册驱动：先将 i2c_driver 与 i2c_bus_type 绑定，再调用 driver_register() 函数，通过设备模型初始化 i2c_driver 内部的 device_driver（流程遵循设备模型标准）。

driver_register() 的调用链为 bus_add_driver()->driver_attach()->bus_for_each_dev()，会遍历 I2C 总线上的所有设备（即 i2c_client），并调用 _driver_attach() 函数判断 i2c_driver 与 i2c_client 是否匹配。匹配过程通过 driver_match_device() 调用总线的 match 函数（即 I2C 总线的 match 函数），该函数内部调用 i2c_match_id()，通过 id_table 与 i2c_client 匹配。

若匹配成功，流程回到 _driver_attach()，再按 driver_probe_device()->really_probe() 的顺序调用 I2C 总线的 probe 函数。I2C 总线的 probe 函数会通过 to_i2c_driver() 将 device_driver 转换为 i2c_driver，最终调用 i2c_driver 自身的 probe 函数。（本段需要背诵）

IIO（Industrial Input/Output，工业输入 / 输出）是 Linux 内核中专门用于管理模数转换器（ADC）、数模转换器（DAC）及各类工业传感器的子系统。

该子系统的开发始于 2009 年。随着嵌入式系统与物联网（IoT）技术的兴起，越来越多设备需要与模拟或数字信号交互，典型场景包括：

  • 模数转换（ADC）：将温度、压力、光强度等模拟信号转换为数字信号；
  • 数模转换（DAC）：将数字信号转换为模拟信号（如控制电机转速、调节音量等）；
  • 其他传感器：如加速度计、陀螺仪、磁力计等。

这些设备功能各异，但均需与操作系统进行交互。若缺乏统一框架，每个设备都需单独开发驱动程序，会造成代码冗余、维护成本激增。为此，Linux 社区引入了 IIO 子系统，以实现对这类设备的标准化管理。

IIO 的特点如下：

  • 统一接口：IIO 提供标准化 API 与数据结构，使不同类型设备可通过相同接口交互。用户空间应用程序能借助标准文件系统接口（/sys/bus/iio/devices/）访问 IIO 设备，实现对传感器、转换器等硬件的统一管理与操作。

  • 多设备类型支持：IIO 子系统兼容多种设备，涵盖 ADC（模数转换器）、DAC（数模转换器）、温度传感器、光传感器、压力传感器，以及加速度计、陀螺仪等。广泛的设备适配性，使其成为处理工业及消费级传感器的理想方案。

  • 事件与触发机制：IIO 支持基于事件的触发机制（如传感器数值超阈值时触发中断），同时兼容硬件触发与软件触发。这一特性适用于实时数据采集场景，可满足时间敏感型应用的需求。

该函数的返回值为文件描述符 fd，由第 24 行的 anon_inode_getfd 函数创建并返回。anon_inode_getfd 的作用是生成匿名文件描述符 —— 这种特殊的 fd 不与实际文件系统路径或设备节点关联，而是内核动态创建的虚拟文件。匿名 fd 拥有独立的文件操作集，通过 ioctl 可借助它创建虚拟接口，让用户无需依赖传统文件系统，即可与内核中的特定功能或数据进行交互。

第 36-40 行代码根据文件打开模式（阻塞 / 非阻塞）及缓冲区状态，动态确定需等待的数据量：非阻塞模式下，to_wait 直接设为 0（不等待，立即返回）；阻塞模式下，则取用户请求字节数对应的单元数量与缓冲区 rb->watermark 的较小值，确保等待数据量既满足需求又不超过缓冲区性能限制。

45-71 行的 do-while 循环流程如下：先检查设备有效性，再判断缓冲区数据是否充足。若数据不足，阻塞模式会进入等待状态（直至数据满足或被信号中断），非阻塞模式则直接返回资源不可用错误；等待中可响应信号并处理中断，且通过循环保证条件满足时重试读取。当缓冲区数据充足，调用 rb->access->read_first_n (rb, n, buf)，按用户请求字节数 n 从缓冲区 rb 读取数据至用户空间 buf，并返回实际读取字节数。

该结构体定义了一组函数指针和标志位，构成 IIO 缓冲区的操作接口。通过这些接口，IIO 子系统可灵活管理缓冲区的存储、读取、配置及生命周期，同时支持多种工作模式与功能扩展。其中，read_first_n 读函数会在 iio_buffer_read_first_n_outer 函数中被实际调用。

该函数首先从文件结构中获取设备及缓冲区信息并验证有效性，随后通过 poll_wait 将当前文件描述符加入等待队列（确保数据到达时能被唤醒），最后调用 iio_buffer_ready 检查缓冲区是否满足读取条件（如达到水印值）。若满足，返回 EPOLLIN | EPOLLRDNORM 表示数据可读；否则返回 0 表示无事件。这种机制支持高效异步 I/O，可让用户空间程序在数据可用时及时响应。
