1. 什么是CAN？

简单来说，CAN 是一种通信协议，就像两个人之间用语言交流一样。只不过它不是人与人之间的交流，而是汽车、工业设备等电子设备之间的“对话”。

  • 用途：CAN 主要用于汽车内部的电子控制单元（ECU）之间的通信，比如发动机控制、刹车系统、车窗控制等。
  • 特点：速度快、可靠性高、抗干扰能力强。

2. 为什么需要CAN？

想象一下，如果一辆汽车里的每个部件（比如发动机、刹车、空调）都需要单独的电线连接到中央控制器，那车里会布满密密麻麻的电线，既复杂又容易出问题。

  • CAN的作用：通过一根总线（两根线），让所有设备共享信息，减少了电线的数量，降低了成本，同时提高了系统的可靠性和效率。

3. CAN的基本工作原理

可以把 CAN 想象成一个“广播电台”：

  • 总线：相当于广播频道，所有设备都连在这个频道上。
  • 节点：每个设备（比如发动机控制单元、刹车系统）都是一个“听众”或“发言人”。
  • 消息传递：当某个设备有信息要发送时，它会把信息“广播”到总线上，其他设备收到后，判断这条信息是否和自己有关。如果是，就处理；如果不是，就忽略。

4. CAN的核心概念

（1）消息帧

消息帧是 CAN 中传输数据的基本单位，就像一封信。一封“信”包括以下内容：

  • 标识符（ID）：用来标记这条消息的重要性，优先级高的消息可以先被处理。
  • 数据：实际要传递的内容，比如“当前车速是60km/h”。
  • 校验位：用来检查信息是否在传输过程中出错。

（2）仲裁机制

如果多个设备同时想发送信息怎么办？CAN 有一套“礼貌规则”：

  • 每个设备在发送信息前会先“监听”总线，看看有没有其他设备正在发送。
  • 如果两个设备同时开始发送，优先级高的消息（ID 小的消息）会胜出，优先级低的设备会自动停止发送，稍后再试。

（3）错误检测

CAN 非常注重可靠性，所以有多种方式来检测错误：

  • 如果某个设备发现总线上的信息有问题，它会发出“错误信号”，通知大家重新发送。
  • 这样可以保证信息的准确性，避免误操作。

5. CAN的应用场景

  • 汽车行业：这是 CAN 最常见的应用领域，比如控制发动机、变速箱、刹车系统等。
  • 工业自动化：工厂里的机器之间也需要通信，CAN 可以用来传递控制指令。
  • 医疗设备：一些复杂的医疗设备内部也需要用 CAN 来协调各个模块的工作。

6. 总结一下

  • CAN 是什么？ 一种让电子设备互相通信的协议。
  • 怎么工作的？ 所有设备共享一根总线，通过广播的方式传递信息。
  • 有什么特点？ 简单高效、可靠性高、抗干扰能力强。
  • 用在哪里？ 主要在汽车、工业设备和医疗设备中。

好的！既然你希望更详细地了解 CAN（Controller Area Network） 的知识，我会尽量全面、深入地为你讲解。以下是关于 CAN 的详细内容，分为多个部分进行说明。

一、CAN的基本概念

1. 什么是CAN？

CAN 是一种串行通信协议，最初由德国博世公司（Bosch）在 1980 年代开发，主要用于汽车内部的电子控制单元（ECU）之间的通信。它是一种多主控（Multi-Master）总线系统，允许多个设备在同一根总线上同时工作。

  • 特点：
    • 高可靠性：具有强大的错误检测和处理机制。
    • 实时性：支持高优先级消息的快速传输。
    • 灵活性：支持多种速率和拓扑结构。
    • 成本低：通过共享总线减少布线复杂度。

二、CAN的工作原理

1. 物理层

CAN 使用差分信号传输数据，通常使用两根线：

  • CAN_H（High） 和 CAN_L（Low）。
  • 差分信号的优点是抗干扰能力强，即使在强电磁干扰环境下也能可靠通信。

（1）电气特性

  • 显性电平（Dominant）：表示逻辑“0”，此时 CAN_H 和 CAN_L 的电压差较大（通常为 2V）。
  • 隐性电平（Recessive）：表示逻辑“1”，此时 CAN_H 和 CAN_L 的电压差接近于零。
  • 显性电平会覆盖隐性电平，这是 CAN 总线仲裁机制的基础。

（2）终端电阻

为了防止信号反射导致通信质量下降，CAN 总线两端需要连接 120Ω 的终端电阻。

2. 数据链路层

CAN 的数据链路层定义了如何将数据封装成帧，并在总线上进行传输。

（1）消息帧类型

CAN 定义了几种不同类型的消息帧，用于不同的用途：

  • 数据帧（Data Frame）：用于传输实际数据。
  • 远程帧（Remote Frame）：用于请求某个节点发送数据。
  • 错误帧（Error Frame）：用于通知总线上的设备发生了错误。
  • 过载帧（Overload Frame）：用于请求延迟后续帧的传输。

（2）数据帧结构

一个标准的数据帧包括以下字段：

  1. 起始位（Start of Frame, SOF）：标志帧的开始，总是显性电平。
  2. 标识符（Identifier, ID）：
    • 标识消息的优先级，ID 越小，优先级越高。
    • 分为 标准帧（11 位 ID） 和 扩展帧（29 位 ID）。
  3. 控制位（Control Bits）：
    • 包括保留位和数据长度码（DLC），用于指示数据段的字节数（0~8 字节）。
  4. 数据段（Data Field）：
    • 实际要传输的数据，最多 8 字节。
  5. CRC 段（Cyclic Redundancy Check）：
    • 用于校验数据的完整性。
  6. 应答段（ACK Slot）：
    • 接收节点在正确接收到数据后会发送一个显性电平作为应答。
  7. 结束位（End of Frame, EOF）：标志帧的结束。

（3）仲裁机制

当多个节点同时尝试发送消息时，CAN 使用 非破坏性逐位仲裁 来决定哪个消息优先发送：

  • 所有节点同时监听总线上的电平。
  • 如果某个节点发送的是隐性电平，但总线上出现显性电平，则该节点会停止发送，让优先级更高的消息继续传输。
  • 这种机制确保高优先级消息不会被中断。

3. 网络层与应用层

CAN 协议本身只定义了物理层和数据链路层，更高层的功能（如网络管理和应用层协议）通常由其他标准实现，例如：

  • CANopen：用于工业自动化。
  • J1939：用于商用车辆（如卡车和巴士）。
  • DeviceNet：用于工厂自动化。

三、CAN的分类

1. 按速率分类

  • 高速 CAN：
    • 速率范围：最高可达 1 Mbps。
    • 应用场景：动力系统（如发动机、变速箱）。
  • 低速 CAN（容错 CAN）：
    • 速率范围：最高 125 kbps。
    • 特点：即使一根线断开，仍能继续通信。
    • 应用场景：舒适系统（如车窗、空调）。

2. 按协议版本分类

  • CAN 2.0A：支持标准帧（11 位 ID）。
  • CAN 2.0B：支持标准帧和扩展帧（29 位 ID）。
  • CAN FD（Flexible Data-rate）：
    • 支持更高的数据速率（最高 8 Mbps）。
    • 数据段长度增加到 64 字节。

四、CAN的错误处理

CAN 具有强大的错误检测和处理机制，主要包括以下几种：

1. 错误类型

  • 位错误（Bit Error）：发送节点检测到自己发送的位与总线上的位不一致。
  • 填充错误（Stuff Error）：连续 5 个相同电平后未插入填充位。
  • CRC 错误（CRC Error）：接收节点计算的 CRC 值与发送节点的不一致。
  • 格式错误（Form Error）：帧格式不符合规范。
  • 应答错误（ACK Error）：发送节点未收到接收节点的应答。

2. 错误帧

当检测到错误时，节点会发送一个错误帧，通知其他节点重新发送数据。

3. 错误计数器

每个节点有两个计数器：

  • 发送错误计数器（TEC）：记录发送错误的次数。
  • 接收错误计数器（REC）：记录接收错误的次数。
  • 当计数器超过一定阈值时，节点会进入错误被动状态或离线状态。

五、CAN的应用场景

1. 汽车行业

  • 动力系统：发动机控制、变速箱控制。
  • 安全系统：ABS、ESP、气囊控制。
  • 舒适系统：空调、车窗、座椅调节。

2. 工业自动化

  • 机器人控制。
  • 工厂生产线设备通信。

3. 医疗设备

  • 复杂医疗设备的模块间通信。

4. 航空航天

  • 飞机内部系统的通信。

七、总结

CAN 是一种高效、可靠的通信协议，广泛应用于汽车、工业自动化等领域。它的核心优势在于：

  • 实时性强：支持高优先级消息的快速传输。
  • 可靠性高：具有强大的错误检测和处理机制。
  • 灵活性好：支持多种速率和拓扑结构。

当然！CAN（Controller Area Network）是一种广泛应用于汽车、工业自动化等领域的通信协议。我们可以用以下几个步骤来简化它的基础知识：

1. 什么是CAN？

CAN是一个用于设备之间相互通信的协议。想象一下，你的车里有很多不同的电子设备：比如发动机控制单元（ECU）、刹车系统、车窗控制等，它们需要互相传递信息。CAN就像是这些设备之间的“语言”，使它们能够在没有直接连接的情况下，通过一根共享的线缆来交换信息。

2. CAN的特点

  • 多主机结构：在CAN网络中，不止一个设备可以发送数据，任何设备都可以在任何时间发送消息，只要它没有被其他设备“打断”。
  • 实时性：CAN协议非常注重实时性，特别适用于那些对时间要求非常高的应用（例如汽车电子系统中的刹车和引擎控制）。
  • 高效性和可靠性：CAN具有自动错误检测和纠正功能，这使得它在长时间运行中更加稳定。

3. CAN的通信原理

  • 消息格式：每个CAN消息都有一个唯一的标识符（ID），可以用来区分不同的信息内容。消息中还包含数据字段（例如传感器的测量值）、错误检测字段等。
  • 广播式通信：当一个设备发送信息时，所有其他设备都能接收到，但是每个设备根据ID判断自己是否需要处理这条信息。

4. CAN的工作方式

  • 总线型拓扑：所有的设备都连接到同一根总线上，通过电线传递信息。任何一个设备都可以“发布”信息，其他设备根据需求来“订阅”。
  • 非破坏性优先级：如果多个设备同时试图发送信息，CAN协议会通过“优先级”来决定哪个设备先发送。优先级高的设备会被允许优先发送。

5. CAN的错误处理

CAN协议会自动检测消息中的错误，并且会重发损坏的消息。常见的错误类型包括“位错误”和“帧错误”。通过这种方式，CAN协议能确保信息的传输可靠。

6. 应用领域

  • 汽车电子：CAN广泛用于汽车中，用于控制发动机、刹车、车窗等。
  • 工业自动化：在工厂自动化系统中，CAN用于设备之间的通信，控制生产线上的各种机械设备。
  • 医疗设备：一些高级医疗设备也使用CAN进行内部通信。

7. 总结

CAN协议就是一种使得多个设备能在同一根线缆上高效、实时、可靠地互相传输信息的方式。它非常适合需要实时响应和高可靠性的系统。

1. CAN协议概述

**CAN（Controller Area Network）**是由博世（Bosch）公司于1983年开发的一个多主机、分布式、串行通信协议，最初用于汽车电子系统，现在广泛应用于各类嵌入式系统中。CAN的设计目标是提供一种可靠、实时、高效的通信方式，特别适用于对实时性和可靠性要求高的系统，比如汽车、工业自动化、医疗设备等。

2. CAN协议的基本结构

  • 总线型结构：CAN是基于总线型拓扑的网络，所有设备共享同一条物理传输介质（通常是双绞线）。每个设备通过发送和接收CAN消息与总线上其他设备进行通信。
  • 多主机：CAN网络支持多主机模式，即每个设备都可以主动发送数据，而不仅仅是接收其他设备的消息。主机之间没有严格的“主从”关系，任何设备都可以在网络上发起通信。

3. CAN的通信原理

3.1 消息格式

CAN协议的核心是消息，每条消息包括以下几个部分：

  1. 起始位（Start of Frame）：每个消息的开始标志，用来同步所有设备。
  2. 标识符（Identifier）：唯一标识每条消息。标识符也决定了消息的优先级，ID越小，优先级越高。
  3. 控制字段（Control）：包括一些控制信息，标识消息长度等。
  4. 数据字段（Data）：这部分存储传输的数据，数据长度可以从0到8字节不等。
  5. CRC（循环冗余校验）：用于检测消息是否被损坏，保证数据传输的可靠性。
  6. 应答位（Acknowledgment）：接收方会发送应答，告知发送方消息是否成功接收。
  7. 结束位（End of Frame）：消息结束标志。

3.2 CAN消息格式图解

一条标准的CAN消息格式如下所示：

Start of Frame | Identifier | Control | Data | CRC | Acknowledge | End of Frame
  • 起始位（Start of Frame）：1位，用于同步通信。
  • 标识符（Identifier）：标准CAN有11位标识符，扩展CAN支持29位标识符。
  • 控制字段（Control）：标识数据字段的长度，通常为5位。
  • 数据字段（Data）：最多可以传输8字节的数据。
  • CRC字段：用于错误检测，通常为15位。
  • 应答位：1位，接收设备确认是否成功接收到消息。
  • 结束位：1位，用于标识消息的结束。

3.3 消息优先级

在CAN协议中，标识符（ID）*的数值越小，消息的优先级越高。当多个设备同时发送消息时，ID较小的设备将拥有更高的优先权。CAN通过一种叫做“非破坏性优先级”的机制，确保较小ID的消息能够优先占用总线进行传输。具体来说，CAN采用*位级优先，即设备在发送时会监控总线的状态，如果发现总线上有其他设备已经在发送消息，它会暂停发送，直到总线空闲。

3.4 广播式通信

CAN网络中的通信是广播式的，即当一个设备发送消息时，所有连接到总线的设备都会接收到这条消息。但每个设备只有在消息的标识符与自己有关时才会对其进行处理。通过这种方式，CAN支持高效的点对多点通信。

4. CAN的错误检测与处理

CAN协议非常注重通信的可靠性，提供了多种错误检测和纠正机制：

4.1 错误类型

  1. 位错误（Bit Error）：在传输过程中如果某个比特位传输错误，接收设备会检测到并要求重传。
  2. 帧错误（Frame Error）：数据帧中传输不完整时（如帧长度不对），接收方会检测到。
  3. CRC错误：传输过程中，消息的CRC校验值与接收到的值不匹配时，会产生CRC错误。
  4. 应答错误（Acknowledgment Error）：如果发送设备没有收到接收方的应答位，认为发送失败。

4.2 错误处理机制

  • 错误标志位：每个CAN节点都具有一个错误计数器，当设备检测到错误时，计数器会增加。当错误计数达到一定阈值时，设备会自动进入错误状态，并暂停发送消息，直到恢复正常。
  • 自动重发：如果消息传输失败，CAN协议会自动重新发送消息，直到成功为止。

4.3 错误状态

  • 错误活动状态（Error Active）：设备正常工作。
  • 错误被动状态（Error Passive）：设备检测到连续的错误，进入被动模式，减少网络对其的影响。
  • 总线关闭状态（Bus Off）：设备出现严重错误，自动断开总线，防止干扰通信。

5. CAN协议的物理层

CAN的物理层并没有规定具体的硬件实现，它允许使用多种物理传输介质，如双绞线、光纤、无线等。但最常见的是使用双绞线。

5.1 CAN_H 和 CAN_L

在最常见的CAN双绞线系统中，有两个主要的信号线：

  • CAN_H（高电平）
  • CAN_L（低电平）

在没有通信时，这两条信号线的电压是相等的，通常为2.5V。当进行数据传输时，CAN_H和CAN_L的电压差异会发生变化，代表不同的逻辑状态。

6. CAN的应用

CAN的应用非常广泛，尤其是在需要多个设备通信且对实时性和可靠性有高要求的领域：

6.1 汽车电子

在汽车中，CAN网络用于连接发动机、刹车系统、车窗、空调等多个ECU（电子控制单元）。这些ECU通过CAN总线交换信息，从而实现高效、实时的控制。

6.2 工业自动化

CAN被广泛用于工业控制中，如PLC（可编程逻辑控制器）、机器人控制系统、传感器、执行器等设备之间的通信。

6.3 医疗设备

一些医疗设备，如患者监护仪、实验室仪器，也使用CAN协议来保证设备之间高效、稳定的通信。

6.4 其他应用

  • 无人机控制
  • 智能电网
  • 楼宇自动化系统

7. CAN的扩展

7.1 CAN FD（Flexible Data Rate）

CAN FD是在标准CAN基础上进行扩展的一种新协议。与传统CAN相比，CAN FD支持更大的数据帧（最多64字节），并且可以在数据传输过程中改变数据传输速率，从而提升了传输效率，适用于大数据量、高速通信的场景。

7.2 CANopen 和 DeviceNet

这两种是基于CAN协议的高层协议，用于实现设备之间的更加复杂的通信和控制。CANopen用于工业自动化领域，DeviceNet则常用于工控设备中。

一、CAN 总线基础概念

1. 什么是 CAN？

  • 定义：CAN 是一种基于广播的 串行通信协议，主要用于实时性要求高、电磁干扰强的分布式系统中（如汽车、工业控制）。
  • 核心目标：实现 多节点（ECU, Electronic Control Unit）之间的 可靠、实时、低成本 通信。
  • 协议特点：
    • 多主结构：所有节点均可主动发送数据，无需中央控制器。
    • 非破坏性仲裁：通过 ID 优先级解决总线冲突。
    • 差分信号传输：抗干扰能力强（双绞线）。
    • 错误检测与恢复：内置 CRC、ACK、帧校验等机制。

2. CAN 发展历史

  • 1983年：由 Bosch 公司开发，用于解决汽车电子系统通信问题。
  • 1986年：发布 CAN 1.0 规范。
  • 1993年：ISO 11898 标准化（涵盖 CAN 2.0A 标准帧和 2.0B 扩展帧）。
  • 2012年：推出 CAN FD（Flexible Data Rate），支持更高数据速率和更大数据负载。

二、CAN 协议核心原理

1. 总线拓扑结构

  • 物理层：采用 双绞线（CAN_H 和 CAN_L），终端需加 120Ω 电阻 消除信号反射。
  • 逻辑电平：
    • 显性电平（Dominant）：逻辑 0（CAN_H - CAN_L ≈ 2V）。
    • 隐性电平（Recessive）：逻辑 1（CAN_H - CAN_L ≈ 0V）。
  • 总线仲裁：通过 线与逻辑，显性位覆盖隐性位，保证高优先级 ID 胜出。

2. 消息帧类型

CAN 定义了 4种帧类型：

  1. 数据帧（Data Frame）：携带数据的标准帧。
  2. 远程帧（Remote Frame）：请求其他节点发送数据。
  3. 错误帧（Error Frame）：节点检测到错误时发送。
  4. 过载帧（Overload Frame）：用于帧间延迟。

3. 数据帧结构（以标准帧为例）

  • 标准帧（CAN 2.0A）：11 位 ID。
  • 扩展帧（CAN 2.0B）：29 位 ID。
  • 数据帧组成：
    字段长度（bit）说明
    SOF1帧起始（Start of Frame），显性位。
    Arbitration11/29ID + RTR（远程传输请求位，显性为数据帧）。
    Control6IDE（扩展标识符位）、DLC（数据长度码，0-8字节）。
    Data0-64实际数据（CAN FD 支持 0-64字节，经典 CAN 为 0-8字节）。
    CRC15循环冗余校验码，包含 CRC 序列和 CRC 界定符。
    ACK2应答位（发送隐性位，接收节点显性应答）。
    EOF7帧结束（End of Frame），7位隐性位。

4. 非破坏性仲裁机制

  • 原理：节点在发送 ID 时同时监听总线，若发现更高优先级 ID（显性位更多），则自动退出发送，等待总线空闲后重试。
  • 优先级：ID 值越小，优先级越高（例如 ID=0x000 的帧优先级最高）。

5. 错误检测与处理

  • 错误类型：
    • 位错误：发送位与总线实际电平不一致。
    • 填充错误：未遵守位填充规则（每5个相同位后插入1个相反位）。
    • CRC错误：接收方计算的 CRC 与发送方不一致。
    • 格式错误：帧格式不符合协议规范。
  • 错误处理：
    • 错误帧：检测到错误的节点立即发送错误帧（6位显性错误标志 + 8位隐性错误界定符）。
    • 错误计数器：每个节点维护 TEC（发送错误计数器） 和 REC（接收错误计数器）。
    • 节点状态：
      • 主动错误状态（正常通信）。
      • 被动错误状态（错误计数器超过阈值，仅监听总线）。
      • 总线关闭状态（完全停止通信）。

三、CAN FD（Flexible Data Rate）

1. 主要改进

  • 可变数据长度：支持 0-64字节 数据负载（经典 CAN 为 0-8字节）。
  • 双速率：仲裁阶段使用标准速率（如 500kbps），数据阶段使用更高速率（如 2Mbps）。
  • 改进 CRC：增加至 21位（经典 CAN 为 15位）。

2. 兼容性

  • CAN FD 控制器可兼容经典 CAN 节点，但混合网络需注意速率匹配。

四、CAN 物理层实现

1. 常见标准

  • ISO 11898-2：高速 CAN（最高 1Mbps），用于动力总成等实时性要求高的场景。
  • ISO 11898-3：低速容错 CAN（最高 125kbps），支持单线工作，用于车身控制。

2. 收发器芯片（Transceiver）

  • 功能：将控制器的逻辑电平转换为差分信号。
  • 常见型号：TI SN65HVD23x、NXP TJA1050/TJA1042。

五、CAN 应用场景

1. 汽车电子

  • 动力系统：发动机控制、变速箱通信。
  • 车身控制：车门、灯光、空调。
  • 诊断系统：OBD-II 接口（ISO 15765）。

2. 工业自动化

  • PLC 通信、传感器网络、机器人控制。

3. 其他领域

  • 医疗设备（如手术机器人）、航空航天（航电系统）、电梯控制。

六、CAN 开发工具与资源

1. 硬件工具

  • CAN 分析仪：如 Vector CANoe、Kvaser、PCAN-USB。
  • 开发板：STM32（内置 CAN 控制器）、Arduino + MCP2515 模块。

2. 软件工具

  • 协议栈：CANopen、J1939（商用车协议）。
  • 开源库：SocketCAN（Linux 内核支持）、CANalyzer。

3. 标准文档

  • ISO 11898：核心协议规范。
  • SAE J1939：商用车应用层协议。

七、CAN 的局限性及替代方案

  • 局限性：
    • 带宽有限（经典 CAN 最大 1Mbps）。
    • 数据负载较小（经典 CAN 最大 8字节）。
  • 替代协议：
    • CAN FD：升级版 CAN。
    • Ethernet/IP：用于高带宽场景（如自动驾驶）。
    • FlexRay：更高实时性，但成本更高。

当前有一台鲲鹏920处理器的服务器，需要在服务器上安装docker，但是由于国内服务器无法访问docker官网，故在收集各资料后，实践后整理出一个简易文档。

1. 卸载默认安装的docker

sudo apt-get remove docker docker-engine docker.io containerd runc

2.安装使用https使用存储库的包

apt-get install -y apt-transport-https ca-certificates curl software-properties-common

3. 添加官网docker的GPG密钥

国内如果没有挂载代理，无法访问docker官方的网址，这里根据实际情况选择对应的源 配置阿里源，如果碰到没有 Release 文件异常，可以先配置官方的GPG秘钥，然后再配置阿里源的GPG秘钥

3.1 官方源

curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo apt-key add -

3.2 阿里源

curl -fsSL https://mirrors.aliyun.com/docker-ce/linux/ubuntu/gpg | sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg

4. 设置docker apt源

4.1 官方源

add-apt-repository "deb [arch=amd64] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable"

4.2 阿里源

add-apt-repository "deb [arch=amd64] https://mirrors.aliyun.com/docker-ce/linux/ubuntu $(lsb_release -cs) stable"

4.3 arm处理器阿里源

华为鲲鹏920处理器可以用这个源

sudo add-apt-repository "deb [arch=arm64] https://mirrors.aliyun.com/docker-ce/linux/ubuntu $(lsb_release -cs) stable"

5. 更新apt源

apt-get update

6. 查看docker-ce和docker-ce-cli的版本选择想要的版本

apt-cache madison docker-ce
apt-cache madison docker-ce-cli

7. 安装docker

这里根据实际需要的版本，可以选择版本安装，也可以不加版本，默认安装最新版本

apt-get install docker-ce=5:27.1.2-1~ubuntu.20.04~focal docker-ce-cli=5:27.1.2-1~ubuntu.20.04~focal

8. 验证是否安装成功

docker --version

9. 给用户添加docker 权限

sudo gpasswd -a 【用户】 docker
