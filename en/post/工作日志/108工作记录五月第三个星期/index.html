● 我仔细看了整个项目，功能很全（Qt GUI、RKLLM 推理、Web 端、语音 ASR/TTS、硬件控制、状态监控）。以下是按优先级排序的优化建议：

🔴 高优先级

  1. handleSend() 和 handleWebRequest() 意图匹配逻辑大量重复（~150 行重复代码）

两个函数里有一模一样的关键字匹配逻辑，维护时要改两处。建议提取为公共函数：

// 在 chat_window.h 中新增 struct IntentResult { bool isLocal = false; QString response; };

IntentResult parseIntent(const QString &command);

然后两个 handle 函数都调用它。

  1. Whisper 语音识别阻塞主线程（UI 卡顿）

onVoiceBtnReleased() 第 537 行： whisperProcess.waitForFinished(8000); // 阻塞 UI 8 秒！

用户在点击”按住说话”松开后，界面会卡死 8 秒。应该把 ASR 放到 LLMWorker 线程或者用 QThreadPool 异步执行。

  1. 对话历史无限增长，最终会内存溢出

conversationHistory 和 fullMarkdownLog 只追加不清理。长时间运行后：

  • 每次推理都要发送越来越长的 prompt → NPU 负载越来越高、响应越来越慢
  • setMarkdown() 渲染越来越大的文档 → UI 越来越卡

建议加轮数限制： const int MAX_HISTORY_ROUNDS = 10; // 超过时裁剪，保留系统设定 + 最近 N 轮

  1. 模型路径多处硬编码

以下路径分散在多个文件，建议集中到配置文件或 config.h：

  • chat_window.cpp:132: /rkllm/model/qwen3-vl-4b-instruct_w8a8_rk3588.rkllm
  • llm_worker.cpp:38: /rkllm/model/qwen3-vl-4b-instruct_w8a8_rk3588.rkllm
  • chat_window.cpp:532: /rkllm/model/ggml-base.bin

🟡 中优先级

  1. fullMarkdownLog 每次 setMarkdown() 全量重渲染

长对话时越来越卡。可以改用 QTextCursor 增量插入： QTextCursor cursor(chatDisplay->document()); cursor.movePosition(QTextCursor::End); cursor.insertMarkdown(text); // Qt 6.4+ 支持

  1. TTS killall -9 edge-tts mpg123 过于暴力

每次打断都用 killall -9 强制杀进程，可能导致音频子系统状态异常。建议用更优雅的方式： // 维护一个 QProcess* ttsProcess 指针 if (ttsProcess && ttsProcess->state() == QProcess::Running) { ttsProcess->terminate(); ttsProcess->waitForFinished(500); }

  1. getAllNpuUsage() 读取 /sys/kernel/debug/ 需要 root

普通用户运行时会静默失败返回 0。可以加权限检查或提示： if (!file.exists()) { qDebug() << “NPU load 需要 root 权限访问 /sys/kernel/debug”; }

  1. Web 服务器没有访问控制

任何能访问该设备 IP

的人都可以控制硬件。建议至少加一个简单的 Token 校验或本地回环限制（只允许 127.0.0.1 和局域网 IP）。

  1. smart_control.sh 中 echo 0/1 直接写 GPIO 路径

不同板子的 GPIO 编号可能不同，脚本目前是写死的。如果换一块 RK3588 开发板可能需要改脚本。

🟢 低优先级 / 代码风格

  1. CMakeLists.txt 链接了 -lpthread

Qt 自己的线程库已经够用了，且 QThread 内部已经处理了线程支持。这个不是必须的（但无害）。

  1. chat_window.cpp 中 QRegExp 已废弃

Qt5 开始 QRegExp 被标记为废弃，建议用 QRegularExpression 替代： // 旧 QRegExp rx(”(\d+)”); // 新 QRegularExpression rx(”(\d+)”);

  1. main.cpp 中 setrlimit(RLIMIT_NOFILE, 2048, 4096) 不够大

你同时在 autorun.sh 的 apply_ulimit() 里设置了 102400。两者取较小值生效，建议统一配置。

十二、常见问题排查

Q1：界面没有全屏 / 只是窗口

  • 检查启动方式：必须是 sudo rkllm_chat 或桌面快捷方式（已 sudo）
  • 程序默认 showFullScreen()，Esc 会退到窗口态，按 F11 重新进全屏

Q2：emoji 显示成方框

  • 缺 fonts-noto-color-emoji 包，运行 sudo apt install -y fonts-noto-color-emoji && sudo fc-cache -f
  • 已经装了还不行：可能 fontconfig 缓存没刷新，再执行一次 fc-cache -f

Q3：语音播报无声音

  • 看 /tmp/tts_err.log，常见原因：
    • edge-tts 没装 → pip3 install --user edge-tts，确保 /usr/local/bin/edge-tts 软链存在
    • 没联网 → Edge-TTS 是云端服务，断网会失败。可点 🔊 按钮关闭播报
    • 声卡识别错误 → 启动日志会有 🔊 已锁定 ES8388 声卡号：X，错了就改 chat_window.cpp::detectAudioCard

Q4：LED / 蜂鸣器不响应

  • 看 /tmp/smart_gateway.log 有没有报错
  • 确认 /topeet_test/03_beep/topeet_beep 存在且可执行
  • 确认 GPIO 节点存在：ls /sys/devices/platform/leds/leds/

Q5：模型加载失败 / NPU 报错

  • 确认 /rkllm/model/qwen3-vl-4b-instruct_w8a8_rk3588.rkllm 存在且大小约 4.5 GB
  • 确认 NPU 驱动加载正常：cat /sys/kernel/debug/rknpu/version
  • 程序底部会弹”⚠️ AI 大模型加载失败”，但硬件直通指令仍可用

Q6：Web 端打不开

  • 确认手机和板子在同一局域网
  • 确认 8080 端口没被占用：sudo netstat -tlnp | grep 8080
  • 防火墙：sudo ufw allow 8080 或直接关闭 ufw

优先建议

  1. 不要做图片识别，反而建议换掉 VL 模型
    现在模型名是 qwen3-vl-4b-instruct_w8a8_rk3588.rkllm，虽然你没用图片输入，但 VL 模型本身通常比纯文本模型更占资源。既然 RK3588 内存紧张，后续应该优先换成同级别或更小的纯文本 RKLLM 模型，比如
    1.5B/3B/4B 文本模型，别保留视觉分支能力。
  2. 把硬编码路径改成配置文件
    现在 /rkllm/model/…、/usr/local/bin/whisper-cli、8080、声卡号、TTS 开关、模型参数都散在代码里。建议增加 /etc/rkllm-chat/config.ini 或项目目录下 config.json，让部署后不用重新编译就能改模型 路径、端口、是否启用 TTS/ASR/Web。
  3. 增加统一推理 busy/队列机制
    Web 端有 isBusy，桌面端有 isGenerating，但它们不是统一锁。极端情况下桌面和 Web 可能同时向同一个 RKLLM worker 发请求。建议做一个全局 InferenceController：统一状态、统一 abort、统一排队或直接 拒绝。
  4. Web 端离线化
    web_server.cpp 里 Web 页面依赖 jsdelivr 的 marked 和 highlight.js。这和“本地离线演示”定位冲突，断网时 Web 端 Markdown/代码高亮会受影响。建议把 JS/CSS 内置到本地资源，或者直接简化为纯文本/轻 量 Markdown 渲染。
  5. 增加真正的 Web 停止生成接口
    当前 Web 端 stopGen() 主要是取消浏览器 fetch，服务端推理未必停止。建议增加 POST /abort，调用 rkllm_abort()，让手机端停止按钮真正中断 NPU 推理。
  6. 拆分 chat_window.cpp
    这个文件已经 130KB+，承担 UI、i18n、硬件、TTS、ASR、Web 协调、性能监控。后续维护会越来越难。建议拆成：
    • intent_engine.*
    • hardware_controller.*
    • audio_asr.*
    • tts_player.*
    • system_monitor.*
    • chat_window_ui.*
  7. 做正式包装部署
    现在 autorun.sh 已经能用，但更像开发部署脚本。实际项目建议补：
    • install.sh
    • uninstall.sh
    • systemd service
    • .desktop 文件不直接写 Exec=sudo …
    • 日志目录改到 /var/log/rkllm-chat/
    • 程序安装到 /opt/rkllm-chat/，/usr/bin/rkllm_chat 只做软链
  8. 减少运行时 shell 调用
    每秒状态刷新里 RAM 调用 free，TTS/录音/硬件也大量 sh -c。在开发板上 shell 启动成本不算小。建议：
    • RAM 直接读 /proc/meminfo
    • 背光、LED、温度等尽量 C++ 直接读写 sysfs
    • shell 脚本保留为兼容层，不作为主路径
  9. 内存优化方向
    不上图片识别的前提下，优先做：
    • 换纯文本小模型
    • 降低 max_new_tokens，现在是 4096，演示场景可改成 512/1024
    • TTS 默认关闭，避免网络和额外进程干扰演示

主要风险点

  1. 不能只换芯片不改型号声明
    旧软件如果一直按 STM32MP157A 设备树跑，普通功能可能没问题，但 C 的加密、安全硬件可能不会被启用。建议 BSP 里明确切到 STM32MP157C。

  2. Secure Boot 是最大差异
    C 支持安全启动，但默认不等于必须启用。
    如果客户只是想换成 C，不用安全启动，影响很小。
    如果客户要求安全启动/防拷贝/固件签名，那就是一个独立项目，需要设计密钥、OTP、签名镜像、量产烧录和售后解锁流程。

  3. 调试口可能被安全策略影响
    C 芯片如果烧了安全相关 OTP 或关闭调试权限，JTAG/SWD 行为会变化。工程样机阶段一定不要过早锁死。

  4. BOM 里其他物料也要一起确认
    你图里 DDR、eMMC 料号也不同。SoC A 换 C 本身风险不大，但 DDR/eMMC 换料会影响启动固件、DDR 参数、稳定性测试。
