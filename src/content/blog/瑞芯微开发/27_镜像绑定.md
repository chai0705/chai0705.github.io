142|> ### **1. 启动入口与硬件初始化**
143|>
144|> - 链接脚本指定入口：u-boot.lds（arch/arm/cpu/u-boot.lds）定义入口为_start
145|>   - `_start`（arch/arm/lib/vectors.S）：复位向量入口，跳转至reset
146|>     - `reset`（arch/arm/cpu/armv7/start.S）：复位处理入口
147|>       - `save_boot_params`（同文件）：保存引导参数到内存
148|>         - `save_boot_params_ret`（同文件）：参数保存完成后返回
149|>           - **`cpu_init_cp15`**（同文件）：初始化 CPU 的 CP15 协处理器（MMU、缓存等）
150|>           - `cpu_init_crit`（同文件）：关键 CPU 初始化
151|>             - **`lowlevel_init`**（arch/arm/cpu/armv7/lowlevel_init.S）：底层硬件初始化（如时钟、DDR）
152|>
153|> ### **2. 早期初始化（重定位前）**
154|>
155|> - `_main`（arch/arm/lib/crt0.S）：进入 C 语言环境入口
156|>   - **`board_init_f_alloc_reserve`**（common/init/board_init.c）：为全局数据结构（`gd`）分配内存空间
157|>   - **`board_init_f_init_reserve`**（同文件）：清零`gd`结构体，初始化基础字段
158|>   - `board_init_f`（common/board_f.c）：重定位前的板级初始化
159|>     - `initcall_run_list`（include/initcall.h）：执行早期初始化序列
160|>       - `init_sequence_f[]`（common/board_f.c）：早期初始化函数数组，包含：
161|>         - `board_early_init_f`（board/freescale/mx6ull_toto/mx6ull_toto.c）：初始化串口 IO 引脚
162|>         - `timer_init`（arch/arm/imx-common/timer.c）：初始化定时器（提供 U-Boot 时钟节拍）
163|>         - `init_baud_rate`（common/board_f.c）：从环境变量获取并设置波特率
164|>         - `serial_init`（drivers/serial/serial.c）：初始化串口控制器
165|>         - `console_init_f`（common/console.c）：初始化控制台（早期输出）
166|>
167|> ### **3. 镜像重定位**
168|>
169|> - **`relocate_code`**（arch/arm/lib/relocate.S）：将 U-Boot 镜像从加载地址拷贝到运行地址（重定位核心）
170|> - **`relocate_vectors`**（同文件）：将中断向量表重定位到新地址
171|>
172|> ### **4. 后期初始化（重定位后）**
173|>
174|> - `board_init_r`（common/board_r.c）：重定位后的板级初始化（依赖重定位完成）
175|>   - `initcall_run_list`（include/initcall.h）：执行后期初始化序列
176|>     - `init_sequence_r[]`（common/board_r.c）：后期初始化函数数组，包含：
177|>       - `initr_reloc`（common/board_r.c）：标记重定位完成（设置`gd->flags`）
178|>       - serial_initialize（drivers/serial/serial-uclass.c）：初始化串口设备（驱动模型）
179|>         - `serial_init`（同文件）：完成串口最终配置
180|>       - initr_mmc（common/board_r.c）：初始化 eMMC/SD 卡
181|>         - mmc_initialize（drivers/mmc/mmc.c）：扫描并初始化 MMC 设备
182|>           - `mmc_do_preinit` -> `mmc_start_init`（同文件）：执行 MMC 初始化流程
183|>       - `console_init_r`（common/console.c）：初始化完整控制台（支持输入输出）
184|>       - `interrupt_init`（arch/arm/lib/interrupts.c）：初始化中断控制器
185|>       - initr_net（common/board_r.c）：初始化网络设备
186|>         - eth_initialize（net/eth-uclass.c）：扫描并初始化以太网设备
187|>           - eth_common_init（net/eth_common.c）：通用以太网初始化
188|>             - `phy_init`（drivers/net/phy/phy.c）：初始化 PHY 芯片
189|>           - uclass_first_device_check（drivers/core/uclass.c）：查找并探测以太网设备
190|>             - device_probe（drivers/core/device.c）：设备探测（驱动匹配与初始化）
191|>               - `device_of_to_plat`：解析设备树信息到平台数据（如`fecmxc_of_to_plat`）
192|>               - drv->probe（驱动 probe 函数，如fecmxc_probe）：初始化以太网控制器
193|>                 - `fec_get_miibus`：初始化 MDIO 总线（管理 PHY）
194|>                 - fec_phy_init：初始化 PHY 芯片
195|>                   - `phy_connect`：连接 PHY 与 MAC 控制器
196|>                   - `phy_config`：配置 PHY（如自协商，调用`genphy_config_aneg`）
197|>               - `uclass_post_probe_device`：执行设备探测后操作（如`eth_post_probe`设置 MAC 地址）
198|>
199|> ### **5. 进入命令交互**
200|>
201|> - `run_main_loop`（common/board_r.c）：进入 U-Boot 主循环
202|>   - `main_loop`（common/main.c）：主循环核心逻辑
203|>     - **`bootdelay_process`**（common/autoboot.c）：读取环境变量`bootdelay`（启动延时）和`bootcmd`（自动启动命令）
204|>     - `autoboot_command`（同文件）：自动启动处理
205|>       - `abortboot`：倒计时提示（如 "Hit any key to stop autoboot: 3"），超时执行`bootcmd`
206|>     - **`cli_loop`**（common/cli.c）：若用户按键中断自动启动，进入命令行交互模式
207|>
208|> 以上流程清晰展现了 U-Boot 从硬件复位到命令交互的完整启动过程，涵盖了初始化、重定位、设备驱动加载等关键环节。

210|## 流程分析
211|
212|### 入口确定
213|
214|首先根据uboot顶层Makefile中的这段代码就决定了uboot的入口，即u-boot-init对应的head-y变量
215|
216|~~~Makefile
217|u-boot-init := $(head-y)
218|u-boot-main := $(libs-y)
219|
220|u-boot:	$(u-boot-init) $(u-boot-main) u-boot.lds FORCE
221|    +$(call if_changed,u-boot__)
222|~~~
223|
224|head-y具体定义在具体架构目录的Makefile中，架构由config.mk所得，也就是CONFIG_SYS_ARCH
225|
226|~~~c
227|ARCH := $(CONFIG_SYS_ARCH:"%"=%)
228|~~~
229|
230|CONFIG_SYS_ARCH具体定义在config文件里面，具体内容如下所示：
231|
232|~~~c
233|CONFIG_SYS_ARCH="arm"
234|~~~
235|
236|RK3588是arm架构，head-y 就定义在arch/arm/Makefile中，具体内容如下所示：
237|
238|~~~makefile
239|head-y := arch/arm/cpu/$(CPU)/start.o
240|~~~
241|
242|这里又有了一个名为CPU的变量，他是根目录的config.mk中由CONFIG_SYS_CPU配置的，具体内容如下所示：
243|
244|~~~C
245|CPU := $(CONFIG_SYS_CPU:"%"=%) 
246|~~~
247|
248|CONFIG_SYS_CPU也定义在config文件里面，具体内容如下所示：
249|
250|~~~c
251|CONFIG_SYS_CPU="armv8"
252|~~~
253|
254|所以head-y的最终值为arch/arm/cpu/armv8/start.o，从而确定入口为arch/arm/cpu/armv8/start.S。

256|### start.S汇编分析
257|
258|首先是入口判断
259|
260|~~~asm
261|.globl	_start
262|_start:
263|#ifdef CONFIG_ENABLE_ARM_SOC_BOOT0_HOOK
264|#include <asm/arch/boot0.h>
265|#else
266|	b	reset
267|#endif
268|~~~
269|
270|如果有特殊SOC需求，会先执行soc代码里的boot0，否则就会直接跳到reset，下面简单的做了一些导出处理，这里先不管，知道有它就行
271|
272|~~~asm
273|#if !CONFIG_IS_ENABLED(TINY_FRAMEWORK)
274|	.align 3                  /* 按8字节对齐（2^3），确保地址对齐要求 */
275|
276|/* 定义代码段基地址的全局符号，供其他模块引用 */
277|.globl	_TEXT_BASE
278|_TEXT_BASE:
279|#if defined(CONFIG_SPL_BUILD)
280|	.quad   CONFIG_SPL_TEXT_BASE  /* 如果是SPL（第二阶段引导）构建，定义为SPL的代码基地址 */
281|#else
282|	.quad	CONFIG_SYS_TEXT_BASE   /* 否则定义为U-Boot主程序的代码基地址（来自配置） */
283|#endif
284|
285|/*
286| * 以下符号的值由链接脚本（如u-boot.lds）确定，
287| * 这里通过计算偏移量（相对于_start）来定义全局符号
288| */
289|
290|/* 定义U-Boot镜像结束位置相对于起始位置的偏移量 */
291|.globl	_end_ofs
292|_end_ofs:
293|	.quad	_end - _start         /* _end是镜像结束地址，_start是入口地址，差值为总长度偏移 */
294|
295|/* 定义BSS段起始位置相对于镜像起始位置的偏移量 */
296|.globl	_bss_start_ofs
297|_bss_start_ofs:
298|	.quad	__bss_start - _start  /* __bss_start是BSS段起始地址，计算其相对于_start的偏移 */
299|
300|/* 定义BSS段结束位置相对于镜像起始位置的偏移量 */
301|.globl	_bss_end_ofs
302|_bss_end_ofs:
303|	.quad	__bss_end - _start    /* __bss_end是BSS段结束地址，计算其相对于_start的偏移 */
304|~~~
305|
306|然后分析reset，上面就是直接跳转到了reset，reset只有一句代码就是 `save_boot_params`，用于保存启动参数，启动参数大多是一些重要的寄存器值，reset内容如下所示：
307|
308|~~~asm
309|reset:
310|	b	save_boot_params
311|~~~
312|
313|但找了半天，我们发现并没找到代码块，但是在最下面发现了这样的一段代码
314|
315|~~~asm
316|WEAK(save_boot_params)
317|	b	save_boot_params_ret	/* back to my caller */
318|ENDPROC(save_boot_params)
319|~~~
320|
321|这段代码的意思是在默认情况下，`save_boot_params` 会跳转到 `save_boot_params_ret`，什么都不做。但如果某个 SoC 或板级代码需要保存启动参数，可以在其它文件里实现自己的 `save_boot_params`，覆盖这个弱符号。但很明显rk3588并没有实现，所以最终还是跳转到了save_boot_params_ret，save_boot_params_ret代码块内容有点多，这里分块进行讲解：
322|
323|**1 位置无关代码修正，如果启用位置无关代码，修正重定位相关符号。**
324|
325|~~~asm
326|#if CONFIG_POSITION_INDEPENDENT
327|	/*
328|	 * Fix .rela.dyn relocations. This allows U-Boot to be loaded to and
329|	 * executed at a different address than it was linked at.
330|	 */
331|pie_fixup:
332|	adr	x0, _start		/* x0 <- Runtime value of _start */
333|	ldr	x1, _TEXT_BASE		/* x1 <- Linked value of _start */
334|	sub	x9, x0, x1		/* x9 <- Run-vs-link offset */
335|	adr	x2, __rel_dyn_start	/* x2 <- Runtime &__rel_dyn_start */
336|	adr	x3, __rel_dyn_end	/* x3 <- Runtime &__rel_dyn_end */
337|pie_fix_loop:
338|	ldp	x0, x1, [x2], #16	/* (x0, x1) <- (Link location, fixup) */
339|	ldr	x4, [x2], #8		/* x4 <- addend */
340|	cmp	w1, #1027		/* relative fixup? */
341|	bne	pie_skip_reloc
342|	/* relative fix: store addend plus offset at dest location */
343|	add	x0, x0, x9
344|	add	x4, x4, x9
345|	str	x4, [x0]
346|pie_skip_reloc:
347|	cmp	x2, x3
348|	b.lo	pie_fix_loop
349|pie_fixup_done:
350|#endif
351|~~~
352|
353|**2 异常向量表设置，设置异常向量表地址，支持不同异常级别（EL3/EL2/EL1）。**
354|
355|~~~asm
356|	adr	x0, vectors
357|	switch_el x1, 3f, 2f, 1f
358|3:	msr	vbar_el3, x0
359|	mrs	x0, scr_el3
360|	orr	x0, x0, #0xf			/* SCR_EL3.NS|IRQ|FIQ|EA */
361|	msr	scr_el3, x0
