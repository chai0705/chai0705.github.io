> ### **1. 启动入口与硬件初始化**
>
> - 链接脚本指定入口：u-boot.lds（arch/arm/cpu/u-boot.lds）定义入口为_start
>   - `_start`（arch/arm/lib/vectors.S）：复位向量入口，跳转至reset
>     - `reset`（arch/arm/cpu/armv7/start.S）：复位处理入口
>       - `save_boot_params`（同文件）：保存引导参数到内存
>         - `save_boot_params_ret`（同文件）：参数保存完成后返回
>           - **`cpu_init_cp15`**（同文件）：初始化 CPU 的 CP15 协处理器（MMU、缓存等）
>           - `cpu_init_crit`（同文件）：关键 CPU 初始化
>             - **`lowlevel_init`**（arch/arm/cpu/armv7/lowlevel_init.S）：底层硬件初始化（如时钟、DDR）
>
> ### **2. 早期初始化（重定位前）**
>
> - `_main`（arch/arm/lib/crt0.S）：进入 C 语言环境入口
>   - **`board_init_f_alloc_reserve`**（common/init/board_init.c）：为全局数据结构（`gd`）分配内存空间
>   - **`board_init_f_init_reserve`**（同文件）：清零`gd`结构体，初始化基础字段
>   - `board_init_f`（common/board_f.c）：重定位前的板级初始化
>     - `initcall_run_list`（include/initcall.h）：执行早期初始化序列
>       - `init_sequence_f[]`（common/board_f.c）：早期初始化函数数组，包含：
>         - `board_early_init_f`（board/freescale/mx6ull_toto/mx6ull_toto.c）：初始化串口 IO 引脚
>         - `timer_init`（arch/arm/imx-common/timer.c）：初始化定时器（提供 U-Boot 时钟节拍）
>         - `init_baud_rate`（common/board_f.c）：从环境变量获取并设置波特率
>         - `serial_init`（drivers/serial/serial.c）：初始化串口控制器
>         - `console_init_f`（common/console.c）：初始化控制台（早期输出）
>
> ### **3. 镜像重定位**
>
> - **`relocate_code`**（arch/arm/lib/relocate.S）：将 U-Boot 镜像从加载地址拷贝到运行地址（重定位核心）
> - **`relocate_vectors`**（同文件）：将中断向量表重定位到新地址
>
> ### **4. 后期初始化（重定位后）**
>
> - `board_init_r`（common/board_r.c）：重定位后的板级初始化（依赖重定位完成）
>   - `initcall_run_list`（include/initcall.h）：执行后期初始化序列
>     - `init_sequence_r[]`（common/board_r.c）：后期初始化函数数组，包含：
>       - `initr_reloc`（common/board_r.c）：标记重定位完成（设置`gd->flags`）
>       - serial_initialize（drivers/serial/serial-uclass.c）：初始化串口设备（驱动模型）
>         - `serial_init`（同文件）：完成串口最终配置
>       - initr_mmc（common/board_r.c）：初始化 eMMC/SD 卡
>         - mmc_initialize（drivers/mmc/mmc.c）：扫描并初始化 MMC 设备
>           - `mmc_do_preinit` -> `mmc_start_init`（同文件）：执行 MMC 初始化流程
>       - `console_init_r`（common/console.c）：初始化完整控制台（支持输入输出）
>       - `interrupt_init`（arch/arm/lib/interrupts.c）：初始化中断控制器
>       - initr_net（common/board_r.c）：初始化网络设备
>         - eth_initialize（net/eth-uclass.c）：扫描并初始化以太网设备
>           - eth_common_init（net/eth_common.c）：通用以太网初始化
>             - `phy_init`（drivers/net/phy/phy.c）：初始化 PHY 芯片
>           - uclass_first_device_check（drivers/core/uclass.c）：查找并探测以太网设备
>             - device_probe（drivers/core/device.c）：设备探测（驱动匹配与初始化）
>               - `device_of_to_plat`：解析设备树信息到平台数据（如`fecmxc_of_to_plat`）
>               - drv->probe（驱动 probe 函数，如fecmxc_probe）：初始化以太网控制器
>                 - `fec_get_miibus`：初始化 MDIO 总线（管理 PHY）
>                 - fec_phy_init：初始化 PHY 芯片
>                   - `phy_connect`：连接 PHY 与 MAC 控制器
>                   - `phy_config`：配置 PHY（如自协商，调用`genphy_config_aneg`）
>               - `uclass_post_probe_device`：执行设备探测后操作（如`eth_post_probe`设置 MAC 地址）
>
> ### **5. 进入命令交互**
>
> - `run_main_loop`（common/board_r.c）：进入 U-Boot 主循环
>   - `main_loop`（common/main.c）：主循环核心逻辑
>     - **`bootdelay_process`**（common/autoboot.c）：读取环境变量`bootdelay`（启动延时）和`bootcmd`（自动启动命令）
>     - `autoboot_command`（同文件）：自动启动处理
>       - `abortboot`：倒计时提示（如 "Hit any key to stop autoboot: 3"），超时执行`bootcmd`
>     - **`cli_loop`**（common/cli.c）：若用户按键中断自动启动，进入命令行交互模式
>
> 以上流程清晰展现了 U-Boot 从硬件复位到命令交互的完整启动过程，涵盖了初始化、重定位、设备驱动加载等关键环节。

## 流程分析

### 入口确定

首先根据uboot顶层Makefile中的这段代码就决定了uboot的入口，即u-boot-init对应的head-y变量

~~~Makefile
u-boot-init := $(head-y)
u-boot-main := $(libs-y)

u-boot:	$(u-boot-init) $(u-boot-main) u-boot.lds FORCE
    +$(call if_changed,u-boot__)
~~~

head-y具体定义在具体架构目录的Makefile中，架构由config.mk所得，也就是CONFIG_SYS_ARCH

~~~c
ARCH := $(CONFIG_SYS_ARCH:"%"=%)
~~~

CONFIG_SYS_ARCH具体定义在config文件里面，具体内容如下所示：

~~~c
CONFIG_SYS_ARCH="arm"
~~~

RK3588是arm架构，head-y 就定义在arch/arm/Makefile中，具体内容如下所示：

~~~makefile
head-y := arch/arm/cpu/$(CPU)/start.o
~~~

这里又有了一个名为CPU的变量，他是根目录的config.mk中由CONFIG_SYS_CPU配置的，具体内容如下所示：

~~~C
CPU := $(CONFIG_SYS_CPU:"%"=%) 
~~~

CONFIG_SYS_CPU也定义在config文件里面，具体内容如下所示：

~~~c
CONFIG_SYS_CPU="armv8"
~~~

所以head-y的最终值为arch/arm/cpu/armv8/start.o，从而确定入口为arch/arm/cpu/armv8/start.S。

### start.S汇编分析

首先是入口判断

~~~asm
.globl	_start
_start:
#ifdef CONFIG_ENABLE_ARM_SOC_BOOT0_HOOK
#include <asm/arch/boot0.h>
#else
	b	reset
#endif
~~~

如果有特殊SOC需求，会先执行soc代码里的boot0，否则就会直接跳到reset，下面简单的做了一些导出处理，这里先不管，知道有它就行

~~~asm
#if !CONFIG_IS_ENABLED(TINY_FRAMEWORK)
	.align 3                  /* 按8字节对齐（2^3），确保地址对齐要求 */

/* 定义代码段基地址的全局符号，供其他模块引用 */
.globl	_TEXT_BASE
_TEXT_BASE:
#if defined(CONFIG_SPL_BUILD)
	.quad   CONFIG_SPL_TEXT_BASE  /* 如果是SPL（第二阶段引导）构建，定义为SPL的代码基地址 */
#else
	.quad	CONFIG_SYS_TEXT_BASE   /* 否则定义为U-Boot主程序的代码基地址（来自配置） */
#endif

/*
 * 以下符号的值由链接脚本（如u-boot.lds）确定，
 * 这里通过计算偏移量（相对于_start）来定义全局符号
 */

/* 定义U-Boot镜像结束位置相对于起始位置的偏移量 */
.globl	_end_ofs
_end_ofs:
	.quad	_end - _start         /* _end是镜像结束地址，_start是入口地址，差值为总长度偏移 */

/* 定义BSS段起始位置相对于镜像起始位置的偏移量 */
.globl	_bss_start_ofs
_bss_start_ofs:
	.quad	__bss_start - _start  /* __bss_start是BSS段起始地址，计算其相对于_start的偏移 */

/* 定义BSS段结束位置相对于镜像起始位置的偏移量 */
.globl	_bss_end_ofs
_bss_end_ofs:
	.quad	__bss_end - _start    /* __bss_end是BSS段结束地址，计算其相对于_start的偏移 */
~~~

然后分析reset，上面就是直接跳转到了reset，reset只有一句代码就是 `save_boot_params`，用于保存启动参数，启动参数大多是一些重要的寄存器值，reset内容如下所示：

~~~asm
reset:
	b	save_boot_params
~~~

但找了半天，我们发现并没找到代码块，但是在最下面发现了这样的一段代码

~~~asm
WEAK(save_boot_params)
	b	save_boot_params_ret	/* back to my caller */
ENDPROC(save_boot_params)
~~~

这段代码的意思是在默认情况下，`save_boot_params` 会跳转到 `save_boot_params_ret`，什么都不做。但如果某个 SoC 或板级代码需要保存启动参数，可以在其它文件里实现自己的 `save_boot_params`，覆盖这个弱符号。但很明显rk3588并没有实现，所以最终还是跳转到了save_boot_params_ret，save_boot_params_ret代码块内容有点多，这里分块进行讲解：

**1 位置无关代码修正，如果启用位置无关代码，修正重定位相关符号。**

~~~asm
#if CONFIG_POSITION_INDEPENDENT
	/*
	 * Fix .rela.dyn relocations. This allows U-Boot to be loaded to and
	 * executed at a different address than it was linked at.
	 */
pie_fixup:
	adr	x0, _start		/* x0 <- Runtime value of _start */
	ldr	x1, _TEXT_BASE		/* x1 <- Linked value of _start */
	sub	x9, x0, x1		/* x9 <- Run-vs-link offset */
	adr	x2, __rel_dyn_start	/* x2 <- Runtime &__rel_dyn_start */
	adr	x3, __rel_dyn_end	/* x3 <- Runtime &__rel_dyn_end */
pie_fix_loop:
	ldp	x0, x1, [x2], #16	/* (x0, x1) <- (Link location, fixup) */
	ldr	x4, [x2], #8		/* x4 <- addend */
	cmp	w1, #1027		/* relative fixup? */
	bne	pie_skip_reloc
	/* relative fix: store addend plus offset at dest location */
	add	x0, x0, x9
	add	x4, x4, x9
	str	x4, [x0]
pie_skip_reloc:
	cmp	x2, x3
	b.lo	pie_fix_loop
pie_fixup_done:
#endif
~~~

**2 异常向量表设置，设置异常向量表地址，支持不同异常级别（EL3/EL2/EL1）。**

~~~asm
	adr	x0, vectors
	switch_el x1, 3f, 2f, 1f
3:	msr	vbar_el3, x0
	mrs	x0, scr_el3
	orr	x0, x0, #0xf			/* SCR_EL3.NS|IRQ|FIQ|EA */
	msr	scr_el3, x0
	msr	cptr_el3, xzr			/* Enable FP/SIMD */
#ifdef COUNTER_FREQUENCY
	ldr	x0, =COUNTER_FREQUENCY
	msr	cntfrq_el0, x0			/* Initialize CNTFRQ */
#endif
	b	0f
2:	msr	vbar_el2, x0
	mov	x0, #0x33ff
	msr	cptr_el2, x0			/* Enable FP/SIMD */
	b	0f
1:	msr	vbar_el1, x0
	mov	x0, #3 << 20
	msr	cpacr_el1, x0			/* Enable FP/SIMD */
0:
~~~

**4.  SMP相关设置，设置SMPEN位，保证多核一致性。**

**5.应用CPU Errata，调用 `apply_core_errata`，修复已知的CPU bug**

~~~asm
bl	apply_core_errata
~~~

**6.低级硬件初始化，进行GIC等硬件初始化。**

~~~asm
bl	lowlevel_init
~~~

**7.多核启动处理,判断当前CPU是主核还是从核，主核继续执行，其他核等待或跳转。**

~~~asm
#if defined(CONFIG_ARMV8_SPIN_TABLE) && !defined(CONFIG_SPL_BUILD)
	branch_if_master x0, x1, master_cpu
	b	spin_table_secondary_jump
	/* never return */
#elif defined(CONFIG_ARMV8_MULTIENTRY)
	branch_if_master x0, x1, master_cpu

	/*
	 * Slave CPUs
	 */
slave_cpu:
	wfe
	ldr	x1, =CPU_RELEASE_ADDR
	ldr	x0, [x1]
	cbz	x0, slave_cpu
	br	x0			/* branch to the given address */
#endif /* CONFIG_ARMV8_MULTIENTRY */

#if defined(CONFIG_SPL_BUILD) && defined(CONFIG_ARM_SMP)
	mrs	x0, mpidr_el1
	and x0, x0, #0xfff
	cmp x0, #0
	beq master_cpu

#ifdef SMP_CPU1
	cmp x0, #(SMP_CPU1)
	ldr x1, =(SMP_CPU1_STACK)
	beq slave_cpu
#endif

#ifdef SMP_CPU2
	cmp x0, #(SMP_CPU2)
	ldr x1, =(SMP_CPU2_STACK)
	beq slave_cpu
#endif

#ifdef SMP_CPU3
	cmp x0, #(SMP_CPU3)
	ldr x1, =(SMP_CPU3_STACK)
	beq slave_cpu
#endif
	dsb sy
	isb

loop:
	wfe
	b loop

slave_cpu:
	bic	sp, x1, #0xf
	bl	smp_entry
	b	loop
#endif
~~~

**8.跳转到C入口**

~~~asm
master_cpu:
	bl	_main
~~~

经过上述一系列步骤，就是初步初始化CPU，然后跳转到crt0.S，这部分代码一般是不需要修改的，然后继续向下分析。

### crt0.S汇编分析

==这里有些疑问，不知道这个_main是如何定位到arch/arm/lib/crt0.S的，这个具体定位先不管，先来继续分析了==

crt0.S的运行逻辑是 U-Boot 启动后从汇编世界切换到 C 世界的关键桥梁，具体运行流程如下

**1. 设置初始栈指针，根据不同的构建类型（TPL/SPL/U-Boot），选择合适的栈地址，并保证 8 字节对齐：**

~~~asm
#if defined(CONFIG_TPL_BUILD) && defined(CONFIG_TPL_NEEDS_SEPARATE_STACK)
	ldr	r0, =(CONFIG_TPL_STACK)
#elif defined(CONFIG_SPL_BUILD) && defined(CONFIG_SPL_STACK)
	ldr	r0, =(CONFIG_SPL_STACK)
#else
	ldr	r0, =(CONFIG_SYS_INIT_SP_ADDR)
#endif
	bic	r0, r0, #7	/* 8-byte alignment for ABI compliance */
	mov	sp, r0
~~~

**2.初始化全局数据（gd）调用一系列 board_init_f 相关的函数，分配和初始化 gd（global data）结构，并设置启动标志，此时 gd 结构体已准备好，硬件初始化工作开始。**

~~~asm
bl	board_init_f_alloc_reserve
mov	sp, r0
mov	r9, r0
bl	board_init_f_init_reserve
bl	board_init_f_boot_flags
bl	board_init_f
~~~

**3.代码重定位（U-Boot主程序），如果不是 SPL，设置新的栈和 gd，然后进行代码重定位，重定位后，代码和数据都在新的内存地址。**

~~~asm
	ldr	r0, [r9, #GD_START_ADDR_SP]	/* sp = gd->start_addr_sp */
	bic	r0, r0, #7	/* 8-byte alignment for ABI compliance */
	mov	sp, r0
	ldr	r9, [r9, #GD_BD]		/* r9 = gd->bd */
	sub	r9, r9, #GD_SIZE		/* new GD is below bd */

#ifndef CONFIG_SKIP_RELOCATE_UBOOT
	adr	lr, here
	ldr	r0, [r9, #GD_RELOC_OFF]		/* r0 = gd->reloc_off */
	add	lr, lr, r0
#if defined(CONFIG_CPU_V7M)
	orr	lr, #1				/* As required by Thumb-only */
#endif
	ldr	r0, [r9, #GD_RELOCADDR]		/* r0 = gd->relocaddr */
	b	relocate_code
#endif
here:
~~~

**4.重定位异常向量表，保证异常向量表在新地址可用。**

~~~asm
bl relocate_vectors
~~~

**5.设置最终运行环境**

调用 `c_runtime_cpu_setup`，做 CPU 相关的最终设置。

**6.清零 BSS 段，将 BSS 段（未初始化的全局变量区）全部清零：**

~~~asm
ldr r0, =__bss_start
ldr r1, =__bss_end
mov r2, #0x00000000
clbss_l:cmp r0, r1
strlo  r2, [r0]
addlo  r0, r0, #4
blo clbss_l
~~~

**7.板级初始化和主流程跳转，调用板级初始化和主入口，进入 C 语言主流程，后续就是 U-Boot 的正常运行。：**

~~~asm
mov   r0, r9         /* gd_t */
ldr r1, [r9, #GD_RELOCADDR] /* dest_addr */
ldr pc, =board_init_r
~~~

> 设置初始栈和 gd
>
> 调用 board_init_f 做早期硬件初始化
>
> 代码重定位（U-Boot主程序）
>
> 重定位异常向量表
>
> 清零 BSS
>
> 跳转到 board_init_r，进入 C 世界
>
> 这是 U-Boot 启动从汇编到 C 的完整桥梁流程。

然后对start.S和crt0.S做一个总结

`start.S` 负责U-Boot的最底层启动流程，从CPU复位后开始执行，完成异常向量表设置、缓存和多核相关配置、硬件初始化等底层准备工作，最终将主核引导到C语言世界的入口 `_main`。它确保所有硬件和运行环境达到安全状态，为后续的C环境初始化和U-Boot主流程打下基础。

crt0.S 则承接了汇编入口的工作，负责建立C运行环境，包括设置栈指针、初始化全局数据结构gd、清零BSS段、代码重定位等，并最终跳转到 `board_init_r`，进入U-Boot的主功能流程。它是U-Boot从底层汇编到高级C代码的桥梁，确保系统能够顺利进入主程序并开始正常运行。

### 重定向代码分析

在crt0.S的第3和第4步骤，分别调用了relocate_code和relocate_vectors进行了代码重定位和异常向量表重定位。

relocate_code定义在arch/arm/lib/relocate_64.S中，具体内容如下所示：

~~~asm
ENTRY(relocate_code)                  ; 重定位代码的入口点
    stp     x29, x30, [sp, #-32]!     ; 保存x29（帧指针）和x30（返回地址）到栈，同时栈指针减32字节（创建栈帧）
    mov     x29, sp                   ; 更新帧指针为当前栈指针
    str     x0, [sp, #16]             ; 将输入参数x0（重定位目标地址）存入栈偏移16字节处

    /* 从flash拷贝u-boot到RAM */
    adrp    x1, __image_copy_start    ; x1获取__image_copy_start的高20位地址（页对齐部分）
    add     x1, x1, :lo12:__image_copy_start  ; 补充低12位，x1指向__image_copy_start（代码段起始地址）
    subs    x9, x0, x1                ; 计算目标地址与当前代码段起始地址的偏移（x9 = 目标地址 - 当前起始地址）
    b.eq    relocate_done             ; 如果偏移为0，说明无需重定位，跳转到relocate_done

    /*
     * 不能直接用ldr x1, __image_copy_start，因为如果代码已在非链接地址运行，
     * 该指令会加载重定位后的__image_copy_start值。为正确重定位，需知道链接时的地址。
     * 链接时的&__image_copy_start存于_TEXT_BASE，它是非重定位值（不是符号引用）
     */
    ldr     x1, _TEXT_BASE            ; x1加载链接时的代码段起始地址（CONFIG_SYS_TEXT_BASE）
    subs    x9, x0, x1                ; 计算目标地址与链接地址的偏移（x9 = 目标地址 - 链接地址）

    adrp    x1, __image_copy_start    ; 重新获取当前__image_copy_start的高20位
    add     x1, x1, :lo12:__image_copy_start  ; x1指向当前代码段起始地址
    adrp    x2, __image_copy_end      ; x2获取__image_copy_end的高20位地址
    add     x2, x2, :lo12:__image_copy_end    ; 补充低12位，x2指向代码段结束地址（__image_copy_end）
~~~

第二个relocate_vectors这个就不找了，在重定向完成就到了board_init_r了。

~~~c
	// 执行重定位后的初始化序列，失败则挂起
	if (initcall_run_list(init_sequence_r))
		hang();
	hang();
}
~~~

第7行的initcall_run_list函数会依次调用 init_sequence_r数组里的各种初始化函数，完成 U-Boot 的主流程初始化，包括内存、设备、环境变量、网络、命令行等，最后进入主循环。

### 命令交互run_main_loop

init列表里的最后一行代码是run_main_loop，这就会进入uboot的最后一步主循环函数

~~~c
/* 运行U-Boot主循环的函数 */
static int run_main_loop(void)
{
#ifdef CONFIG_SANDBOX
	sandbox_main_loop_init();  // 沙箱模式下初始化主循环（用于模拟环境）
#endif

	/* main_loop()可能返回以重试自动启动，因此需要循环调用 */
	for (;;)  // 无限循环
		main_loop();  // 调用主循环函数（处理命令交互、自动启动等）

	return 0;  // 理论上不会执行到这里（循环不会退出）
}
~~~

该函数的重点在第10行main_loop，它定义在common/main.c目录下，具体代码为

~~~c
void main_loop(void)
{
	const char *s;  // 用于存储自动启动命令字符串

	// 标记启动阶段为"main_loop"，用于跟踪启动进度
	bootstage_mark_name(BOOTSTAGE_ID_MAIN_LOOP, "main_loop");

#ifdef CONFIG_VERSION_VARIABLE
	// 设置环境变量"ver"为U-Boot版本信息
	env_set("ver", version_string);  /* 设置版本变量 */
#endif /* CONFIG_VERSION_VARIABLE */

	// 初始化命令行接口（CLI）
	cli_init();

	// 执行预启动环境变量中定义的命令（如preboot）
	run_preboot_environment_command();

#if defined(CONFIG_UPDATE_TFTP)
	// 通过TFTP更新U-Boot（如果配置了该功能）
	update_tftp(0UL, NULL, NULL);
#endif /* CONFIG_UPDATE_TFTP */

	// 处理启动延时（bootdelay），返回自动启动命令（bootcmd）
	s = bootdelay_process();
	// 处理设备树中的命令，若启用安全启动则执行安全相关命令
	if (cli_process_fdt(&s))
		cli_secure_boot_cmd(s);

	// 执行自动启动命令（倒计时结束无操作时）
	autoboot_command(s);

	// 进入命令行交互循环（用户中断自动启动后）
	cli_loop();

	// 如果命令行循环意外退出，触发恐慌提示
	panic("No CLI available");
}
~~~
