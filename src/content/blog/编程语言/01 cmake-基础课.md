20|12|12|# 一、准备知识
21|13|13|
22|14|14|## 1.1 C++的编译过程
23|15|15|
24|16|16|- -E 仅预处理；不编译、汇编或链接。
25|17|17|- -S 仅编译；不汇编或链接。
26|18|18|- -c 编译和汇编，但不链接。
27|19|19|- -o <file> 将输出放入<file>中。
28|20|20|
29|21|21|**C++源代码的编译过程**
30|22|22|
31|23|23|1. **预处理**：在预处理阶段，C++源代码中的预处理指令会被处理，包括宏展开和条件编译等。在此阶段，需要添加所有头文件的引用路径。
32|24|24|
33|25|25|   ~~~shell
34|26|26|   # 将xx.cpp源文件预处理为xx.i文件（文本文件）
35|27|27|   g++ -E main.cpp -o main.i
36|28|28|   ```
37|29|29|   ~~~
38|30|30|
39|31|31|2. **编译**：编译阶段会对预处理后的代码进行语法检查和编译，将代码翻译为汇编语言文件。
40|32|32|
41|33|33|   ~~~shell
42|34|34|   # 将xx.i文件编译为xx.s的汇编文件（文本文件）
43|35|35|   g++ -S main.i -o main.s
44|36|36|   ```
45|37|37|   ~~~
46|38|38|
47|39|39|3. **汇编**：汇编阶段将汇编语言文件转换为二进制格式的目标文件。
48|40|40|
49|41|41|   ~~~shell
50|42|42|   # 将xx.s文件汇编为xx.o的二进制目标文件
51|43|43|   g++ -c main.s -o main.o
52|44|44|   ```
53|45|45|   ~~~
54|46|46|
55|47|47|4. **链接**：链接阶段将目标文件与所依赖的库文件进行关联或组装，生成可执行文件。
56|48|48|
57|49|49|   ~~~shell
58|50|50|   # 将目标文件进行链接，生成可执行程序
59|51|51|   g++ main.o -o main
60|52|52|   ```
61|53|53|   ~~~

63|55|55|## 1.2 静态链接库和动态链接库
64|56|56|
65|57|57|静态链接库和动态链接库的区别在于链接的阶段不同。
66|58|58|
67|59|59|**静态链接库**的名称通常以`.a`结尾（表示archive library），它在编译阶段进行链接。如果一个工程依赖于静态链接库，那么生成的可执行文件或库会将静态链接库`.a`打包到输出文件中，因此生成的文件比较大。在运行时，不再需要单独的库文件。
68|60|60|
69|61|61|**动态链接库**的链接发生在程序的执行过程中，它在编译阶段仅进行链接检查，而不进行真正的链接过程。动态链接库的后缀名通常为`.so`（表示shared object，在Linux上）或`.dylib`（在macOS上）。动态链接库在加载后，在内存中只保存一份拷贝。多个程序依赖于它时，不会重复加载和拷贝，节省了内存空间。

127|119|119|### 1.3.2 CMake简介
128|120|120|
129|121|121|在实际工作中，推荐使用CMake来构建C++项目。CMake是一个开源的跨平台工具，用于构建、测试和软件打包。
130|122|122|
131|123|123|CMake具有以下特性：
132|124|124|
133|125|125|- **自动搜索依赖项**：CMake具有自动搜索可能需要的程序、库和头文件的能力，可以简化依赖项的配置过程。
134|126|126|- **独立的构建目录**：CMake支持使用独立的构建目录（例如`build`目录），这样可以安全地清理构建产生的中间文件和输出文件，不会污染源代码目录。
135|127|127|- **自定义命令**：CMake支持定义复杂的自定义命令，例如下载文件、生成各种文件等，可以满足项目构建过程中的特定需求。
136|128|128|- **自定义配置**：CMake支持根据需求进行自定义配置，可以选择性地启用或禁用特定的组件或功能。
137|129|129|- **文本文件生成工作区和项目**：CMake使用简单的文本文件（`CMakeLists.txt`）来描述项目的配置和构建规则，可以根据这些文件自动生成工作区和项目。
138|130|130|- **文件依赖项自动生成和并行构建**：CMake可以在主流平台上自动生成文件之间的依赖关系，从而使构建过程更高效。同时，CMake支持并行构建，可以加快构建速度。
139|131|131|- **支持多种IDE**：CMake几乎支持所有主流的集成开发环境（IDE），包括Visual Studio、Xcode、Eclipse等，可以方便地在不同的开发环境中进行项目开发和调试。

215|207|207|## 2.3 语法基础
216|208|208|
217|209|209|### 2.3.1 指定版本
218|210|210|
219|211|211|在CMake中，可以使用`cmake_minimum_required`命令指定当前项目所需的最低CMake版本。它的语法如下：
220|212|212|
221|213|213|```cmake
222|214|214|cmake_minimum_required(VERSION <version_number>)
223|215|215|```
224|216|216|
225|217|217|其中，`<version_number>`是所需的最低CMake版本号。在这个命令之后，CMake将会检查系统中安装的CMake版本是否满足这个要求，如果不满足则会产生错误。
226|218|218|
227|219|219|例如，如果要指定最低的CMake版本为3.10，可以在CMakeLists.txt文件中添加以下命令：
228|220|220|
229|221|221|```cmake
230|222|222|cmake_minimum_required(VERSION 3.10)
231|223|223|```
232|224|224|
233|225|225|这样，CMake将会检查系统中的CMake版本是否大于等于3.10。
234|226|226|
235|227|227|除了`cmake_minimum_required`命令，CMake中还有其他类似的命令，它们不区分大小写，并且有许多关键字来引导命令的参数输入，类似于函数的参数传递。这些命令使用的关键字在CMake中是不区分大小写的。

237|229|229|### 2.3.2 设置项目

240|232|232|
241|233|233|```cmake
242|234|234|project(ProjectName
243|235|235|    [VERSION <version_number>]
244|236|236|    [DESCRIPTION "project_description"]
245|237|237|    [LANGUAGES <language>]
246|238|238|)
247|239|239|```
248|240|240|
249|241|241|其中，`ProjectName`是项目的名称，在例子中使用的是"first_cmake"。`VERSION`关键字后面是项目的版本号，可以是任意格式的版本号，例如"1.0.0"。`DESCRIPTION`关键字后面是项目的描述，可以是一个字符串。`LANGUAGES`关键字后面是项目所使用的语言，这里使用的是"Cxx"，表示C++。
250|242|242|
251|243|243|示例中的`project`命令：
252|244|244|
253|245|245|```cmake
254|246|246|project(first_cmake
255|247|247|    VERSION 1.0.0
256|248|248|    DESCRIPTION "项目描述"
257|249|249|    LANGUAGES Cxx
258|250|250|)
259|251|251|```
260|252|252|
261|253|253|这样，通过`project`命令，可以在CMakeLists.txt中指定项目的基本信息，方便管理和描述项目。

351|343|343|### 2.3.6 CMake中的PUBLIC、PRIVATE、INTERFACE
352|344|344|
353|345|345|在CMake中，可以使用`target_...()`系列命令来操作目标（Target）。这些命令通常支持通过`PUBLIC`、`PRIVATE`、`INTERFACE`关键字来控制属性的传播。
354|346|346|
355|347|347|以`target_link_libraries(A B)`为例，下面是对这些关键字的理解：
356|348|348|
357|349|349|- `PRIVATE`：依赖项B仅链接到目标A。如果有目标C链接了目标A，目标C不会链接目标B。
358|350|350|- `INTERFACE`：依赖项B并不链接到目标A。如果有目标C链接了目标A，目标C会链接目标B。
359|351|351|- `PUBLIC`：依赖项B链接到目标A。如果有目标C链接了目标A，目标C也会链接目标B。
360|352|352|
361|353|353|可以将其类比为一个散烟的比方：
362|354|354|
363|355|355|- `PRIVATE`：就是自己抽烟，不给别人抽。
364|356|356|- `INTERFACE`：就是自己不抽烟，给别人抽。
365|357|357|- `PUBLIC`：就是自己抽烟，也给别人抽。
366|358|358|
367|359|359|从使用的角度来看，假设有目标C链接了目标A：
368|360|360|
369|361|361|- 如果目标B仅用于目标A的实现，并且不在头文件中提供给目标C使用，可以使用`PRIVATE`。
370|362|362|- 如果目标B不用于目标A的实现，仅在头文件中作为接口给目标C使用，可以使用`INTERFACE`。
371|363|363|- 如果目标B既用于目标A的实现，也在头文件中提供给目标C使用，可以使用`PUBLIC`。
372|364|364|
373|365|365|以下是一个示例：
374|366|366|
375|367|367|```cmake
376|368|368|# 创建库
377|369|369|add_library(c c.cpp)
378|370|370|add_library(D d.cpp)
379|371|371|add_library(B b.cpp)
380|372|372|
381|373|373|# 使用target_link_libraries命令进行链接
382|374|374|target_link_libraries(A PRIVATE B)
383|375|375|target_link_libraries(A INTERFACE C)
384|376|376|target_link_libraries(A PUBLIC D)
385|377|377|```
386|378|378|
387|379|379|在上述示例中，目标`A`通过`target_link_libraries`命令链接了目标`B`、`C`和`D`，使用了不同的传播属性。具体属性的选择取决于目标之间的关系和使用需求。
