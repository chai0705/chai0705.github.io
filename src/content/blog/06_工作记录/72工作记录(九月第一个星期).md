在海思 SS928V100 SDK 中，**RTC（实时时钟）的配置与 sys_config.ko 驱动及内核驱动的关系**需要结合 SDK 架构和具体场景来分析。以下是详细说明：

### 一、RTC 配置的核心逻辑

#### 1. **sys_config.ko 的作用范围**

- **系统级初始化**：`sys_config.ko`负责硬件平台的**底层初始化**，包括管脚复用、DDR 参数、时钟树配置等核心功能。这些配置通过 Linux DTS（设备树）集中管理，并在系统启动早期（加载 MPP 模块前）执行。
- RTC 相关配置：
  - **管脚复用**：若 RTC 使用特定 GPIO（如 RTC_CLK、RTC_INT），需在 DTS 中定义管脚功能，并通过`sys_config.ko`加载生效。
  - **时钟源选择**：RTC 的时钟源（如 32.768kHz 晶振或内部 RC）需在 DTS 中配置，并由`sys_config.ko`初始化。
  - **电源管理**：RTC 的电源域配置（如备用电源使能）可能通过`sys_config.ko`处理。

#### 2. **RTC 驱动的分层设计**

- **内核标准驱动**：RTC 的通用功能（如时间读写、闹钟）由内核`drivers/rtc`目录下的代码实现（如`rtc-hisi.c`），这部分代码可直接在内核中修改。
- **平台特定配置**：SS928V100 的 RTC 硬件特性（如寄存器地址、中断号）通过 DTS 描述，并由`sys_config.ko`解析和初始化。若实际硬件与 DTS 描述不符，需修改`interdrv/sys_config`中的 DTS 文件。

#### 3. **smp/a55_linux/interdrv/rtc 目录的作用**

- 平台适配代码：该目录包含针对 SS928V100 平台的 RTC 驱动适配层，例如：
  - **RTC 寄存器操作**：对 RTC 硬件寄存器的读写接口（如`rtc_reg_read()`、`rtc_reg_write()`）。
  - **中断处理**：RTC 闹钟中断的注册与回调函数。
  - **电源管理接口**：与平台电源管理模块的交互逻辑。
- 与内核驱动的关系：
  - 该目录下的代码是**内核驱动的补充**，而非替代。例如，内核驱动通过`platform_driver`注册设备，而`interdrv/rtc`提供底层硬件操作的具体实现。
  - 若需修改 RTC 的硬件操作逻辑（如寄存器时序），需在此目录中调整代码，并重新编译`sys_config.ko`。

### 二、修改 RTC 配置的具体路径

#### 1. **场景 1：修改 RTC 管脚复用或时钟源**

- 操作步骤：

  1. 打开`interdrv/sys_config`目录下的 DTS 文件（如`ss928v100.dtsi`）。
  2. 在`rtc`节点中修改管脚复用属性（如`pinctrl-0`）或时钟源配置（如`clock-frequency`）。
  3. 重新编译`sys_config.ko`并替换到设备中，确保驱动在系统启动时加载。

- 示例

  

  ```dts
  rtc@12345678 {
      compatible = "hisilicon,ss928v100-rtc";
      pinctrl-0 = <&rtc_clk_pins &rtc_int_pins>; // 修改管脚复用
      clock-frequency = <32768>; // 设置外部晶振频率
  };
  ```

#### 2. **场景 2：修改 RTC 驱动功能（如添加闹钟功能）**

- 操作步骤：

  1. 在`smp/a55_linux/interdrv/rtc`目录中修改驱动代码（如`rtc.c`）。
  2. 修改`Makefile`以包含新代码，并重新编译`sys_config.ko`。
  3. 若需新增用户空间接口（如`/dev/rtc0`的 ioctl 命令），需同步修改内核驱动中的`rtc_class_ops`结构体。

- 示例

  

  ```c
  // 在rtc.c中添加闹钟回调函数
  static int rtc_set_alarm(struct device *dev, struct rtc_wkalrm *alarm) {
      // 实现硬件闹钟设置逻辑
      return 0;
  }
  
  static const struct rtc_class_ops rtc_ops = {
      .set_time = rtc_set_time,
      .set_alarm = rtc_set_alarm, // 新增闹钟功能
  };
  ```

#### 3. **场景 3：修改 RTC 时间同步逻辑**

- 操作步骤：
  1. 若需修改 RTC 与系统时间的同步机制，需在内核`drivers/rtc/rtc-dev.c`中调整代码。
  2. 若涉及硬件时间校准（如温度补偿），需在`smp/a55_linux/interdrv/rtc`目录中实现校准算法，并通过`sys_config.ko`加载。

### 三、关键限制与注意事项

#### 1. **sys_config.ko 的加载顺序**

- `sys_config.ko`必须在 MPP 模块和内核 RTC 驱动之前加载，以确保硬件配置生效。若修改后未正确加载，可能导致 RTC 功能异常。

#### 2. **内核驱动与平台代码的依赖**

- **内核驱动**：负责提供标准接口（如`/dev/rtcN`）和通用逻辑（如时间转换）。
- **平台代码**：负责硬件操作和初始化。两者需协同工作，不可单独修改一方而忽略另一方。

#### 3. **编译与调试工具**

- **编译工具链**：需使用 SDK 提供的交叉编译工具链（如`aarch64-linux-gnu-`）编译`sys_config.ko`。

- 调试方法

  ：

  - 使用`dmesg`查看 RTC 驱动加载日志。
  - 通过`hwclock`命令验证 RTC 时间是否正确同步。

### 四、总结：修改 RTC 的推荐路径

| **需求类型**                 | **修改位置**                               | **依赖条件**                             |
| ---------------------------- | ------------------------------------------ | ---------------------------------------- |
| 管脚复用 / 时钟源配置        | `interdrv/sys_config`中的 DTS 文件         | 需重新编译`sys_config.ko`并加载          |
| 硬件操作逻辑（如寄存器读写） | `smp/a55_linux/interdrv/rtc`目录中的代码   | 需重新编译`sys_config.ko`并加载          |
| 通用功能扩展（如闹钟）       | 内核`drivers/rtc`目录 + `interdrv/rtc`目录 | 需同步修改内核驱动和平台代码，并重新编译 |

### 五、常见问题排查

1. **RTC 时间无法保存**：
   - 检查 DTS 中是否启用备用电源配置（如`backup-supply = <&vbat>`）。
   - 确认 RTC 芯片是否焊接正确且电池电压正常。
2. **RTC 中断未触发**：
   - 检查 DTS 中的中断号配置是否正确（如`interrupts = <GIC_SPI 123 IRQ_TYPE_LEVEL_HIGH>`）。
   - 查看内核日志是否有 RTC 中断注册失败的提示。
3. **sys_config.ko 加载失败**：
   - 确保驱动依赖的内核版本与设备一致。
   - 使用`modinfo sys_config.ko`检查驱动签名是否与内核匹配。



通过以上分析，可明确 RTC 配置在海思 SS928V100 SDK 中的具体路径和注意事项。实际开发中需结合 SDK 文档和硬件原理图，灵活调整配置策略。
