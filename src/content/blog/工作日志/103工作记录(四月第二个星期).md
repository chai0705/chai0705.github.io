Rockchip Matrix IO（简称 RM_IO）是瑞芯微（Rockchip）在其新一代芯片（如 RK3506 系列）中引入的一种革命性引脚复用技术。

传统的引脚复用（Pinmux）通常是“一对多”的固定映射，而 RM_IO 实现了“多对多”的任意路由。

------

### 1. 核心概念：从“多选一”到“矩阵交换”

在传统的芯片设计中，一个物理引脚（Pin）通常只能在有限的几个功能间切换。例如：

- 传统方式：GPIO0_A1 只能配置为 `GPIO`、`UART2_TX` 或 `PWM0`。如果你想把 `I2C1_SCL` 接在这个引脚上，硬件上是不可能的。
- Matrix IO 方式：引入了一个类似于“交换矩阵”的逻辑层。在定义的 RM_IO 引脚池内，内部的任何一个低速外设信号（如 UART, I2C, SPI, PWM, I2S）都可以软件路由到该池中的任意一个物理引脚。

### 2. 主要特点

- 极高的灵活性：不再受限于芯片手册上固定的 `mux` 选项。开发者可以根据 PCB 走线最便利的方向，自由分配功能引脚。
- 功能复用最大化：
  - 以 RK3506 为例，它支持 98 种功能信号映射到 32 个指定的物理引脚接口上（通常涵盖 GPIO0_A0 到 GPIO0_C7 等范围）。
- 支持实时切换：通过寄存器配置，可以在不重新布线的情况下，通过软件动态改变引脚定义的信号。
- 减少 PCB 复杂度：因为引脚可以任意分配，布线时可以避免大量的交叉和绕线，有助于减小板卡尺寸并降低电磁干扰（EMI）。

------

### 3. 硬件实现原理

RM_IO 实际上是在外设控制器与物理 Pad（引脚）之间增加了一层 Matrix Logic（矩阵逻辑）。

1. 功能池（Function Pool）：包含所有支持矩阵路由的信号（如 UART1~5, I2C1~3, SPI1~2, 多组 PWM 等）。
2. 引脚池（Pin Pool）：芯片划出一组特定的物理引脚（如 32 个或更多）专门用于 Matrix IO。
3. 寄存器映射：系统内部有一组特殊的配置寄存器，每一位代表一个交叉点。通过写入寄存器，可以像拨动开关一样，将“外设 A 的输出”连接到“物理引脚 B”。

------

### 4. 软件配置（Device Tree / DTS）

在 Linux SDK 中，配置 Matrix IO 与传统 pinctrl 略有不同。通常会在设备树中使用带有 `rm_io` 前缀的定义。

以配置 I2C1 为例：

在传统的 DTS 中，你可能会看到：

DTS

```
&i2c1 {
    pinctrl-0 = <&i2c1_xfer>; // 固定的 mux 选项
};
```

在支持 Matrix IO 的芯片（如 RK3506）中，配置变得更加具体：

DTS

```
&i2c1 {
    status = "okay";
    // 表示将 I2C1 的 SCL 映射到 RM_IO 的第 12 号引脚，SDA 映射到 13 号
    pinctrl-0 = <&rm_io12_i2c1_scl &rm_io13_i2c1_sda>; 
};
```

> 注意：这里的 `rm_io12` 对应的是芯片手册中 Matrix IO 分配表里的第 12 个物理引脚。你可以将其轻松改为 `rm_io01`，只要硬件上连接的是那个引脚即可。

------

### 5. 适用场景与局限性

#### 适用场景：

- 工业控制：需要灵活分配大量的 UART、CAN 或 PWM 信号。
- 紧凑型硬件：由于引脚定义灵活，可以极大地优化布线空间。
- 多机型通用底板：同一块核心板通过软件配置即可适配不同的底板引脚定义。

#### 局限性：

- 仅限低速 IO：目前 Matrix IO 主要针对 UART、I2C、SPI、PWM、I2S 等低速或中速信号。像 MIPI、HDMI、USB、以太网 等高速差分信号由于对物理电气特性（阻抗、时序）要求极高，目前仍采用固定的物理引脚，不支持矩阵路由。
- 特定芯片支持：并非所有瑞芯微芯片都支持此技术。目前该技术主要出现在 RK3506 等专为工业和嵌入式市场设计的新款 SoC 上。

### 总结

Rockchip Matrix IO 是瑞芯微在引脚分配逻辑上的一次重大升级。它把硬件引脚从“硬布线复用”变成了“软件定义引脚”，极大地降低了硬件开发的门槛和 PCB 设计的难度，是嵌入式开发者在处理复杂外设分配时的利器。

# --- 4. 配置与工具链管理 ---
# 从 .config 文件中加载指定的变量
load_config()
{
    [ -r "$RK_CONFIG" ] || return 0
    for var in $@; do
        export "$(grep "^$var=" "$RK_CONFIG" | tr -d '"' || true)" \
            &>/dev/null || true
    done
}

# 检查必要的配置变量是否存在，缺失则跳过当前任务
check_config()
{
    unset missing
    for var in $@; do
        eval [ -z \"\$$var\" ] || continue
        missing="$missing $var"
    done
    [ "$missing" ] || return 0
    notice "Skipping $(basename "${BASH_SOURCE[1]}") - ${FUNCNAME[1]} for missing configs: $missing."
    return 1
}

# 获取内核版本号（从 Makefile 解析）
kernel_version_raw()
{
    [ -d kernel ] || return 0
    VERSION_KEYS="VERSION PATCHLEVEL"
    VERSION=""
    for k in $VERSION_KEYS; do
        v=$(grep "^$k = " kernel/Makefile | cut -d' ' -f3)
        VERSION=${VERSION:+${VERSION}.}$v
    done
    echo $VERSION
}

# 获取内核版本（优先识别目录名中的版本号）
kernel_version()
{
    [ -d kernel ] || return 0
    KERNEL_DIR="$(basename "$(realpath kernel)")"
    case "$KERNEL_DIR" in
        kernel-*) echo ${KERNEL_DIR#kernel-}; return 0 ;;
    esac
    kernel_version_raw
}

# 创建编译日志文件并维护软链接
start_log()
{
    LOG_FILE="$RK_LOG_DIR/${2:-$1_$(date +%F_%H-%M-%S)}.log"
    ln -rsf "$LOG_FILE" "$RK_LOG_DIR/$1.log" # 总是指向最新的模块日志
    echo "# $(date +"%F %T")" >> "$LOG_FILE"
    echo "$LOG_FILE"
}

# 自动寻找交叉编译工具链 (GCC)
get_toolchain()
{
    MODULE="$1"
    TC_ARCH="${2/arm64/aarch64}"
    TC_VENDOR="${3-none}"
    TC_OS="${4:-linux}"

    MACHINE=$(uname -m)
    # 如果不是在 x86 平台运行，提示使用非 x86 工具链
    if [ "$MACHINE" != x86_64 ]; then
        notice "Using Non-x86 toolchain for $MODULE!" >&2
        # ... 架构兼容性判断 ...
        return 0
    fi

    # RV1126 芯片特殊处理
    if [ "$RK_CHIP_FAMILY" = "rv1126_rv1109" ]; then
        TC_VENDOR=rockchip830
    fi

    # 在 prebuilts 目录下按模式搜索 gcc
    TC_DIR="$RK_SDK_DIR/prebuilts/gcc/linux-x86/$TC_ARCH"
    if [ "$TC_VENDOR" ]; then
        TC_PATTERN="$TC_ARCH-$TC_VENDOR-$TC_OS-[^-]*-gcc"
    else
        TC_PATTERN="$TC_ARCH-$TC_OS-[^-]*-gcc"
    fi
    GCC="$(find "$TC_DIR" -name "*gcc" | grep -m 1 "/$TC_PATTERN$" || true)"
    
    if [ ! -x "$GCC" ]; then
        error "No prebuilt GCC toolchain for $MODULE!"
        exit 1
    fi
    echo ${GCC%gcc} # 返回工具链前缀
}

# --- 6. 错误处理钩子 ---
# 当任何命令出错时执行，打印调用栈，方便调试
err_handler()
{
    ret=${1:-$?}
    [ "$ret" -eq 0 ] && return 0
    fatal "ERROR: Running $BASH_SOURCE - ${2:-${FUNCNAME[1]}} failed!"
    fatal "ERROR: exit code $ret from line ${BASH_LINENO[0]}"
    # 打印调用栈
    for i in $(seq 1 $((${#FUNCNAME[@]} - 1))); do
        SOURCE="${BASH_SOURCE[$i]}"
        LINE=${BASH_LINENO[$(( $i - 1 ))]}
        fatal "    $(basename "$SOURCE"): ${FUNCNAME[$i]}($LINE)"
    done
    exit $ret
}

# --- 7. Hook（钩子）调度系统 ---
# 检查当前命令是否在支持的钩子命令列表中
hook_check()
{
    case "$2" in
        init | pre-build | build | post-build) ;;
        *) return 0 ;;
    esac
    # ... 从解析好的命令列表中匹配 ...
}

# 执行指定目录下的所有 .sh 钩子脚本
run_hooks()
{
    DIR="$1"; shift
    # 优先执行芯片私有目录的钩子，再执行通用钩子
    for dir in "$RK_CHIP_DIR/$(basename "$DIR")/" "$DIR"; do
        [ -d "$dir" ] || continue
        for hook in $(find "$dir" -maxdepth 1 -name "*.sh" | sort); do
            hook_check "$hook" "$1" "$2" || continue
            if ! "$hook" $@; then
                err_handler $? "${FUNCNAME[0]} $*" "$hook $*"
            fi
        done
    done
}

# 运行构建过程中的钩子（带日志记录）
run_build_hooks()
{
    case "$1" in
        init | pre-build | make-* | usage | parse-cmds)
            run_hooks "$RK_BUILD_HOOK_DIR" $@ || true
            return 0 ;;
    esac

    LOG_FILE="$(start_log "$1")"
    run_hooks "$RK_BUILD_HOOK_DIR" $@ 2>&1 | tee -a "$LOG_FILE"
    # ... 检查返回状态 ...
}

# --- 8. SDK 环境初始化 ---
setup_environments()
{
    export LC_ALL=C
    # 定义核心路径变量
    export RK_SCRIPTS_DIR="$(dirname "$(realpath "$BASH_SOURCE")")"
    export RK_SDK_DIR="$(realpath "$RK_SCRIPTS_DIR/../../../")"
    export RK_OUTDIR="$RK_SDK_DIR/output"
    export RK_LOG_DIR="$RK_OUTDIR/log/$(date +%F_%H-%M-%S)"
    export RK_CONFIG="$RK_OUTDIR/.config"
    # ... 其他路径 ...
}

# --- 9. 主入口函数 ---
main()
{
    # 处理 Makefile 调用的特殊选项
    case "$@" in
        make-*) makefile_options $@ ;;
    esac

    [ -z "$DEBUG" ] || set -x # 如果开启 DEBUG，显示详细执行过程
    trap 'err_handler' ERR   # 捕获错误
    set -eE                  # 脚本内任何命令出错立即停止

    setup_environments      # 1. 设置路径环境
    check_sdk               # 2. 检查 SDK 完整性

    # 处理选项，支持 ./build.sh rk3588:rockchip_defconfig 这种写法
    OPTIONS="${@:-$RK_DEFAULT_TARGET}"
    # ... 参数转换逻辑 ...

    parse_scripts           # 3. 解析并加载所有可用的命令插件
    source "$RK_PARSED_CMDS"

    # 4. 执行构建阶段
    run_build_hooks init $OPTIONS        # 初始化配置（如加载 defconfig）
    
    # 加载最终生成的 .config 环境变量
    set -a
    source "$RK_CONFIG"
    set +a

    # 5. 执行具体构建流程
    run_build_hooks pre-build $OPTIONS   # 构建前准备（如编译工具）
    run_build_hooks build $OPTIONS       # 核心构建（内核、文件系统等）
    run_build_hooks post-build $OPTIONS  # 构建后处理（打包固件）
}

# 脚本启动点：判断是直接运行还是被 source
if [ "$0" != "$BASH_SOURCE" ]; then
    "$BASH_SOURCE" ${@:-shell}
elif [ "$0" == "$BASH_SOURCE" ]; then
    main "${@%%:}"
fi
