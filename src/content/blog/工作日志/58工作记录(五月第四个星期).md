在 SPI 设备驱动开发中，`device`部分的核心是通过设备树（Device Tree）完成硬件资源描述，并由内核自动转换为`spi_device`结构体。以下从**设备树节点深度解析**、**内核转换全流程拆解**、**`spi_device`结构体成员全景说明**三个维度展开详细说明，覆盖开发中可能遇到的关键细节。

### 一、设备树中 SPI 设备节点的深度编写

设备树是描述硬件资源的标准化工具，SPI 设备节点需严格遵循**挂载在 SPI 控制器节点下**的层级关系，并通过属性精确传递硬件参数。以下是更完整的设备树示例及关键属性的深度解析：

#### 1.1 设备树节点典型结构

```c
// SPI控制器节点（由芯片厂商提供，通常位于arch/arm/boot/dts/xxx.dtsi）
spi@12340000 {
    compatible = "vendor,spi-controller-v2";  // 控制器驱动匹配字符串（关键）
    reg = <0x12340000 0x1000>;  // 控制器寄存器基地址与大小（{基地址, 长度}）
    interrupts = <GPIOH 6 IRQ_TYPE_LEVEL_LOW>;  // 控制器自身的中断（如传输完成）
    #address-cells = <1>;  // 子节点地址（片选）的cells数（SPI设备用1个u32表示片选）
    #size-cells = <0>;     // 子节点size的cells数（SPI设备无需size，固定为0）
    dmas = <&dma0 3>, <&dma1 4>;  // 控制器支持的TX/RX DMA通道（可选）
    dma-names = "tx", "rx";       // DMA通道名称（与dmas顺序对应）

    // SPI设备子节点（用户根据实际硬件添加，如一个SPI Flash）
    w25q128@0 {  // 节点名格式："设备名@片选编号"（片选0对应chip_select=0）
        compatible = "winbond,w25q128";  // 与spi_driver的id_table匹配的核心属性
        reg = <0>;  // 片选编号（必须与节点名中的@后数值一致，对应spi_device->chip_select）
        spi-max-frequency = <50000000>;  // 设备支持的最大速率（50MHz，对应max_speed_hz）
        spi-mode = <0>;  // SPI模式0（CPOL=0, CPHA=0，对应mode=0）
        bits-per-word = <8>;  // 每字8位（默认值，可省略）
        interrupts = <GPIOH 5 IRQ_TYPE_EDGE_RISING>;  // 设备中断（如写完成触发，对应irq）
        spi-cpha;  // 可选属性：等价于spi-mode的CPHA位（若存在则mode |= SPI_CPHA）
        spi-cpol;  // 可选属性：等价于spi-mode的CPOL位（若存在则mode |= SPI_CPOL）
        dmas = <&dma0 3>;  // 设备专用TX DMA通道（覆盖控制器的dmas属性，可选）
        dma-names = "tx";   // 设备DMA通道名称（与dmas对应）
        status = "okay";    // 设备状态（"okay"表示启用，"disabled"表示禁用）
    };
};
```



#### 1.2 关键属性深度解析

- **`compatible`**：
  格式为`"厂商,设备名"`（如`"winbond,w25q128"`），与`spi_driver`的`id_table`中的`compatible`字段严格匹配后，才会触发驱动绑定。若设备有多个兼容名称（如旧版本），可填写多个值（如`"winbond,w25q128", "generic,spi-flash"`），内核会按顺序匹配。
- **`reg`**：
  由 SPI 控制器的`#address-cells`决定格式（此处为 1 个 u32），表示设备的片选编号（`chip_select`）。SPI 控制器通过片选信号（CS）选中目标设备，总线上同一时刻仅能有一个设备被选中（片选拉低）。
- **`spi-max-frequency`**：
  内核在初始化`spi_device`时会将此值赋给`max_speed_hz`。SPI 控制器驱动在通信前会检查当前时钟频率是否超过`max_speed_hz`，若超过则自动降低速率（如通过分频），确保设备不被超规格驱动。
- **`spi-mode`**与`spi-cpol`/`spi-cpha`**：
  `spi-mode`是 4 位掩码，编码 SPI 通信的时钟极性（CPOL）和相位（CPHA）：
  - 第 0 位（`SPI_CPHA`）：时钟相位（0 = 数据在 SCK 首个边沿采样，1 = 数据在 SCK 第二个边沿采样）。
  - 第 1 位（`SPI_CPOL`）：时钟极性（0=SCK 空闲时为低，1=SCK 空闲时为高）。
    例如，`spi-mode = <3>`等价于`SPI_CPOL | SPI_CPHA`（CPOL=1, CPHA=1，模式 3）。
    若设备树中同时存在`spi-cpol`和`spi-mode`，则`spi-mode`优先级更高（`spi-cpol`/`spi-cpha`为旧版兼容写法）。
- **`bits-per-word`**：
  定义每次传输的字长（常见 8 位、16 位）。若设备支持多种字长（如某些 ADC 支持 12 位），驱动可在运行时通过`spi_device->bits_per_word`动态修改（需调用`spi_setup()`同步到控制器）。
- **`interrupts`**：
  格式为`<中断控制器 中断号 触发类型>`（如`<GPIOH 5 IRQ_TYPE_EDGE_RISING>`）。内核会将此解析为`spi_device->irq`，设备驱动可通过`request_irq()`注册中断处理函数（如处理设备的 “数据就绪” 信号）。
- **`dmas`与`dma-names`**：
  若设备支持 DMA 传输（如高速 SPI Flash），可通过此属性指定 DMA 通道。`dmas`列出 DMA 控制器和通道号（如`<&dma0 3>`表示 DMA 控制器 0 的通道 3），`dma-names`为通道命名（如`"tx"`对应发送 DMA）。内核会将此信息存储在`spi_device->dev.platform_data`中，供驱动初始化 DMA 时使用。

### 二、设备树节点到`spi_device`的内核转换全流程

#### 2.3 步骤 2：验证节点合法性

调用`of_spi_match()`检查子节点是否为 SPI 设备：

- 必须包含`compatible`属性（否则跳过）。
- 必须包含`reg`属性（否则报错`"spi-dev: missing reg property"`）。

#### 2.4 步骤 3：分配`spi_device`结构体

调用`spi_alloc_device(controller)`分配`struct spi_device`内存，并初始化基础成员：

```c
struct spi_device *spi_alloc_device(struct spi_controller *ctlr) {
    struct spi_device *spi;
    spi = kzalloc(sizeof(*spi), GFP_KERNEL);  // 分配内存
    spi->controller = ctlr;                   // 绑定所属控制器
    spi->master = ctlr;                       // 兼容旧版（与controller相同）
    spi->dev.parent = &ctlr->dev;             // 父设备为控制器
    spi->dev.bus = &spi_bus_type;             // 总线类型为SPI总线
    return spi;
}
```

#### 2.5 步骤 4：解析设备树属性并填充`spi_device`

通过`of_property_read_xxx()`系列函数解析设备树属性，逐一填充`spi_device`成员：

| ------------------- | --------------------------------- | --------------------------------------------------------- | ------------------------------------------------------ |
| `reg`               | `chip_select`                     | `of_property_read_u32(node, "reg", &cs)`                  | 片选编号，由`#address-cells`决定解析方式（此处为 u32） |
| `spi-max-frequency` | `max_speed_hz`                    | `of_property_read_u32(node, "spi-max-frequency", &speed)` | 若未指定，默认使用控制器的最大速率                     |
| `spi-mode`          | `mode`                            | `of_property_read_u16(node, "spi-mode", &mode)`           | 若未指定，检查`spi-cpol`/`spi-cpha`属性，默认`mode=0`  |
| `bits-per-word`     | `bits_per_word`                   | `of_property_read_u8(node, "bits-per-word", &bpw)`        | 若未指定，默认`8`位                                    |
| `interrupts`        | `irq`                             | `irq_of_parse_and_map(node, 0)`                           | 若未指定，`irq=-ENXIO`（无中断）                       |
| `dmas`/`dma-names`  | `dev.platform_data`（需驱动解析） | `of_dma_get(node, 0)`                                     | DMA 通道信息存储在`platform_data`中，由驱动按需处理    |

#### 2.6 步骤 5：关联控制器私有数据

SPI 控制器驱动可通过`spi_device->controller_state`存储与该设备相关的私有数据（如控制器的 FIFO 配置、时钟分频值等）。例如，某些控制器需要为不同设备配置不同的延迟参数，可在此字段保存。

#### 2.7 步骤 6：注册`spi_device`到 SPI 总线

调用`spi_add_device(spi)`将`spi_device`注册到 SPI 总线。此函数内部会触发：

- 调用`device_register(&spi->dev)`将设备加入 Linux 设备模型。
- 触发 SPI 总线的匹配逻辑（`spi_bus_type.match`），遍历所有已注册的`spi_driver`，检查`spi_device->dev.compatible`是否与`spi_driver->id_table`匹配。
- 若匹配成功，调用`spi_driver->probe()`函数，完成设备驱动的初始化（如分配内存、注册文件操作接口等）。

#### 2.8 调试验证：查看`spi_device`是否成功注册

注册完成后，可通过以下方式验证：

- **查看 sysfs**：`ls /sys/bus/spi/devices/spiX-Y`（如`spi0-0`表示控制器 0 的片选 0 设备）。
- **dmesg 日志**：内核会打印`"spi spiX-Y: ..."`的日志（如`"spi spi0-0: w25q128: registered"`）。
- **`spi_device`成员转储**：在驱动的`probe`函数中添加`dev_info(&spi->dev, "spi_device: cs=%d, speed=%dHz, mode=%d", spi->chip_select, spi->max_speed_hz, spi->mode)`，验证参数是否正确。

### 三、`spi_device`结构体成员全景说明

`spi_device`结构体（定义在`include/linux/spi/spi.h`）是连接设备树与驱动的核心桥梁，以下是所有关键成员的详细说明（含实际驱动开发中的使用场景）：

#### 3.1 基础成员（继承自设备模型）

```c
struct device dev;  // 继承自struct device的通用设备结构体
```



核心子成员：

- `dev.bus`：指向 SPI 总线（`&spi_bus_type`），用于触发总线匹配逻辑。
- `dev.driver`：匹配成功后指向对应的`spi_driver`（`struct spi_driver *`）。
- `dev.parent`：指向所属的 SPI 控制器（`&controller->dev`），用于层级管理。
- `dev.of_node`：指向设备树中的原始节点（`struct device_node *`），驱动可通过此指针读取额外设备树属性（如自定义的`vendor,delay`）。

#### 3.2 控制器关联成员

```c

struct spi_controller *master;      // 兼容旧版本（与controller相同，已弃用）
```

`spi_controller`关键作用：

控制器驱动通过

```

```

提供底层通信方法（如


函数），


通过


```c

    .len = data_len,
```

#### 3.3 硬件参数成员

```c
u32 max_speed_hz;  // 设备支持的最大通信速率（Hz）
u8 chip_select;    // 片选编号（0~控制器最大片选数-1）
u8 bits_per_word;  // 每字位数（8、16、32等）
u16 mode;          // SPI模式（SPI_CPOL | SPI_CPHA等）
int irq;           // 设备中断号（-ENXIO表示无中断）
```



- 实际使用场景：
  - `max_speed_hz`：控制器在传输前会检查当前时钟是否超过此值，若超过则调整分频（如控制器时钟为 100MHz，设备`max_speed_hz=50MHz`，则分频系数设为 2）。
  - `mode`：控制器根据`mode`配置 SCK 的极性（CPOL）和相位（CPHA），确保与设备的时序要求一致（如模式 0 要求 SCK 空闲为低，数据在上升沿采样）。
  - `bits_per_word`：控制器根据此值配置移位寄存器的宽度（如 8 位时，每次传输 8 位数据）。

#### 3.4 私有状态成员

```c
void *controller_state;  // 控制器私有状态（由SPI控制器驱动维护）

```



- **`controller_state`**：
  控制器驱动可在此存储与设备相关的私有数据。例如，某些控制器需要为不同设备配置不同的延迟（如片选到 SCK 的建立时间），可将延迟参数保存在`controller_state`中，在每次传输前应用。
- **`dev_state`**：
  设备驱动可在此存储私有数据（如设备的寄存器缓存、配置参数等）。例如，SPI Flash 驱动可在此保存`struct flash_info *`指针，避免使用全局变量。

### 四、常见问题与调试技巧

1. **设备树节点未被解析为`spi_device`**：
   - 检查`spi@xxx`控制器节点的`status`是否为`"okay"`（默认启用）。
   - 检查子节点的`compatible`属性是否与控制器驱动的`of_match_table`兼容（部分控制器驱动会过滤不支持的设备）。
   - 查看 dmesg 日志，搜索`"spi: probe of spiX-Y failed"`，定位具体错误（如`reg`属性缺失）。
2. **`spi_device`参数与设备树不一致**：
   - 在驱动的`probe`函数中打印`spi_device`成员（如`dev_info(&spi->dev, "speed=%dHz", spi->max_speed_hz)`），确认是否与设备树匹配。
   - 检查设备树属性名是否拼写错误（如`spi-max-frequency`写成`spi-max-freq`）。
3. **SPI 通信失败（如无数据返回）**：
   - 确认`mode`是否与设备手册一致（如设备要求模式 3，但设备树中误写为`spi-mode = <0>`）。
   - 检查`max_speed_hz`是否超过设备支持的最大值（可能导致时序错误）。
   - 使用逻辑分析仪抓取 SCK、MOSI、MISO、CS 信号，验证时序是否符合`mode`配置。

### 总结

通过设备树描述 SPI 设备节点后，内核会严格按照上述流程将其转换为`spi_device`结构体。`spi_device`不仅是硬件资源的抽象，更是 SPI 设备驱动（`spi_driver`）与控制器驱动交互的核心媒介。掌握设备树属性的精确编写、`spi_device`的转换流程及成员含义，是 SPI 驱动开发的基础，后续`spi_driver`的`probe`、`transfer`等函数的实现均需基于`spi_device`提供的硬件信息展开。





以下是 SPI 设备树节点转换为`spi_device`的简化流程（保留主线逻辑）：

### **1. SPI 控制器驱动加载与注册**

- **设备树匹配**：内核通过`compatible="rockchip,rk3066-spi"`找到 SPI 控制器驱动`spi-rockchip.c`。

- `probe`函数初始化
- 分配`spi_controller`结构体（代表 SPI 控制器）。
  - 解析设备树中的寄存器地址、时钟、中断等资源并初始化硬件。
- 调用`devm_spi_register_controller`注册控制器，最终触发`spi_register_controller`。

### **2. 注册控制器时触发子节点解析**

- 在`spi_register_controller`中，调用`of_register_spi_devices`遍历 SPI 控制器的所有子节点（如`mcp2515@0`）。

- 对每个子节点执行以下操作：

  1. **分配`spi_device`**：调用`spi_alloc_device`创建结构体。

  2. 解析设备树属性：

     - `reg` → `chip_select`（片选号，如`reg=0`对应 CS0）。
     - `spi-max-frequency` → `max_speed_hz`（最大通信速率）。
     - 其他模式属性（如`spi-mode`）设置`mode`成员。
     
  3. **注册设备**：调用`spi_add_device`将`spi_device`加入内核管理。

### **3. 关键函数调用链**



```plaintext
rockchip_spi_probe()
  → devm_spi_register_controller()
    → spi_register_controller()
      → of_register_spi_devices()  // 遍历子节点
        → of_register_spi_device()  // 处理单个子节点
          → of_spi_parse_dt()       // 解析属性到spi_device
          → spi_add_device()        // 注册设备
```



### **4. 驱动匹配与设备激活**

- `spi_device`注册后，内核通过其`compatible`属性（如`"my-mcp2515"`）匹配对应的`spi_driver`。
- 匹配成功后，调用驱动的`probe`函数完成设备初始化（如配置 SPI 通信参数、启动硬件）。

### **5. 必选属性与错误处理**

- **关键属性**：子节点必须包含`reg`（片选号）和`spi-max-frequency`（最大速率），否则`of_spi_parse_dt`会返回错误，导致设备注册失败。
- **结果**：若属性缺失，`spi_device`无法创建，上层驱动无法匹配，设备不可用。

在 Linux 内核中，SPI 驱动通过`spi_driver`结构体进行抽象表示，其核心成员包括设备 ID 匹配表、设备探测函数、设备移除函数、系统关机回调函数，以及继承自`device_driver`的基础驱动属性。具体结构如下：

```c
struct spi_driver {
    const struct spi_device_id *id_table;  // 传统设备ID匹配表（用于非设备树方式）
    int         (*probe)(struct spi_device *spi);  // 设备探测与初始化函数
    int         (*remove)(struct spi_device *spi); // 设备移除与资源释放函数
    void        (*shutdown)(struct spi_device *spi); // 系统关机时的回调函数
    struct device_driver driver;  // 基类驱动结构体，包含驱动名称、所有者、设备树匹配表等
};
```

SPI 驱动的注册过程通过`spi_register_driver`接口完成，该函数将`spi_driver`注册到 SPI 总线子系统，实现驱动与硬件的绑定。其核心逻辑如下：

```c
#define spi_register_driver(driver)  __spi_register_driver(THIS_MODULE, driver)

int __spi_register_driver(struct module *owner, struct spi_driver *sdrv)
{
    // 初始化驱动的基本属性
    sdrv->driver.owner = owner;  // 设置驱动所属模块
    sdrv->driver.bus = &spi_bus_type;  // 将驱动关联到SPI总线类型
    
    // 设置驱动的回调函数（这些函数会在匹配成功后被调用）
    sdrv->driver.probe = spi_drv_probe;  // 设备探测回调
    sdrv->driver.remove = spi_drv_remove;  // 设备移除回调
    
    // 如果驱动提供了shutdown回调，则也进行设置
    if (sdrv->shutdown)
        sdrv->driver.shutdown = spi_drv_shutdown;
        
    // 调用通用驱动注册接口，将驱动注册到内核
    return driver_register(&sdrv->driver);
}
```







在注册过程中，`spi_register_driver`会完成以下关键操作：



1. 将 SPI 驱动与 SPI 总线类型（`spi_bus_type`）关联，使内核知道该驱动用于处理 SPI 设备
2. 设置驱动的回调函数，这些函数会在设备匹配、移除、系统关机等场景下被自动调用
3. 通过`driver_register`将 SPI 驱动注册到内核的通用驱动管理系统中



这种分层抽象机制使得 SPI 驱动开发遵循统一的框架，同时保持与具体硬件的解耦，提高了驱动的可维护性和可移植性。

当 SPI 设备与驱动匹配成功后，内核会自动调用驱动的`probe`函数完成设备初始化。在 Linux SPI 子系统中，设备与驱动的匹配过程由总线层的`spi_bus_type`主导，核心逻辑通过`spi_match_device`函数实现，遵循**设备树匹配 → ID 表匹配 → 驱动名称匹配**的优先级顺序。结合代码清单 16.8 中的 MCP2515 驱动示例，具体匹配逻辑与硬件操作实现如下：

### 一、设备树（Device Tree）匹配机制（核心匹配方式）

代码清单 16.8 中，驱动通过`of_match_table`声明了支持的设备`compatible`属性（如`"my-mcp2515"`），而设备树节点需包含完全一致的`compatible`字段才能触发匹配：

#### 1. 设备树节点示例

```c
spi-controller@12340000 {  /* SPI控制器节点 */
    compatible = "vendor,spi-controller";
    #address-cells = <1>;
    #size-cells = <0>;
    mcp2515@0 {              /* MCP2515设备节点 */
        compatible = "my-mcp2515";  /* 必须与驱动of_match_table中的compatible一致 */
        reg = <0>;                /* SPI片选信号（CS）对应从机地址，通常为0~N */
        spi-max-frequency = <10000000>;  /* 最大SPI时钟频率（10MHz） */
        interrupt-parent = <&gpio控制器>;
        interrupts = <17 IRQ_TYPE_LEVEL_HIGH>;  /* 中断引脚 */
    };
};
```



#### 2. 驱动匹配逻辑

- 内核解析设备树时，会为每个 SPI 设备生成`spi_device`结构体，并填充其`of_node`字段（指向设备树节点）。
- `spi_match_device`优先遍历驱动的`mcp2515_of_match_table`，调用`of_device_id_match`检查设备节点的`compatible`属性是否匹配表中条目。
- 若匹配成功（如本例`"my-mcp2515"`），直接触发`mcp2515_probe`函数，无需后续 ID 表或名称匹配。

### 总结

I2C 设备的注册流程以**设备树解析**为核心，通过`i2c_add_adapter`注册控制器后，内核自动遍历其设备树子节点，利用`of_i2c_register_device`创建`i2c_client`结构体。该结构体包含从设备地址、中断号等关键信息，并通过`i2c_bus_type`总线机制与`i2c_driver`匹配，最终触发驱动的`probe`函数完成初始化。这一过程与 SPI 设备的注册逻辑类似，均体现了 Linux 设备驱动模型中 “总线 - 设备 - 驱动” 的分层设计思想。





在通过函数调用链完成设备树子节点遍历与属性解析后，设备树中描述的 I2C 从设备信息（如地址、中断、通信参数等）将被系统性地转化为内核中的`i2c_client`结构体。其关键成员如代码清单 XX.5 所示：

```c
struct i2c_client {
	unsigned short flags;/* 标志位 */
	unsigned short addr;/* 芯片地址*/
	char name[I2C_NAME_SIZE];	/* 设备名称 */
	struct i2c_adapter *adapter;	/* 所在的 I2C 适配器 */
	struct device dev;		/* 设备结构体 */
	int init_irq;			/* 初始化时设置的中断号 */
	int irq;			/* 设备产生的中断号 */
	struct list_head detected;	/* 检测到的设备列表 */
#if IS_ENABLED(CONFIG_I2C_SLAVE)
	i2c_slave_cb_t slave_cb;	/* 从模式下的回调函数 */
#endif
};
```
