# 第19章 网络设备

### 本章导读

网络设备是Linux三大设备类型之一，但其与字符设备、块设备在设计理念上存在显著差异。网络设备不通过文件系统中的设备节点进行访问，而是通过套接字（Socket）接口与上层应用交互，其核心任务是在网络协议栈与物理介质之间建立高效的数据传输通道。

本章首先从网络设备的硬件组成及PHY接口标准入手，深入剖析以太网物理连接的基础。随后，本章将详细介绍Linux网络驱动的分层框架，重点讲解`net_device`结构体、数据包载体`sk_buff`以及驱动开发中的注册、打开、收发等关键流程。最后，本章将延伸至应用层，讲解如何通过TCP/UDP套接字实现网络编程。

------

### 19.1 网络设备概述

“Internet”诞生于“冷战”时期，最初的雏形是“阿帕网”。广义上，接入Internet进行通信的设备称为网络设备，如计算机、智能手机和网络摄像头等。网络设备会将上层协议传来的数据包按照特定的媒介访问控制方式发送出去，同时将接收到的数据包再传递给上层协议。

**注意：** 与字符设备、块设备不同，网络设备在`/dev`目录下没有对应的设备文件，应用程序通过套接字与网络设备进行交互。

#### 19.1.1 硬件组成

嵌入式中网络设备硬件主要由以下五部分组件构成：

1. **网线：** 如图19.1所示，是网络通信中数据传输的物理载体。
2. **RJ45插座：** 如图19.2所示，用于连接网线。
3. **网络变压器：** 位于RJ45插座与PHY芯片之间。主要起到信号耦合、电气隔离和阻抗匹配的作用。可以是独立芯片（如图19.3），也可以集成在RJ45插座内部（如iTOP-RK3568使用的HR911130A）。
4. **PHY芯片：** 将数字信号转换为模拟信号（发送），或反之（接收）。常见型号如Realtek RTL8211E、YT8531C（见图19.4）。
5. **MAC控制器：** 负责数据链路层协议，管理与PHY的物理层交互。RK3568内部集成了两个MAC控制器。

#### 19.1.2 连接方式

根据CPU、MAC和PHY的不同组合，硬件连接主要分为三种方式：

- **方式一（全集成）：** CPU内部集成MAC和PHY。成本高、灵活性差，实际应用较少。
- **方式二（主流方案）：** CPU集成MAC，外部连接独立PHY芯片。通过MII、RGMII等接口互联（见图19.5）。这是iTOP-RK3568开发板采用的方式。
- **方式三（外挂方案）：** CPU通过总线外接集成MAC+PHY的芯片。适用于CPU无MAC资源的情况。

------

### 19.2 以太网PHY接口

PHY与MAC之间的通信接口是驱动开发中配置设备树的核心依据。

#### 19.2.1 MII接口

MII（Media Independent Interface）支持10/100Mbps速率。

- **信号线：** 共16根。
- **性能：** 100Mbps下时钟为25MHz。

#### 19.2.2 RGMII接口

RGMII（Reduced Gigabit Media Independent Interface）是目前千兆网最常用的接口，它是GMII的简化版本，引脚数从25个减少到14个。

表19.1 RGMII接口信号线描述

| 信号名称 | 描述 | 方向 |

| :--- | :--- | :--- |

| TXC | 发送时钟信号 (125MHz @ 1000Mbps) | MAC -> PHY |

| TX_CTL | 发送通道控制信号（TX_EN + TX_ER） | MAC -> PHY |

| TXD[3:0] | 发送通道数据信号（DDR模式） | MAC -> PHY |

| RXC | 接收时钟信号 | PHY -> MAC |

| RX_CTL | 接收通道控制信号 | PHY -> MAC |

| RXD[3:0] | 接收通道数据信号 | PHY -> MAC |

### 19.3 网络体系架构与协议

网络体系采用分而治之的设计方法，通过分层架构将网络功能划分为不同的层级。以OSI七层模型和TCP/IP四层模型最具代表性。

表19.2 TCP/IP四层模型核心功能

| 层级 | 名称 | 核心功能 |

| :--- | :--- | :--- |

| 4 | 应用层 | 直接与应用交互，如HTTP、FTP、DNS等。 |

| 3 | 传输层 | 提供端口到端口的服务。TCP（可靠连接）、UDP（无连接）。 |

| 2 | 网络层 | 提供主机到主机的通信。核心协议为IP，负责寻址与路由。 |

| 1 | 网络接口层 | 包含物理层与数据链路层，负责比特流传输与帧同步。 |

### 19.4 网络设备驱动框架

Linux网络驱动采用四层架构设计，使协议栈与硬件解耦，如图19.6所示。

#### 19.4.1 网络协议接口层

该层通过统一的函数处理数据：

- **发送：** `dev_queue_xmit(struct sk_buff *skb)`
- **接收：** `netif_rx(struct sk_buff *skb)`

核心数据结构`sk_buff`定义在内核源码的`include/linux/skbuff.h`文件中。它通过`head`、`data`、`tail`、`end`四个指针管理缓冲区，避免了数据在各协议层传输时的内存拷贝开销。

**代码清单19.1 sk_buff结构体关键成员**

C

```
struct sk_buff {
    struct sk_buff *next;    // 指向下一个sk_buff，构成双向链表
    struct sk_buff *prev;
    struct net_device *dev;  // 指向处理该包的网络设备
    unsigned int len, data_len; // 实际数据长度与分片长度
    __u16 transport_header;  // 传输层头部偏移
    __u16 network_header;    // 网络层头部偏移
    __u16 mac_header;        // 链接层头部偏移
    unsigned char *head, *data, *tail, *end; // 缓冲区管理指针
};
```

#### 19.4.2 网络设备接口层

该层核心是`net_device`结构体（定义在`include/linux/netdevice.h`），它描述了设备的属性（如MAC地址、MTU、中断号）及其操作函数集`net_device_ops`。

### 19.6 网络应用编程

应用程序通过套接字进行通信。

#### 19.6.1 TCP通信

TCP是面向连接的、可靠的传输协议。其通信流程如图19.7所示。

**TCP服务器核心步骤：**

1. **socket()：** 创建套接字。
2. **bind()：** 绑定IP地址与端口。
3. **listen()：** 进入监听状态。
4. **accept()：** 阻塞等待客户端连接。
5. **recv()/send()：** 数据交换。
6. **close()：** 关闭连接。

#### 19.6.2 UDP通信

UDP是无连接的传输协议，适用于对实时性要求高但对丢包不敏感的场景。UDP服务器直接通过`recvfrom()`接收任何客户端的数据，无需`accept()`过程。

### 19.1 网络设备概述

“Internet”诞生于“冷战”时期，最初的雏形是“阿帕网”。广义上，接入Internet进行通信的设备称为网络设备。在嵌入式系统中，典型的网络设备包括计算机、智能手机、工业网关等。

**注意：** 与字符设备、块设备不同，网络设备在`/dev`目录下没有对应的设备文件。应用程序通常通过套接字与网络设备进行交互。

#### 19.1.1 硬件组成

嵌入式系统中的网络设备硬件主要由以下五部分组件构成：

1. **网线**：如图19.1所示，是数据传输的物理载体。
2. **RJ45插座**：如图19.2所示，用于连接网线。
3. **网络变压器**：位于RJ45插座和PHY芯片之间，主要起到信号耦合、电气隔离和阻抗匹配的作用。
4. **PHY芯片**：物理层芯片，负责将数字信号转换为模拟信号（发送）以及将模拟信号还原为数字信号（接收）。常见型号如Realtek RTL8211E等。
5. **MAC控制器**：媒体访问控制层，负责管理数据链路层与物理层之间的传输。

#### 19.1.2 连接方式

在嵌入式系统中，根据处理器（SoC）、MAC控制器与PHY芯片的集成度不同，网络设备的硬件架构主要分为以下三种方案：

**方案一：全集成方案（CPU内联MAC与PHY）** 在此方案中，SoC内部集成了MAC控制器和PHY物理层芯片。

- **优点**：能够最大程度地减少外围元器件数量，简化PCB布线，降低系统整体功耗。
- **缺点**：由于PHY属于模拟电路，集成在SoC内部会显著增加芯片的工艺复杂度和成本，且容易受到数字电路的电磁干扰。此外，该方案缺乏灵活性，用户无法根据需求更换不同性能的PHY芯片。因此，在高性能嵌入式平台中较为少见。

**方案二：主流方案（CPU集成MAC，外挂独立PHY）** 这是目前工业界最主流的实现方式。SoC内部集成MAC控制器，外部通过标准的媒体独立接口（如MII、RMII、RGMII等）连接独立的PHY芯片，硬件连接如图19.6所示。

- **特点**：该方案在芯片成本与系统灵活性之间取得了最佳平衡。开发者可以根据应用场景（如百兆或千兆网络）灵活选择不同厂商的PHY芯片。
- **实例**：本书配套的**iTOP-RK3568开发板**即采用了此种方案，其内部MAC通过RGMII接口驱动外部PHY芯片。

图 19.6 CPU集成MAC外接PHY示意图

**方案三：外挂方案（CPU不集成MAC与PHY）** 当SoC本身不具备以太网控制器（MAC）时，需采用独立的MAC+PHY芯片或集成式网络芯片。

- **特点**：此类设备通常通过USB、SPI、PCIe等通用总线接口与CPU进行数据交换，如图19.7所示。这种方式虽然增加了外设成本和系统开销，但为不带网络功能的处理器提供了扩展能力。
- **实例**：**iTOP-4412开发板**上采用的即是此类方案（通过USB总线转接以太网）。

### 19.2 以太网PHY接口

在嵌入式网络开发中，目前最主流的硬件架构如图19.6所示。在此架构中，PHY芯片位于MAC控制器与物理介质（RJ45插座）之间，起到了承上启下的“桥梁”作用。MAC控制器与PHY芯片之间通过媒体独立接口（Media Independent Interface）进行数据交互，根据传输速率、引脚数量以及时钟同步方式的不同，常见的接口标准包括MII、RMII、GMII、RGMII以及用于管理配置的MDIO接口。

#### 19.2.1 MII接口

MII（Media Independent Interface）即“媒体独立接口”，是最初定义的标准以太网接口。它将MAC层与物理层（PHY）解耦，使得MAC控制器可以连接不同物理介质的PHY芯片。

- **传输速率**：MII接口支持10Mbps和100Mbps两种工作模式。
- **时钟同步**：该接口采用双时钟架构，发送（TX_CLK）和接收（RX_CLK）时钟均由PHY芯片产生并提供给MAC控制器。在100Mbps速率下，工作时钟为25MHz；在10Mbps速率下，工作时钟为2.5MHz。
- **硬件特性**：MII接口通常需要16根信号线（包括数据线、控制线及状态线），其典型的硬件连接示意如图19.8所示。

PHY芯片与MAC控制器之间需要通过特定的接口进行数据交互，常见的接口协议包括MII、RMII、GMII、RGMII等。

#### 19.2.1 RGMII接口

RGMII（Reduced Gigabit Media Independent Interface）是目前千兆以太网最常用的简化接口。它将引脚数量从GMII的25个减少到了14个，在1000Mbps速率下时钟频率为125MHz。

| ---------- | ----------------------------- |
| TX_ER      | 发送错误，高电平有效          |
| TXD[3:0]   | 发送数据（4 根）              |
| TX_EN      | 发送使能                      |

## Linux 网络设备驱动分层架构详解

Linux 网络设备驱动遵循分层设计原则，旨在通过硬件操作的抽象化实现网络协议栈与底层硬件的解耦。整个架构由上至下分为四层：

### 1. 网络协议接口层

该层为上层协议栈（如 TCP/IP）提供统一的收发数据入口。无论上层协议如何变化，均通过以下两个核心函数进行交互：

- **发送数据**：`dev_queue_xmit(struct sk_buff *skb)`
- **接收数据**：`netif_rx(struct sk_buff *skb)`

#### 核心数据结构：`sk_buff`

`sk_buff`（Socket Buffer）是 Linux 网络子系统的灵魂，它贯穿于协议栈的各层，充当数据包的载体。其核心设计思想是通过指针操作避免昂贵的数据拷贝。

**内存布局关键指针：**

- **`head` / `end`**：定义了整个分配给数据包的物理缓冲区的边界。
- **`data` / `tail`**：定义了当前协议层有效数据的起始和结束位置。

> **原理：** 当数据包下行（发送）时，各层协议只需通过移动 `data` 指针并填充协议头（Header）；当数据包上行（接收）时，各层只需剥离协议头。这种“零拷贝”机制极大地提升了网络性能。

C

```
struct sk_buff {
    struct sk_buff     *next;           // 双向链表后继
    struct sk_buff     *prev;           // 双向链表前驱
    
    union {
        struct net_device *dev;         // 关联的网络设备
    };

    unsigned int       len,             // 数据总长度（包括分片）
                       data_len;        // 分片中的数据长度
    
    __be16             protocol;        // 链路层协议类型
    __u16              transport_header;// 传输层头部偏移
    __u16              network_header;  // 网络层头部偏移
    __u16              mac_header;      // MAC层头部偏移

    sk_buff_data_t     tail;            // 指向有效数据尾部
    sk_buff_data_t     end;             // 指向内存缓冲区尾部
    unsigned char      *head,           // 指向内存缓冲区头部
                       *data;           // 指向有效数据头部
};
```

### 1. 网络协议接口层

网络协议接口层是 Linux 网络子系统的“统一门户”，其核心作用是向协议栈屏蔽底层硬件的差异，实现协议栈与硬件驱动的完全解耦。

不论上层运行的是何种协议（如 TCP、UDP 或 ICMP），网络协议接口层均通过一套标准化的 API 负责数据的传递：协议栈通过 **`dev_queue_xmit()`** 函数将数据包下发给驱动，而驱动程序则在接收到硬件数据后，通过 **`netif_rx()`** 函数将数据包递交给上层。这种设计为下层驱动提供了统一的数据收发入口，其函数原型如代码清单 19.1 所示。

**代码清单 19.1**

```c
int dev_queue_xmit(struct sk_buff *skb); // 协议栈调用，用于发送数据包
int netif_rx(struct sk_buff *skb);       // 驱动程序调用，负责接收数据包并推入上层
```

上述两个函数均使用 **`struct sk_buff`** 指针作为核心参数。该结构体定义在 `include/linux/skbuff.h` 中，是 Linux 网络子系统最核心的数据结构，充当着网络数据包在各层之间传递的“载体”。

`sk_buff` 的设计精髓在于其**“零拷贝”数据管理机制**。它本身并不直接分配一块连续的内存来存储数据，而是通过 `head`、`data`、`tail` 和 `end` 四个关键指针来动态管理缓冲区。这种机制允许各层协议在处理数据包时，只需简单地移动指针即可完成协议头（Header）的添加或剥离，从而避免了在内存中频繁复制数据的开销。

- **发送数据包时**：内核网络模块会预留出足够的空间创建 `sk_buff`。当数据包自上而下经过各层协议时，通过向上移动 `data` 指针并填充协议头（这一过程称为封装），直到最终交给网络设备发送。
- **接收数据包时**：驱动程序将硬件接收到的原始数据封装进 `sk_buff` 并向上传递。上层协议通过向下移动 `data` 指针来逐层剥离协议头（这一过程称为解包），最终将纯净的载荷数据交给用户态程序。

**代码清单 19.2**

### 19.6.4 数据收发流程

在 Linux 内核中，**`ndo_start_xmit()`** 回调函数是驱动程序处理发送任务的核心入口。每当上层协议栈有数据包下发时，系统会通过调用 `net_device_ops` 中的该函数来触发驱动的发送操作。驱动程序需在该函数内实现将 `sk_buff` 数据映射到硬件发送队列的具体逻辑。以 CAN 网络设备驱动为例，其发送流程在 `rockchip_canfd_start_xmit()` 函数中实现，详见代码清单 19.15。

#### 1. 数据发送（TX）

发送流程由内核协议栈发起。当上层有数据包需要发送时，会回调驱动中的 **`ndo_start_xmit()`** 函数。驱动程序需要在此函数中将 `sk_buff` 里的数据提取出来，并写入网卡的发送寄存器或 DMA 描述符中。

在 `rockchip_canfd_start_xmit` 的实现中，驱动首先通过 `netif_stop_queue()` 暂停发送队列以防止并发竞争，随后根据 CAN 帧类型（标准帧/扩展帧）配置硬件寄存器并触发发送命令，最后通过 `can_put_echo_skb()` 处理本地回环。

#### 2. 数据接收（RX）

网络设备的接收通常是异步的。为了在高负载下平衡性能与实时性，Linux 推荐使用 **NAPI（New API）机制**，即“中断 + 轮询”的混合模式：

- **中断阶段**：当硬件收到数据包时触发中断。在中断处理函数中，驱动暂时**禁用硬件中断**，并调用 **`napi_schedule()`** 唤醒轮询任务。
- **轮询阶段（Poll）**：内核在软中断上下文中调用驱动的 `poll` 函数。驱动在该函数内批量读取硬件接收缓冲区中的数据，并调用 `netif_receive_skb()` 将封装好的 `skb` 提交给协议栈处理。

**代码清单 19.16（NAPI 接收逻辑）**

C

```
/* 中断处理：仅触发 NAPI 调度 */
static irqreturn_t rockchip_canfd_interrupt(int irq, void *dev_id)
{
    if (irq_status & CAN_RX_IRQ) {
        rockchip_canfd_disable_rx_int(rcan); // 禁用中断，防止中断风暴
        napi_schedule(&rcan->napi);          // 调度 Poll 过程
    }
    return IRQ_HANDLED;
}

/* NAPI 轮询：批量处理数据包 */
static int rockchip_canfd_poll(struct napi_struct *napi, int budget)
{
    while ((skb = rockchip_canfd_receive_skb(rcan)) && work_done < budget) {
        skb->protocol = eth_type_trans(skb, ndev);
        netif_receive_skb(skb); // 递交协议栈
        work_done++;
    }
    
    if (work_done < budget) {
        napi_complete_done(napi, work_done);
        rockchip_canfd_enable_rx_int(rcan, true); // 数据处理完，重新使能中断
    }
    return work_done;
}
```

通过 NAPI 机制，系统可以在大流量压力下有效减少 CPU 被频繁中断干扰的情况，大幅提升了系统的网络整体吞吐量。

```
#define INDIO_BUFFER_HARDWARE      0x08 // 硬件缓冲模式
#define INDIO_EVENT_TRIGGERED      0x10 // 事件触发模式
#define INDIO_HARDWARE_TRIGGERED   0x20 // 硬件触发模式
```

这些模式决定了用户空间访问设备数据的方式：

- **直接模式 (DIRECT_MODE)**：不对数据进行缓存，适用于单次采样。用户通过访问 sysfs 属性文件（如 `in_voltageX_raw`）获取实时数据。
- **缓冲模式 (BUFFER_XXX)**：支持数据缓存，适用于连续的高频采样。用户需通过访问 `/dev/iio:deviceX` 字符设备文件读取连续的数据流。
- **事件模式 (EVENT_TRIGGERED)**：支持阈值报警功能，例如当传感器检测到温度超过上限时主动上报事件。
- **硬件触发模式 (HARDWARE_TRIGGERED)**：支持由外部硬件信号（如 GPIO 电平变化）来触发驱动进行数据采集。

虽然 ADC、电流计、温湿度传感器等不同设备都统一使用 `iio_dev` 框架，但 IIO 子系统依靠 **`iio_chan_spec`** 结构体来精确区分和描述这些物理设备。该结构体详细定义了每个通道的物理类型、索引及采样属性。内核正是通过 `type` 字段（如 `IIO_VOLTAGE`、`IIO_TEMP`）结合通道索引，来识别并生成对应的 sysfs 节点。`iio_chan_spec` 的核心成员如代码清单 20.3 所示。

### 一、正确的内容要点

1. **扩展遥控帧的仲裁字段构成**

   扩展帧的仲裁字段确实包含**29 位标识符 ID**（由 11 位基础 ID + 18 位扩展 ID 组成）、**1 位 SRR 位（替代远程请求位）\**和\**1 位 IDE 位（扩展标识符位）**，这一结构描述是正确的。

2. **SRR 位的作用与电平特性**

   SRR 位在扩展遥控帧中为**隐性 1**，其作用是替代标准帧中 RTR 位的位置。当标准帧（数据帧，RTR=0，显性）与扩展帧的 11 位基础 ID 相同时，由于显性位（0）在 CAN 总线仲裁中优先级高于隐性位（1），因此扩展帧的仲裁优先级会低于标准数据帧，这部分逻辑描述正确。

3. **IDE 位的标识作用**

   IDE 位为**隐性 1**时，用于明确标识该帧为**扩展帧**；若 IDE 位为显性 0，则标识为标准帧，这一规则符合 CAN 协议规范。

### 二、关键错误点

**“控制段则与标准数据帧和扩展数据帧中的控制字段相同” 表述错误**。

标准帧与扩展帧的**控制段结构存在明显差异**，具体区别如下：

- **标准帧的控制段**：共 6 位，包含`IDE位（显性0，固定）` + `r0（保留位，1位）` + `DLC（数据长度码，4位）`。

- 扩展帧的控制段

  ：共 6 位，包含

  ```
  r1（保留位，1位）
  ```

   

  +

   

  ```
  r0（保留位，1位）
  ```

   

  +

   

  ```
  DLC（数据长度码，4位）
  ```

  。

  

  （扩展帧的 IDE 位属于

  仲裁字段

  ，而非控制段；控制段中无 IDE 位，替换为保留位 r1，这是与标准帧控制段的核心区别）

此外，**扩展遥控帧与扩展数据帧的控制段虽结构一致（均为 r1+r0+DLC），但与标准数据帧的控制段（IDE+r0+DLC）结构不同**，原文未区分这一关键差异。

### 三、补充 “错误帧” 的核心知识点（原文仅标注标题无内容）

错误帧是 CAN 总线的异常处理帧，用于在检测到总线错误时通知总线上的其他节点，其结构和特性如下：

1. 组成

   ：由

   错误标志

   （主动错误标志 / 被动错误标志）和

   错误界定符

   两部分组成。

   - 主动错误标志：6 位显性位（主动错误节点发送，强制打断当前帧）；
   - 被动错误标志：6 位隐性位（被动错误节点发送，不强制打断）；
   - 错误界定符：8 位隐性位，用于总线恢复同步。

2. **触发条件**：节点检测到位错误、填充错误、CRC 错误、应答错误、格式错误等时，会立即发送错误帧。

3. **特性**：错误帧无标识符，不参与总线仲裁，且会破坏当前传输帧的完整性，触发相关节点的重传机制（若为主动错误节点）。

### 修正后的完整表述参考

在扩展遥控帧中，仲裁字段由 29 位标识符 ID（11 位基础 ID+18 位扩展 ID）、1 位替代远程请求位 SRR 和 1 位扩展标志位 IDE 组成。其中，SRR 位为隐性 1，替代了标准帧中 RTR 位的位置，当标准帧与扩展帧的 11 位基础 ID 相同时，扩展帧的仲裁优先级低于标准数据帧；IDE 位为隐性 1，用于标识该帧为扩展帧。

扩展遥控帧的控制段为 6 位（r1+r0+DLC），与扩展数据帧的控制段结构一致，但与标准数据帧的控制段（IDE+r0+DLC）存在明显差异（标准帧控制段含 IDE 位，扩展帧控制段为保留位 r1）。
