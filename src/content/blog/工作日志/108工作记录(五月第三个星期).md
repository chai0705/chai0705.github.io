● 我仔细看了整个项目，功能很全（Qt GUI、RKLLM 推理、Web 端、语音 ASR/TTS、硬件控制、状态监控）。以下是按优先级排序的优化建议：

---
  🔴 高优先级

  1. handleSend() 和 handleWebRequest() 意图匹配逻辑大量重复（~150 行重复代码）

  两个函数里有一模一样的关键字匹配逻辑，维护时要改两处。建议提取为公共函数：

  // 在 chat_window.h 中新增
  struct IntentResult {
      bool isLocal = false;
      QString response;
  };

  IntentResult parseIntent(const QString &command);

  然后两个 handle 函数都调用它。

  2. Whisper 语音识别阻塞主线程（UI 卡顿）

  onVoiceBtnReleased() 第 537 行：
  whisperProcess.waitForFinished(8000);  // 阻塞 UI 8 秒！

  用户在点击"按住说话"松开后，界面会卡死 8 秒。应该把 ASR 放到 LLMWorker 线程或者用 QThreadPool 异步执行。

  3. 对话历史无限增长，最终会内存溢出

  conversationHistory 和 fullMarkdownLog 只追加不清理。长时间运行后：
  - 每次推理都要发送越来越长的 prompt → NPU 负载越来越高、响应越来越慢
  - setMarkdown() 渲染越来越大的文档 → UI 越来越卡

  建议加轮数限制：
  const int MAX_HISTORY_ROUNDS = 10;
  // 超过时裁剪，保留系统设定 + 最近 N 轮

  4. 模型路径多处硬编码

  以下路径分散在多个文件，建议集中到配置文件或 config.h：
  - chat_window.cpp:132: /rkllm/model/qwen3-vl-4b-instruct_w8a8_rk3588.rkllm
  - llm_worker.cpp:38: /rkllm/model/qwen3-vl-4b-instruct_w8a8_rk3588.rkllm
  - chat_window.cpp:532: /rkllm/model/ggml-base.bin

---
  🟡 中优先级

  5. fullMarkdownLog 每次 setMarkdown() 全量重渲染

  长对话时越来越卡。可以改用 QTextCursor 增量插入：
  QTextCursor cursor(chatDisplay->document());
  cursor.movePosition(QTextCursor::End);
  cursor.insertMarkdown(text);  // Qt 6.4+ 支持

  6. TTS killall -9 edge-tts mpg123 过于暴力

  每次打断都用 killall -9 强制杀进程，可能导致音频子系统状态异常。建议用更优雅的方式：
  // 维护一个 QProcess* ttsProcess 指针
  if (ttsProcess && ttsProcess->state() == QProcess::Running) {
      ttsProcess->terminate();
      ttsProcess->waitForFinished(500);
  }

  7. getAllNpuUsage() 读取 /sys/kernel/debug/ 需要 root

  普通用户运行时会静默失败返回 0。可以加权限检查或提示：
  if (!file.exists()) {
      qDebug() << "NPU load 需要 root 权限访问 /sys/kernel/debug";
  }

  8. Web 服务器没有访问控制

  任何能访问该设备 IP:8080 的人都可以控制硬件。建议至少加一个简单的 Token 校验或本地回环限制（只允许 127.0.0.1 和局域网 IP）。

  9. smart_control.sh 中 echo 0/1 直接写 GPIO 路径

  不同板子的 GPIO 编号可能不同，脚本目前是写死的。如果换一块 RK3588 开发板可能需要改脚本。

---
  🟢 低优先级 / 代码风格

  10. CMakeLists.txt 链接了 -lpthread

  Qt 自己的线程库已经够用了，且 QThread 内部已经处理了线程支持。这个不是必须的（但无害）。

  11. chat_window.cpp 中 QRegExp 已废弃

  Qt5 开始 QRegExp 被标记为废弃，建议用 QRegularExpression 替代：
  // 旧
  QRegExp rx("(\\d+)");
  // 新
  QRegularExpression rx("(\\d+)");

  12. main.cpp 中 setrlimit(RLIMIT_NOFILE, 2048, 4096) 不够大

  你同时在 autorun.sh 的 apply_ulimit() 里设置了 102400。两者取较小值生效，建议统一配置。

客户项目需求：

**内核版本与实时性**：锁定 Linux 5.10.160 或 6.1 LTS 分支，必须提供 PREEMPT_RT 补丁版本。验收时使用 cyclictest 在 CPU 满载下测试，最大延迟需控制在 200μs 以内，确保 ROS2 控制周期不抖动。

**编译构建体系**：交付完整 SDK 源码，包含 U-Boot、Kernel、Buildroot/Debian 根文件系统。必须支持纯净环境下一键编译（`./build.sh all`），严禁依赖主机特定环境变量，且工具链需内置于 prebuilts 目录。

**设备树（DTS）适配**：根据定制板卡原理图深度修改 DTS。PMIC（RK806）电压电流限制需匹配 PCB 载流能力；DDR 时序需提供全温范围（-40℃~85℃）眼图扫描报告；外设引脚复用（pinctrl）需与硬件走线一致，无冲突。

**多媒体零拷贝通路**：V4L2 驱动必须暴露 DMABUF 接口，支持 io-mode=dmabuf。验收标准：4K@60fps 视频流从 v4l2src 到 kmssink/NPU，CPU 占用率低于 15%。驱动内部需自动处理 16x16 像素对齐及 64 字节 DMA 地址对齐，杜绝 CIF_ISP_PIC_SIZE_ERROR 与 ALIGN_ERR。(==没有4K 60的摄像头==)

**NPU 驱动与内存预留**：RKNN Toolkit2 支持版本需 ≥ v1.4.0。系统启动时需在 DTS 中为 NPU 预留 ≥ 256MB 专用 CMA 内存池，防止长期运行后推理失败。`/dev/rknn` 节点权限需配置为 666 或归属 video 组，支持非 root 用户调用。==现在的NPU调用使用的不是设备节点==

**外设接口映射**：CAN 驱动必须注册为 SocketCAN（netdevice），支持 ip link 命令配置波特率，严禁仅提供字符设备。UART 驱动需优化 FIFO 阈值，921600 波特率下连续传输 1 小时丢包率为 0。

**ROS2 兼容性**：BSP 需预装 rmw_cyclonedds_cpp 并设为默认 RMW。内核参数需预设 `kernel.shmmax=536870912`，满足 DDS 共享内存需求。提供 `ros2_test_suite` 脚本，同时启动 talker/listener、RViz2 点云显示及 NPU 推理节点，运行 24 小时无 DDS 发现失败日志。

**工业级可靠性**：驱动需通过 -40℃ 低温静置 24 小时启动测试，及 85℃ 高温满负荷（stress-ng）运行 72 小时无看门狗复位。USB/以太网接口在高频振动下驱动不挂死，拔插后能自动恢复 link 状态。

**交付物规范**：拒绝仅提供 .img 镜像。必须交付包含所有自定义驱动源码的完整 SDK，提供 U-Boot + PMON 双固件支持底层诊断。驱动修改处需通过 configfs/sysfs 暴露配置节点，支持应用层动态调整参数而无需重编内核。（==这里为啥会出现PMON==）

static int phy_zxp211f_led_fixup(struct phy_device *phydev)
{
        int val;
        phy_write(phydev, 0x1e, 0xa00d);
        phy_write(phydev, 0x1f, 0x70);
        phy_write(phydev, 0x1e, 0xa00e);
        phy_write(phydev, 0x1f, 0x607);

        /*关闭智能降速机制*/
        val = phy_read(phydev, 0x14);
        if (val >= 0) {
                val &= ~0x0020;
                phy_write(phydev, 0x14, val);
        }

        /*执行软件复位使0x14的配置生效*/
        val = phy_read(phydev, 0x00);
        if (val >= 0) {
                val |= 0x8000;
                phy_write(phydev, 0x00, val);
        }
        return 0;
}

# RK3588 AI 智能中枢 · 项目介绍与使用手册

> 一台跑在 **RK3588 NPU** 上的本地大模型对话与硬件控制网关。
> Qt5 桌面端 + Web 端双通道，离线推理，毫秒级硬件直通。

---

## 一、项目简介

**rkllm_chat_gui** 是面向 RK3588 系列开发板（如 iTOP-RK3588）的演示级 AI 应用，把"大语言模型 + 智能家居硬件控制"做成了一个开箱即用的桌面终端。整套对话推理 **100% 在板上的 6 TOPS NPU 完成**，不依赖任何云端 API，特别适合展会演示、教学样例和嵌入式 AI 项目原型。

### 核心能力

| 能力 | 实现要点 |
| :--- | :--- |
| 🧠 本地大模型对话 | RKLLM 运行时加载 `qwen3-vl-4b-instruct W8A8` 量化模型，三核 NPU 并行推理，~30 tokens/s |
| ⚡ 自然语言硬件控制 | 内置意图引擎，"开灯 / 调亮屏幕 / 播放音乐" 等命令**绕过大模型**，毫秒级直驱 GPIO / 背光 / 音频 |
| 🌡️ 实时环境感知 | DHT11 温湿度传感器读数，可通过对话或快捷指令查询 |
| 🎤 语音输入 (ASR) | 按住说话 → `whisper-cli` 离线转写 → 自动发送 |
| 🔊 语音播报 (TTS) | Edge-TTS 微软晓晓（**需联网**，可一键关闭进入完全离线模式） |
| 📱 Web 端双通道 | 同进程开 8080 端口，手机扫码即可同局域网访问，多人并发 |
| 🎬 演示模式 | 30 秒无人操作自动循环播放预置话术，展会无人值守 |
| 🌐 中英双语 + 🌗 浅深主题 | 一键切换，适配海外客户与不同光线环境 |
| 📊 性能仪表盘 | CPU / RAM / NPU 三核占用 / SoC 温度 / 首字延时 / Tokens-per-second 实时可视 |

## 二、硬件与软件需求

### 硬件

- **核心板**：Rockchip RK3588（推荐 iTOP-RK3588 开发板，4×Cortex-A76 + 4×Cortex-A55 + 3 核 NPU @ 6 TOPS）
- **外设**（按需）：
  - LED：板载 `work1` / `work2` 指示灯（路径 `/sys/devices/platform/leds/leds/`）
  - 蜂鸣器：板载，由 `/topeet_test/03_beep/topeet_beep` 驱动
  - DHT11 温湿度传感器：挂在 IIO 设备节点
  - ES8388 音频编解码（板载，自动探测）
  - 显示屏 + 背光（`/sys/class/backlight/backlight/brightness`）

### 软件

- Debian / Ubuntu （板厂出厂系统即可）
- Qt5（Widgets + Network）、CMake、g++、Python3-pip
- `mpg123`（TTS 播放）、`qrencode`（手机扫码）、`fonts-noto-color-emoji`（emoji 渲染）

> 所有依赖会被 `autorun.sh` 自动安装，正常情况下**无需手动处理**。

## 十二、常见问题排查

### Q1：界面没有全屏 / 只是窗口
- 检查启动方式：必须是 `sudo rkllm_chat` 或桌面快捷方式（已 sudo）
- 程序默认 `showFullScreen()`，**Esc 会退到窗口态**，按 **F11** 重新进全屏

### Q2：emoji 显示成方框
- 缺 `fonts-noto-color-emoji` 包，运行 `sudo apt install -y fonts-noto-color-emoji && sudo fc-cache -f`
- 已经装了还不行：可能 fontconfig 缓存没刷新，再执行一次 `fc-cache -f`

### Q3：语音播报无声音
- 看 `/tmp/tts_err.log`，常见原因：
  - **edge-tts 没装** → `pip3 install --user edge-tts`，确保 `/usr/local/bin/edge-tts` 软链存在
  - **没联网** → Edge-TTS 是云端服务，断网会失败。可点 🔊 按钮关闭播报
  - **声卡识别错误** → 启动日志会有 `🔊 已锁定 ES8388 声卡号：X`，错了就改 `chat_window.cpp::detectAudioCard`

### Q4：LED / 蜂鸣器不响应
- 看 `/tmp/smart_gateway.log` 有没有报错
- 确认 `/topeet_test/03_beep/topeet_beep` 存在且可执行
- 确认 GPIO 节点存在：`ls /sys/devices/platform/leds/leds/`

### Q5：模型加载失败 / NPU 报错
- 确认 `/rkllm/model/qwen3-vl-4b-instruct_w8a8_rk3588.rkllm` 存在且大小约 **4.5 GB**
- 确认 NPU 驱动加载正常：`cat /sys/kernel/debug/rknpu/version`
- 程序底部会弹"⚠️ AI 大模型加载失败"，**但硬件直通指令仍可用**

### Q6：Web 端打不开
- 确认手机和板子**在同一局域网**
- 确认 8080 端口没被占用：`sudo netstat -tlnp | grep 8080`
- 防火墙：`sudo ufw allow 8080` 或直接关闭 ufw

  优先建议                                                                                                                                                                                         
                                                                                                                                                                                                   
  1. 不要做图片识别，反而建议换掉 VL 模型                                                                                                                                                          
     现在模型名是 qwen3-vl-4b-instruct_w8a8_rk3588.rkllm，虽然你没用图片输入，但 VL 模型本身通常比纯文本模型更占资源。既然 RK3588 内存紧张，后续应该优先换成同级别或更小的纯文本 RKLLM 模型，比如  
     1.5B/3B/4B 文本模型，别保留视觉分支能力。
  2. 把硬编码路径改成配置文件                                                                                                                                                                      
     现在 /rkllm/model/...、/usr/local/bin/whisper-cli、8080、声卡号、TTS 开关、模型参数都散在代码里。建议增加 /etc/rkllm-chat/config.ini 或项目目录下 config.json，让部署后不用重新编译就能改模型 
     路径、端口、是否启用 TTS/ASR/Web。                                                                                                                                                            
  3. 增加统一推理 busy/队列机制                                                                                                                                                                    
     Web 端有 isBusy，桌面端有 isGenerating，但它们不是统一锁。极端情况下桌面和 Web 可能同时向同一个 RKLLM worker 发请求。建议做一个全局 InferenceController：统一状态、统一 abort、统一排队或直接 
     拒绝。                                                                                                                                                                                        
  4. Web 端离线化                                                                                                                                                                                  
     web_server.cpp 里 Web 页面依赖 jsdelivr 的 marked 和 highlight.js。这和“本地离线演示”定位冲突，断网时 Web 端 Markdown/代码高亮会受影响。建议把 JS/CSS 内置到本地资源，或者直接简化为纯文本/轻 
     量 Markdown 渲染。                                                                                                                                                                            
  5. 增加真正的 Web 停止生成接口                                                                                                                                                                   
     当前 Web 端 stopGen() 主要是取消浏览器 fetch，服务端推理未必停止。建议增加 POST /abort，调用 rkllm_abort()，让手机端停止按钮真正中断 NPU 推理。                                               
  6. 拆分 chat_window.cpp                                                                                                                                                                          
     这个文件已经 130KB+，承担 UI、i18n、硬件、TTS、ASR、Web 协调、性能监控。后续维护会越来越难。建议拆成：                                                                                        
      - intent_engine.*                                                                                                                                                                            
      - hardware_controller.*                                                                                                                                                                      
      - audio_asr.*                                                                                                                                                                                
      - tts_player.*                                                                                                                                                                               
      - system_monitor.*                                                                                                                                                                           
      - chat_window_ui.*                                                                                                                                                                           
  7. 做正式包装部署                                                                                                                                                                                
     现在 autorun.sh 已经能用，但更像开发部署脚本。实际项目建议补：                                                                                                                                
      - install.sh                                                                                                                                                                                 
      - uninstall.sh                                                                                                                                                                               
      - systemd service                                                                                                                                                                            
      - .desktop 文件不直接写 Exec=sudo ...                                                                                                                                                        
      - 日志目录改到 /var/log/rkllm-chat/                                                                                                                                                          
      - 程序安装到 /opt/rkllm-chat/，/usr/bin/rkllm_chat 只做软链                                                                                                                                  
  8. 减少运行时 shell 调用                                                                                                                                                                         
     每秒状态刷新里 RAM 调用 free，TTS/录音/硬件也大量 sh -c。在开发板上 shell 启动成本不算小。建议：                                                                                              
      - RAM 直接读 /proc/meminfo                                                                                                                                                                   
      - 背光、LED、温度等尽量 C++ 直接读写 sysfs                                                                                                                                                   
      - shell 脚本保留为兼容层，不作为主路径                                                                                                                                                       
  9. 内存优化方向                                                                                                                                                                                  
     不上图片识别的前提下，优先做：                                                                                                                                                                
      - 换纯文本小模型                                                                                                                                                                             
      - 降低 max_new_tokens，现在是 4096，演示场景可改成 512/1024                                                                                                                                  
      - TTS 默认关闭，避免网络和额外进程干扰演示

可以评估为：**从 STM32MP157A 换到 STM32MP157C，若封装同为 `AA3`，硬件大概率兼容，软件常规功能也大概率兼容；主要差异在安全启动和硬件加密能力。**

你们当前图片里是：

| 项目     | 当前                              | 客户想换                                  |
| -------- | --------------------------------- | ----------------------------------------- |
| SoC      | `STM32MP157AAA3`                  | `STM32MP157CAA3`                          |
| 安全等级 | `A = Basic security`              | `C = Secure boot + cryptography hardware` |
| 封装     | `AA = LFBGA448, 18x18 mm, 0.8 mm` | `AA = LFBGA448, 18x18 mm, 0.8 mm`         |
| 温度等级 | `3 = -40 到 +125 degC`            | `3 = -40 到 +125 degC`                    |

**硬件兼容性**

| 检查项                          | 结论                                                         |
| ------------------------------- | ------------------------------------------------------------ |
| PCB 封装                        | 兼容                                                         |
| 管脚                            | pin-to-pin 兼容                                              |
| 电源                            | 兼容                                                         |
| DDR/eMMC/SD/ETH/CAN/UART 等外设 | 接口资源一致                                                 |
| PMIC STPMIC1                    | 相同                                                         |
| 散热/功耗                       | 基本一致                                                     |
| 生产烧录                        | 如果不开启 Secure Boot，流程基本不变；如果要启用安全启动，需要新增 OTP/密钥/签名烧录流程 |

**软件兼容性**

| 软件层         | 结论                                                         |
| -------------- | ------------------------------------------------------------ |
| Linux 应用程序 | 基本无需改                                                   |
| M4 固件        | 基本无需改                                                   |
| 普通外设驱动   | 基本无需改                                                   |
| 设备树 DTS/DTB | 建议改成 C 型号对应配置，例如从 `stm32mp157a` 系列切到 `stm32mp157c` 系列，以便正确暴露 crypto/hash/rng 等硬件 |
| TF-A / U-Boot  | 常规启动大概率兼容，但建议按 `STM32MP157C` 重新配置/验证     |
| DDR 初始化     | 理论上与 A/C 无直接关系，更多取决于 DDR 颗粒和板级走线；但换 SoC 后仍建议完整跑 DDR stress test |
| Secure Boot    | 如果不开启，影响很小；如果开启，需要重做启动链、镜像签名、OTP 烧录、调试权限策略 |
| 硬件加密       | C 支持 AES/TDES/HASH/TRNG 等，A 没有；如果客户需要加密加速，软件需要启用对应驱动/配置 |

**主要风险点**

1. **不能只换芯片不改型号声明**  
   旧软件如果一直按 `STM32MP157A` 设备树跑，普通功能可能没问题，但 C 的加密、安全硬件可能不会被启用。建议 BSP 里明确切到 `STM32MP157C`。

2. **Secure Boot 是最大差异**  
   C 支持安全启动，但默认不等于必须启用。  
   如果客户只是想换成 C，不用安全启动，影响很小。  
   如果客户要求安全启动/防拷贝/固件签名，那就是一个独立项目，需要设计密钥、OTP、签名镜像、量产烧录和售后解锁流程。

3. **调试口可能被安全策略影响**  
   C 芯片如果烧了安全相关 OTP 或关闭调试权限，JTAG/SWD 行为会变化。工程样机阶段一定不要过早锁死。

4. **BOM 里其他物料也要一起确认**  
   你图里 DDR、eMMC 料号也不同。SoC A 换 C 本身风险不大，但 DDR/eMMC 换料会影响启动固件、DDR 参数、稳定性测试。

**建议结论**

如果客户只是要求 `STM32MP157A` 换成 `STM32MP157C`，并且仍然使用 `AA3` 封装：  
**硬件上可以按兼容替换评估，不需要改 PCB 的概率很高。软件上普通功能兼容，但建议 BSP/设备树切到 C 型号并重新做整板验证。**

建议验证清单：

| 验证项               | 必测                             |
| -------------------- | -------------------------------- |
| TF-A 启动            | 是                               |
| U-Boot 启动          | 是                               |
| Linux 启动           | 是                               |
| DDR stress test      | 是                               |
| eMMC/SD 启动和读写   | 是                               |
| Ethernet             | 是                               |
| UART/CAN/USB         | 是                               |
| M4 固件加载          | 是                               |
| 休眠/唤醒            | 建议                             |
| crypto/hash/rng 驱动 | 如果客户要用安全/加密功能则必测  |
| Secure Boot          | 如果客户要安全启动则单独立项验证 |

一句话给客户可以这样说：  
**STM32MP157AAA3 替换为 STM32MP157CAA3，在相同 AA3 封装下硬件基本兼容，普通软件功能基本兼容；差异主要是 C 型号增加 Secure Boot 和硬件加密能力，若启用安全功能则需要调整 BSP、启动链和量产烧录流程。**
