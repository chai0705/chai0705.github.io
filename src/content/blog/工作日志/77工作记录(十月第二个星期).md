## 一、先搞明白什么是MIPI

**MIPI** = Mobile Industry Processor Interface（移动行业处理器接口）

**简单说**：MIPI就是手机/平板里各部件之间通信的"语言"。

想象一下：
- 摄像头拍的照片需要传给手机处理器
- 处理器处理完后需要传给屏幕显示
- 没有MIPI，就像不同国家的人说不同语言，没法交流

MIPI联盟制定了统一标准，让摄像头、屏幕、处理器等部件能互相"说话"，就像全世界都用英语交流一样。

## 二、MIPI接口的常见类型

在MIPI标准中，有几种常用接口：

1. **MIPI CSI**：摄像头和处理器之间的通信接口（"摄像头接口"）
2. **MIPI DSI**：处理器和屏幕之间的通信接口（"屏幕接口"）
3. **MIPI D-PHY**：物理层标准（数据怎么在物理线上传输）
4. **MIPI C-PHY**：另一种物理层标准

## 三、重点：DPHY vs CPHY（物理层标准）

这是最核心的区别，我用最简单的比喻来解释：

### 1. DPHY（D-PHY）

- **像什么**：普通的高速公路（4车道）
- **特点**：
  - 每个通道需要4根线（2对差分线，每对2根线）
  - 传输速度：约2.5-4.5Gbps/每根线（速率是每条线的速度）
  - 简单、成熟、常用
- **比喻**：就像一条4车道的高速公路，每辆车（数据）占用1个车道

### 2. CPHY（C-PHY）

- **像什么**：更高效的高速公路（3车道，但每车道运载能力更强）
- **特点**：
  - 每个通道只需要3根线（比DPHY少1根线）
  - 传输速度：约2.5Gsps/每根线（速率是每条线的速率，但带宽效率更高）
  - 更复杂，但效率更高（CPHY的带宽是DPHY的约2.3倍）
- **比喻**：就像一条3车道的高速公路，但每车道能运载更多车辆，总运载能力反而比4车道的高速公路还高

### 为什么要有两种？

因为不同摄像头传感器支持不同的接口：
- 有些摄像头只支持DPHY（就像有些车只能在4车道高速上跑）
- 有些摄像头支持CPHY（就像有些车能在3车道高速上跑，但需要特殊设计）

RK3588的MIPI接口设计为**同时支持DPHY和CPHY**，这样它可以兼容更多类型的摄像头。

## 四、什么是Lane（通道/车道）？

**Lane**（通道）是MIPI接口的基本传输单位，就像高速公路的车道。

- 一个Lane = 2根线（差分信号，一根正，一根负）
- 1个Lane可以传输一定量的数据
- 2个Lane = 4根线，可以同时传输2倍的数据
- 4个Lane = 8根线，可以同时传输4倍的数据

**简单说**：Lane就是数据传输的"车道"，车道越多，数据传输能力越强。

## 五、为什么需要多个Lane？

摄像头需要传输大量数据，尤其是高清视频：
- 1080p视频：需要约2.5Gbps的数据传输速率
- 4K视频：需要约10Gbps的数据传输速率
- 8K视频：需要约20Gbps的数据传输速率

所以，为了满足不同分辨率摄像头的需求，MIPI接口设计为：
- 1 Lane：适合低分辨率摄像头
- 2 Lane：适合1080p摄像头
- 4 Lane：适合4K/8K摄像头

## 六、RK3588的MIPI接口具体怎么用？

RK3588的摄像头接口设计很灵活：

1. **两个MIPI DC接口**（可以是DPHY或CPHY）
   - 每个DC接口可以工作在：
     - DPHY模式：4 Lane（4×4.5Gbps = 18Gbps）
     - CPHY模式：3 Lane（3×2.5Gsps = 7.5Gsps，带宽约17.5Gbps）

2. **额外的四个MIPI CSI DPHY**
   - 每个CSI DPHY是2 Lane（2×2.5Gbps = 5Gbps）
   - 可以组合成4 Lane（2×2 Lane = 4 Lane）

## 七、用实际例子说明

假设我们要连接6个摄像头：
- 每个摄像头需要2 Lane（1080p视频）
- 6个摄像头需要12 Lane

RK3588的MIPI接口可以这样配置：
- 2个MIPI DC接口，每个用2 Lane（CPHY模式）：2×3 Lane = 6 Lane（CPHY每Lane带宽高）
- 4个MIPI CSI DPHY，每个2 Lane：4×2 Lane = 8 Lane
- 总共：6+8=14 Lane，足够支持6个摄像头

## 八、总结成一句话

> **MIPI是手机内部各部件通信的"语言"；DPHY和CPHY是两种"说话方式"，DPHY像4车道高速公路，CPHY像3车道高效高速公路；Lane是"车道"，车道越多，传输数据越快。**

## 九、小白常见疑问解答

**Q：为什么CPHY能用更少的线达到更高的带宽？**
A：CPHY不是每根线传得更快，而是通过更复杂的编码方式，让每根线能传输更多数据。就像3车道的高速公路，虽然车道少，但每车道能运载更多车辆。

**Q：DPHY和CPHY哪个更好？**
A：没有绝对的好坏，取决于摄像头传感器支持哪种。CPHY效率更高，但需要更复杂的硬件支持。DPHY更简单成熟，目前用得更多。

**Q：Lane越多越好吗？**
A：是的，但也要看摄像头需求。如果摄像头只需要2 Lane，用4 Lane是浪费。RK3588的设计是灵活的，可以根据摄像头需求配置Lane数量。

## 一句话总体概览

多路摄像头（左边）把高速串行或并行的视频信号送进「PHY + CSI 接收」→ 由 VICAP（视频采集单元）汇聚并写入 DDR（帧缓冲）→ ISP（图像信号处理器）对原始图像做去噪/去马赛克/白平衡/色彩校正等 → 处理后或原始帧放到 DDR → 最后给编码器（FEC/encoder）压缩或送外设/网络。

## 各方框逐项解释（由左到右）

### 左侧：D/CPHY-0、D/CPHY-1、inno-dphy2L-0/1/…、DVP IO(CIF_IO)

这些是**物理层（PHY）端口**，也就是把摄像头模块输出来的电信号变成芯片内部可识别的数据的电路。

- **D-PHY / C-PHY / inno-dphy2L**：常见于 MIPI CSI-2 摄像头接口。D-PHY、C-PHY 是 MIPI 的物理接口标准，负责把差分高速信号（几百 Mbps 到几 Gbps 每 lane）可靠地接入芯片。`inno-dphy2L` 看名字像厂商（Innolux）实现的 2-lane D-PHY。
- **DVP IO (CIF_IO)**：并行并旧式的视频接口（Digital Video Port / CIF），摄像头把每个像素在并行总线上送出来，同时给 HSYNC/VSYNC/PIXCLK。速度与带宽低于 MIPI，但实现简单。

**本质原理**：PHY 做的事是「物理层的收发」：把差分信号做时钟恢复并把串行位流还原成 CSI 数据帧，做 bit-level 的同步、去抖、错误检测等。

------

### CSI-Host 0..5

这些是**CSI-2 协议层（接收端）**的硬件模块。PHY 把比特流还原后，CSI-Host 负责把 MIPI CSI-2 的包（packet）解析成像素数据。

- CSI-2 协议把图像数据封装成 packet（short/long packets），可能带有帧 ID、虚拟通道（virtual channel）、数据类型（RAW10/12/14、YUV 等）。
- **CSI-Host 的职责**：解析包，按帧/行/像素重组数据，做错误检测（CRC）、处理虚拟通道、多路复用等，把“像素流”交给更高层（VICAP）。

**原理**：从协议层把串行数据拆包并整理成可被 DMA 写入内存的帧缓冲结构。

------

### VICAP（Video CAPture，或 Video Input Controller）

VICAP 是图中中央的大方块，作用是**多路视频输入的汇聚、同步、内存搬运（DMA）和基础格式转换**。

- 它把多个 CSI-Host / DVP 的数据做时序对齐、路由、可能做简单 crop/scale/format 转换，然后通过 DMA 把帧写到外部 DDR（或把 DDR 的帧送到 ISP）。
- VICAP 通常管理帧缓冲队列（frame ring buffer）、中断、帧头/元数据（例如 exposure/time stamp）。
- 对多摄像头系统，VICAP 还负责切换/合并、以及给 ISP 分配输入。

**原理**：硬件 DMA 控制器 + 缓冲管理，把高速流持续稳定地写入大容量内存，避免丢帧。

------

### DDR（外部 DRAM，图中紫色竖块）

系统的**帧缓冲存储器**。原因：

- 摄像头数据量大（4K/30fps、Raw12 等），片内 SRAM 容量不足，需要外部 DDR 存放整帧或多个帧。
- VICAP 把原始帧（raw）或 YUV 结果写入 DDR；ISP 从 DDR 读帧做处理或把处理结果写回 DDR；编码器（FEC）从 DDR 取数据编码。
- DDR 也用于临时缓存、回放、并行处理（比如一个 ISP 在处理第 N 帧，另一个 ISP 处理第 N+1 帧）。

**原理**：提供大量低成本存储，但访问延迟/带宽是系统性能瓶颈，需要精心管理（例如 ping-pong buffers / burst transfers）。

------

### ISP-0、ISP-1（Image Signal Processor）

ISP 是**图像处理的核心单元**，把传感器的原始数据（通常是 Bayer RAW）处理成可观看/编码的图像（RGB 或 YUV）。

常见的处理步骤（硬件流水线）：

1. **坏点/黑电平校正**（defect pixel/black-level）
2. **去噪**（temporal / spatial denoise）
3. **去马赛克（demosaic）**：Bayer -> RGB
4. **白平衡（AWB）**、色彩矩阵（color correction）
5. **曝光控制（AE）**、自动对焦（AF）统计（通常以统计信息反馈）
6. **镜头/色彩/畸变校正**（lens shading / distortion）
7. **伽马、色彩空间转换**（RGB -> YUV）
8. **缩放/裁剪/格式转换**
9. 输出格式可供显示/编码（例如 YUV420）。

- 图里有两个 ISP（ISP-0、ISP-1），说明 SoC 支持并行图像处理：可以同时处理多路摄像头或把高分辨率帧分片并行处理以加速。

**原理**：把传感器原始物理量（单色敏感像素）通过算法变成人眼或编码器友好的格式，很多步骤可以在专用硬件流水线上极高效地并行完成（硬件加速）。

------

### FEC-0、FEC-1（图右）

图中标注为 FEC，常见可能含义有两种——我会说明最可能的那种并给出另一种可能性：

1. **最常见（在视频流水线语境）**：FEC 很可能是 **Frame Encoder / Front-End Codec**（即视频编码器模块）。它读取处理后的 YUV/RGB 帧，把视频压缩成 H.264 / H.265 / JPEG 等格式以便存储或网络传输。编码器也通常是硬件加速的。
2. **另一可能**：FEC 也可指 **Forward Error Correction**（纠错）——这是通信领域的术语，但在这张图里更像是编码器/压缩单元的表示。

**原理**：把帧数据经过压缩算法减少带宽/存储，占用较少资源便于传输和长时间存储。

------

## 一帧图像的典型“走向”——步骤示意

1. 摄像头传感器把像素数据以 MIPI CSI-2（或 DVP）格式发出来（串行差分信号或并行信号）。
2. **PHY**（D-PHY/C-PHY）把电信号恢复成比特流，做时钟恢复。
3. **CSI-Host** 解析 CSI 数据包，把像素按帧/行/像素恢复成像素流并做错误检测。
4. **VICAP** 接收来自多个 CSI-Host 的流，做汇聚、格式确认并通过 DMA 把帧写入 **DDR**（frame buffer）。
5. ISP（可能由 VICAP 再触发）从 DDR 读取原始帧做一系列图像运算，输出处理后的帧到 DDR。
6. 编码器（FEC）从 DDR 拿处理后帧进行压缩（H.264/H.265 等）。
7. 压缩后数据送到存储器、网络或显示模块。
8. 过程中，CPU/驱动会通过中断/状态寄存器获得帧完成、错误、统计信息等，并可调整 ISP 参数（比如曝光、增益）。

------

## 一些关键概念解释（小白必读）

- **RAW/Bayer**：大多数摄像头传感器每个像素只采集一种颜色（R/G/B 的一个），排成马赛克（Bayer）。要得到彩色图像需要**去马赛克（demosaic）**。
- **Virtual Channel（虚拟通道）**：MIPI CSI-2 可以在一根物理链路上承载多个逻辑通道（例如多个流或 metadata）。
- **Lane（通道）**：PHY 上的并行差分对数，更多 lane = 更高带宽。
- **DMA（Direct Memory Access）**：VICAP/CSI/ISP 用 DMA 把帧直接搬到 DDR，降低 CPU 负载。
- **Ping-pong buffer**：为了避免冲突，常用双缓冲（一个写，一个读）交替工作，保证连续流畅不丢帧。
- **3A（AE/AWB/AF）**：自动曝光/自动白平衡/自动对焦，是 ISP 常有的闭环控制算法，基于图像统计数据调整参数。
- **带宽瓶颈**：DDR 的带宽和总线是瓶颈，多个高分辨率摄像头时必须保证 DDR/内存控制器能支撑。

------

## 软件/驱动方面（高层提示）

- 在 Linux/嵌入式上，通常会有 **PHY 驱动、CSI Receiver 驱动、VICAP 驱动、ISP 驱动、硬件编码器驱动**，并与 **V4L2** 框架集成。驱动负责缓冲队列（vb2）、DMA 映射、ioctl 控制等。
- 设备树（Device Tree）会描述哪些 PHY、CSI-Host、ISP 存在及它们的连接关系（和中断、时钟）。
- 用户层可以用 `v4l2-ctl` 或者 GStreamer 来采集/预览/编码。

------

## 为什么系统这样划分？（设计动机）

- **分工明确**：PHY 处理物理信号，CSI-Host 处理协议，VICAP 管理流/缓冲，ISP 做视觉处理，编码器做压缩。每一层都可专用硬件加速以提高吞吐。
- **可扩展性**：多个 CSI-Host / 多个 ISP / 多个 FEC，便于支持多摄像头或并行处理高分辨率视频。
- **效率**：大数据量直接靠 DMA 和硬件流水线处理，CPU 只做控制和策略，不拉低性能。

## 图例 / 基本概念（先把术语讲清）

- **实体（Entity）**：图里每个白色方块，比如 `csi2_dphy0`、`mipi0_csi2`、`rkcif_mipi_lvds0`，在 Linux media 子系统里就是一个 *subdev/entity*。应用 `media-ctl` 可以枚举这些实体。([docs.kernel.org](https://docs.kernel.org/driver-api/media/mc-core.html?utm_source=chatgpt.com))
- **Pad（源 Pad / Sink Pad）**：实体的输入/输出端点（图中蓝色表示 Source Pad）。数据总是从某个实体的 source pad 流向另一个实体的 sink pad。([docs.kernel.org](https://docs.kernel.org/driver-api/media/mc-core.html?utm_source=chatgpt.com))
- **Link（链接）**：连接两个 pad 的线。Link 可以是 *active*（启用的数据通路）或 *disabled*（暂时不走数据）。图中实线表示“真实的/活动的链路”，虚线表示“逻辑/虚拟的链路”（即软件上表示的连接，不一定每时每刻都有真实数据流）。
- **为什么要这样分层？** 因为摄像头系统有物理层（PHY）、协议层（CSI host）、采集/路由模块（CIF/VICAP）、图像处理单元（ISP）与编码/外设，每层职责不同、能并行加速，Linux 用 media graph 把这些硬件模块抽象出来，方便驱动与用户态（libcamera/gstreamer）配置。([docs.kernel.org](https://docs.kernel.org/driver-api/media/mc-core.html?utm_source=chatgpt.com))

------

## 从左到右逐个模块详解（对应图中每一列的方块）

### 左端：`mipi camera0` … `mipi camera5` / `dvp camera`

- **这就是传感器（摄像头模组）**。它通过 MIPI-CSI2（串行差分）或 DVP 并行接口输出像素流（常见 RAW10/12/14 等格式或 YUV）。
- 传感器通常通过 I²C/SPI 被 CPU 配置（帧率、分辨率、曝光等），数据是高带宽的串行或并行流。

------

### `csi2_dphy0`、`csi2_dphy1` …（D-PHY 实体）

- **物理层（PHY）**：把摄像头板上的差分电平信号（MIPI D-PHY / C-PHY）变成 SoC 内部可处理的比特流（时钟恢复、接收、解串）。
- D-PHY 有 *clock lane*（时钟线）和若干 *data lanes*（数据线）；更多 lane 或更高速率 = 更大带宽。底层规范是 MIPI CSI-2 / D-PHY。([mipi.org](https://www.mipi.org/specifications/csi-2?utm_source=chatgpt.com))

------

### `mipi0_csi2`、`mipi1_csi2` …（CSI-2 Host）

- **协议层（CSI-2 Host Controller）**：将 D-PHY 恢复的比特流按 CSI-2 协议拆包（short/long packets），处理数据类型（RAW10/12/YUV）、虚拟通道 (virtual channel)、CRC/错误检测，并输出“像素流/帧”。
- 这个模块负责把串行包重组成行/帧，并把帧交给上层（CIF/VICAP）或 DMA。它是 MIPI CSI-2 Host 的典型职责。([rockchip.fr](https://rockchip.fr/RK3288 TRM/rk3288-chapter-36-mipi-csi-2-host-controller.pdf?utm_source=chatgpt.com))

------

### `rkcif_mipi_lvdsX`（Rockchip CIF / Video Input Front-End）

- **Rockchip 摄像头输入控制器（CIF / VICAP 类）**：这个模块是 SoC 的“摄像头收集器”——它从 CSI host（或 LVDS、DVP）接收像素流，做时序对齐、帧边界检测、并把帧通过 DMA 写入 DDR，或者把帧直接送到 ISP。
- 名字里带 `mipi_lvds` 表示这个 CIF 可处理来自 MIPI 或 LVDS（有些平台上 LVDS 转接器也通过同一路径） 的输入。内核里有专门的 rkcif 驱动来管理这些链路与 DMA。([patchew.org](https://patchew.org/linux/20250306-v6-8-topic-rk3568-vicap-v5-0-f02152534f3c@wolfvision.net/20250306-v6-8-topic-rk3568-vicap-v5-7-f02152534f3c@wolfvision.net/?utm_source=chatgpt.com))

------

### `rkcif_mipi_lvdsX_sditf`（所谓的 “sditf” 子接口）

- 这是 CIF 模块下面一个 *接口/子设备（sub-device interface）*，名字里 `sditf` 在内核 DTS/日志中常见（device-tree 里也会有 `rkcif-mipi-lvds-sditf` 节点）。它的作用是把 CIF 的输出“挂”到下面的虚拟 ISP / 媒体节点上，便于 media-controller 构建完整图并让 rkisp 等消费这路数据。也就是说：它是硬件采集和 ISP 之间的“连接点 / 中间子设备”。（内核/Device Tree 中能看到对应节点）([Armbian Community Forums](https://forum.armbian.com/topic/24091-efforts-to-develop-firmware-for-h96-max-v56-rk3566-4g32g/?utm_source=chatgpt.com))

------

### `rkisp0_vir0`、`rkisp0_vir1` … （rkisp 虚拟 ISP / rkisp-vir）

- 这些是 **ISP 的“虚拟”子设备/通路节点**，代表 ISP 的不同输入通道或虚拟实例（有的 SoC 会把 ISP 划分成若干虚拟子通路供不同摄像头切换/并行使用）。ISP（rkisp）负责把传感器的 RAW 做去马赛克（demosaic）、去噪、白平衡、色彩校正、缩放、输出 YUV/RGB 等一整套图像处理（3A 控制、Gamma、CSC 等）。内核里 rkisp 驱动会在 media graph 上暴露这些虚拟节点。([docs.kernel.org](https://docs.kernel.org/admin-guide/media/rkisp1.html?utm_source=chatgpt.com))

## 图中“实线 / 虚线 / 源 Pad / Sink Pad”到底代表啥（回到图例）

- **蓝色小块 = Source Pad（源 pad）**，白色小块 = Sink Pad（接收 pad）。数据永远是 “source → sink”。你可以把 pad 想象成实体的接口针脚（但在软件上）。([docs.kernel.org](https://docs.kernel.org/driver-api/media/mc-core.html?utm_source=chatgpt.com))
- **实线（Active Link）**：表示某条链路已被建立/激活，数据可以流过。Linux 的 media 链路可以被启用或禁用（比如多个摄像头同时挂到同一个 ISP 的同一个 sink，只有一个 link 能被 active）。([docs.kernel.org](https://docs.kernel.org/driver-api/media/mc-core.html?utm_source=chatgpt.com))
- **虚线（Virtual Link / logical link）**：图里用虚线表示“逻辑上存在的连接（例如通过中间 sditf 或虚拟节点连接）”，或是在 device-tree 中的 endpoint/port 关系，在内核上是 media link 的表达方式（有时候 driver 会在绑定时创建这些 virtual links）。（开发板 dmesg 日志中常看到和 sditf / vir 节点有关的 endpoint/port 依赖消息）。([DietPi](https://dietpi.com/forum/uploads/short-url/7yvRt5wLTAZzsY7rsIBpOk4awl3.log?utm_source=chatgpt.com))

## 把整条链路连起来的“真实流程”举例（一帧数据怎么走）

1. 摄像头 sensor 输出 MIPI-CSI2 数据。
2. `csi2_dphyX` 恢复串行比特并交给 `mipiX_csi2`。([mipi.org](https://www.mipi.org/specifications/csi-2?utm_source=chatgpt.com))
3. `mipiX_csi2` 按 CSI-2 协议拆包，按帧/行把像素流送到 `rkcif_mipi_lvdsX`。([rockchip.fr](https://rockchip.fr/RK3288 TRM/rk3288-chapter-36-mipi-csi-2-host-controller.pdf?utm_source=chatgpt.com))
4. `rkcif_mipi_lvdsX` 做帧边界/格式确认并通过 DMA 把帧存入 DDR（或直接通过 sditf/virtual 链路把帧交给 rkisp）。([patchew.org](https://patchew.org/linux/20250306-v6-8-topic-rk3568-vicap-v5-0-f02152534f3c@wolfvision.net/20250306-v6-8-topic-rk3568-vicap-v5-7-f02152534f3c@wolfvision.net/?utm_source=chatgpt.com))
5. `rkisp`（ISP）从内存读 RAW，做一系列图像处理，处理后的帧再写回内存或送到编码/显示管线。([docs.kernel.org](https://docs.kernel.org/admin-guide/media/rkisp1.html?utm_source=chatgpt.com))

## 🧠 一、核心概念背景

要理解这段话，先明确两个关键点：

| 名词             | 含义                                                         |
| ---------------- | ------------------------------------------------------------ |
| **MIPI**         | 一种高速串行接口标准，用于连接摄像头（CSI）或显示屏（DSI）。 |
| **DPHY / CPHY**  | 是 MIPI 物理层（PHY）标准的两种模式： ➡️ DPHY：传统的差分双线传输方式。 ➡️ CPHY：更新的三线编码方式，效率更高。 |
| **lane（通道）** | 类似“数据线”，MIPI 带宽由 lane 数决定（例如 2lane、4lane）。 |

------

## 📘 二、原文逐句解释

### **1️⃣ “Two MIPI DC(DPHY/CPHY) combo PHY”**

> 有 **两组支持 DPHY 或 CPHY 模式的 MIPI DC PHY（组合PHY）**

意思是：

- 芯片上有两套 **可切换工作模式的 MIPI 接口控制器**；
- 它们可以根据需要选择 **DPHY** 或 **CPHY** 模式；
- 通常用于**高性能主摄**或**外部多摄像头模块**。

📌 **“DC”** 表示 Dual Combo（既能 DPHY 又能 CPHY）。

------

### **2️⃣ “Each MIPI DPHY V2.0, 4lanes, 4.5Gbps per lane”**

> 每个 DPHY 支持 4 通道，速率每 lane 可达 4.5 Gbps。

📍这说明：

- 如果你用 DPHY 模式；
- 一组 PHY 有 4 根数据线；
- 每根线最高速率 4.5Gbps；
- 总带宽 = 4 × 4.5Gbps = **18Gbps（单PHY）**。

------

### **3️⃣ “Each MIPI CPHY V1.1, 3lanes, 2.5Gsps per lane”**

> 如果切换成 CPHY 模式，则每个 PHY 有 3lane，每lane最高 2.5Gsps。

CPHY 和 DPHY 的带宽效率不同，
虽然每 lane 数少（3 vs 4），但效率高。
换算后它们的总带宽接近甚至略高于 DPHY 模式。

------

### **4️⃣ “Four MIPI CSI DPHY”**

> 除了上面的2个“DC PHY”，芯片还额外有 **4组专用的 MIPI CSI DPHY 接口**。

这些 CSI DPHY 是专为摄像头（Camera Serial Interface）设计的：

- 每组独立；
- 只能用 DPHY 模式；
- 常用于中低分辨率的摄像头。

------

### **5️⃣ “Each MIPI DPHY V1.2, 2lanes, 2.5Gbps per lane”**

> 每个 DPHY 是 V1.2 版本，2 条数据线，每条线 2.5Gbps。

📍每个 DPHY 总带宽：

```
2 × 2.5Gbps = 5Gbps
```

对于 1080P 摄像头完全够用。

------

### **6️⃣ “Support to combine 2 DPHY together to one 4lanes”**

> 支持将两个 DPHY 组合成一个 4lane 接口。

这点非常关键！

- 一般的高分辨率摄像头（如 8MP、4K）需要 **4lane**；
- 若你的每个 DPHY 只有 2lane；
- 就可以把两组“绑在一起”组成一个 4lane；
- 软件或设备树中通过 “combine mode” 或 “dual-dphy” 方式配置。

------

## 🧩 三、总结前面的小节

| 类型          | 数量 | 模式           | 每PHY带宽                       | 典型用途               |
| ------------- | ---- | -------------- | ------------------------------- | ---------------------- |
| MIPI DC PHY   | 2    | DPHY/CPHY 任选 | DPHY:4x4.5Gbps / CPHY:3x2.5Gsps | 主摄、高分辨率摄像头   |
| MIPI CSI DPHY | 4    | DPHY-only      | 2x2.5Gbps                       | 次摄、辅助摄、ISP 输入 |

------

## 📦 四、组合支持情况（关键表）

原文列出三种典型摄像头输入组合方式：

| 组合类型                                                     | 说明                                         | 可支持摄像头总数 |
| ------------------------------------------------------------ | -------------------------------------------- | ---------------- |
| **① 2 MIPI DCPHY + 4 MIPI CSI DPHY(2 lanes)**                | 所有接口都用最基本配置（每个2lane）          | **6个摄像头**    |
| **② 2 MIPI DCPHY + 1 MIPI CSI DPHY(4 lanes) + 2 MIPI CSI DPHY(2 lanes)** | 一个高带宽通道(4lane) + 两个中带宽 + 2主通道 | **5个摄像头**    |
| **③ 2 MIPI DCPHY + 2 MIPI CSI DPHY(4 lanes)**                | 所有都是高带宽配置（每个4lane）              | **4个摄像头**    |

## 🧠 一、总体逻辑：数据流从哪里到哪里？

你这段设备树描述的是：

```
摄像头 (IMX415)
   ↓
MIPI DPHY (csi2_dcphy0)
   ↓
MIPI-CSI Host (mipi0_csi2)
   ↓
VICAP (rkcif_mipi_lvds)
   ↓
VICAP 输出接口 (rkcif_mipi_lvds_sditf)
   ↓
ISP 虚拟通道 (rkisp0_vir0)
   ↓
ISP0 (rkisp0)
```

📌 **总结一句话：**

> 这就是从摄像头传感器（sensor）一路经过硬件接口（PHY → CSI → VICAP）到 ISP（图像信号处理器）的数据路径。

------

## 🧩 二、每个模块的功能解释

| 节点                     | 作用                          | 关键属性/说明                                                |
| ------------------------ | ----------------------------- | ------------------------------------------------------------ |
| `imx415@1a`              | 摄像头 sensor 节点            | 定义 I2C 地址、电源、时钟、GPIO、MIPI 输出端口等。           |
| `&csi2_dcphy0`           | MIPI 物理层 (PHY)             | 负责从 sensor 接收 MIPI 差分信号（电信号），解串行化成数字流。 |
| `&mipi0_csi2`            | MIPI-CSI Host 控制器          | 接收来自 PHY 的视频流，并包装成标准的 CSI2 数据包格式。      |
| `&rkcif_mipi_lvds`       | VICAP（Video Capture 接口）   | 将 MIPI-CSI 接口接收到的原始图像数据送入系统内存或传递给 ISP。 |
| `&rkcif_mipi_lvds_sditf` | VICAP 的 “输出接口”           | VICAP 采集后，把视频数据送到 ISP 虚拟输入接口。              |
| `&rkisp0_vir0`           | ISP 的虚拟输入节点            | 接收来自 VICAP 的数据流（一般对应 `/dev/video11`）。         |
| `&rkisp0`                | ISP（Image Signal Processor） | 完成去噪、自动曝光、白平衡、亮度等图像处理功能。             |

## 🔗 三、链路连接逻辑详解（每一跳）

我们把每个 `remote-endpoint` 的对应关系连起来看：

### **1️⃣ sensor → csi2_dcphy0**

```
imx415_out0 → mipi_in_ucam0
```

📍说明：
摄像头的 MIPI 数据输出端（4lane）连接到 DC PHY 的输入端。
这一步是 **物理层的电信号连接**。

------

### **2️⃣ csi2_dcphy0 → mipi0_csi2**

```
csidcphy0_out → mipi0_csi2_input
```

📍说明：
DC PHY 的数据输出（已经被解串）输入到 MIPI-CSI 控制器。
这时数据变成了符合 **CSI-2 协议格式的像素流**。

------

### **3️⃣ mipi0_csi2 → rkcif_mipi_lvds**

```
mipi0_csi2_output → cif_mipi_in0
```

📍说明：
CSI 控制器把数据交给 **VICAP（RKCIF）模块**，
RKCIF 负责图像采集和缓冲（Capture Interface）。

------

### **4️⃣ rkcif_mipi_lvds → rkcif_mipi_lvds_sditf**

这是 VICAP 的子节点，它负责把采集到的数据送给 ISP。
`sditf` 表示 **Streaming Data Interface**，即数据流接口。

------

### **5️⃣ rkcif_mipi_lvds_sditf → rkisp0_vir0**

```
mipi_lvds_sditf → isp0_vir0
```

📍说明：
这表示 VICAP 的输出端通过虚拟链路（virtual link）连接到 ISP 的虚拟输入口。
这时视频帧进入 ISP 处理流程。

~~~C
static int dw_hdmi_rockchip_probe(struct platform_device *pdev)
{
    struct rockchip_hdmi *hdmi;
    const struct of_device_id *match;
    struct dw_hdmi_plat_data *plat_data;
    int id;

    /* 1. 为 HDMI 驱动私有数据结构分配内存 */
    hdmi = devm_kzalloc(&pdev->dev, sizeof(*hdmi), GFP_KERNEL);
    if (!hdmi)
        return -ENOMEM;

    /* 2. 从设备树 alias 中获取 HDMI 控制器编号（如 hdmi0、hdmi1） */
    id = of_alias_get_id(pdev->dev.of_node, "hdmi");
    if (id < 0)
        id = 0;  /* 若未指定 alias，默认使用 ID 0 */

    hdmi->id = id;
    hdmi->dev = &pdev->dev;

    /* 3. 根据设备树 compatible 字符串匹配平台数据 */
    match = of_match_node(dw_hdmi_rockchip_dt_ids, pdev->dev.of_node);
    if (!match)
        return -EINVAL;  /* 可选：增加健壮性检查 */

    /* 4. 复制匹配到的平台数据（plat_data）到设备私有内存 */
    plat_data = devm_kmemdup(&pdev->dev, match->data,
                             sizeof(*plat_data), GFP_KERNEL);
    if (!plat_data)
        return -ENOMEM;

    /* 5. 补充运行时所需的动态字段 */
    plat_data->id = hdmi->id;
    hdmi->plat_data = plat_data;
    hdmi->chip_data = plat_data->phy_data;

    /* 6. 设置驱动私有数据 */
    platform_set_drvdata(pdev, hdmi);

    /* 7. 启用电源管理运行时控制 */
    pm_runtime_enable(&pdev->dev);
    pm_runtime_get_sync(&pdev->dev);

    /* 8. 注册 component，用于多设备协同绑定（如与 VOP 组合） */
    return component_add(&pdev->dev, &dw_hdmi_rockchip_ops);
}
~~~

~~~c
/**
 * dw_hdmi_rockchip_bind - HDMI 驱动绑定函数（在 component 框架中被调用）
 * @dev: 当前 HDMI 平台设备
 * @master: DRM 主设备（通常是 Rockchip DRM 设备）
 * @data: 指向 drm_device 的指针
 *
 * 此函数在 probe 阶段通过 component_add 延迟调用，用于：
 * - 初始化时钟、PHY、寄存器
 * - 设置平台回调函数
 * - 绑定 Synopsys DW HDMI 核心驱动（或 QP 版本）
 * - 注册 DRM encoder/connector 子设备
 */
static int dw_hdmi_rockchip_bind(struct device *dev, struct device *master,
				 void *data)
{
	struct platform_device *pdev = to_platform_device(dev);
	struct drm_device *drm = data;
	struct drm_encoder *encoder;
	struct rockchip_hdmi *hdmi;
	struct dw_hdmi_plat_data *plat_data;
	struct rockchip_hdmi *secondary;  // 用于双 HDMI 分屏模式的另一个 HDMI 实例
	int ret;
	u32 val;

	// 检查设备树节点是否存在
	if (!pdev->dev.of_node)
		return -ENODEV;

	// 获取 probe 阶段保存的 hdmi 结构体
	hdmi = platform_get_drvdata(pdev);
	if (!hdmi)
		return -ENOMEM;

	plat_data = hdmi->plat_data;
	hdmi->drm_dev = drm;

	// 将 hdmi 实例传给 PHY 驱动使用
	plat_data->phy_data = hdmi;

	// 注册一系列平台相关的回调函数（用于颜色格式、HDR、EDID、总线配置等）
	plat_data->get_input_bus_format = dw_hdmi_rockchip_get_input_bus_format;
	plat_data->get_output_bus_format = dw_hdmi_rockchip_get_output_bus_format;
	plat_data->get_enc_in_encoding = dw_hdmi_rockchip_get_enc_in_encoding;
	plat_data->get_enc_out_encoding = dw_hdmi_rockchip_get_enc_out_encoding;
	plat_data->get_quant_range = dw_hdmi_rockchip_get_quant_range;
	plat_data->get_hdr_property = dw_hdmi_rockchip_get_hdr_property;
	plat_data->get_hdr_blob = dw_hdmi_rockchip_get_hdr_blob;
	plat_data->get_color_changed = dw_hdmi_rockchip_get_color_changed;
	plat_data->get_yuv422_format = dw_hdmi_rockchip_get_yuv422_format;
	plat_data->get_edid_dsc_info = dw_hdmi_rockchip_get_edid_dsc_info;
	plat_data->get_next_hdr_data = dw_hdmi_rockchip_get_next_hdr_data;
	plat_data->get_colorimetry = dw_hdmi_rockchip_get_colorimetry;
	plat_data->get_link_cfg = dw_hdmi_rockchip_get_link_cfg;
	plat_data->set_hdcp2_enable = rk3588_set_hdcp2_enable;
	plat_data->set_hdcp_status = rk3588_set_hdcp_status;
	plat_data->set_grf_cfg = rk3588_set_grf_cfg;
	plat_data->get_grf_color_fmt = rk3588_get_grf_color_fmt;
	plat_data->convert_to_split_mode = drm_mode_convert_to_split_mode;
	plat_data->convert_to_origin_mode = drm_mode_convert_to_origin_mode;
	plat_data->dclk_set = dw_hdmi_dclk_set;
	plat_data->link_clk_set = dw_hdmi_link_clk_set;
	plat_data->get_vp_id = dw_hdmi_rockchip_get_vp_id;
	plat_data->update_color_format = dw_hdmi_rockchip_update_color_format;
	plat_data->check_hdr_color_change = dw_hdmi_rockchip_check_hdr_color_change;
	plat_data->set_prev_bus_format = dw_hdmi_rockchip_set_prev_bus_format;
	plat_data->set_ddc_io = dw_hdmi_rockchip_set_ddc_io;
	plat_data->set_hdcp14_mem = dw_hdmi_rockchip_set_hdcp14_mem;
	plat_data->property_ops = &dw_hdmi_rockchip_property_ops;

	// 查找另一个 HDMI 实例（用于双屏分屏模式）
	secondary = rockchip_hdmi_find_by_id(dev->driver, !hdmi->id);

	/* 如果启用了双 HDMI（hdmi0 + hdmi1）且支持 split_mode，则配置分屏 */
	if (hdmi->chip_data->split_mode && secondary) {
		/*
		 * 在分屏模式下，只有最后一个绑定的 HDMI 才能初始化 bridge 和 encoder/connector，
		 * 否则 hdmi_qp 未初始化，left/right 指针为空。
		 * 因此需判断设备树中是否启用 split-mode，并确定绑定顺序。
		 */
		if (device_property_read_bool(dev, "split-mode") ||
		    device_property_read_bool(secondary->dev, "split-mode")) {
			plat_data->split_mode = true;
			secondary->plat_data->split_mode = true;
			// 标记第一个屏幕（用于后续初始化顺序控制）
			if (!secondary->plat_data->first_screen)
				plat_data->first_screen = true;
		}
	}

	// 如果当前不是第一个屏幕（或非分屏模式），初始化 encoder
	if (!plat_data->first_screen) {
		encoder = &hdmi->encoder;
		// 根据设备树查找关联的 CRTC（显示控制器）
		encoder->possible_crtcs = rockchip_drm_of_find_possible_crtcs(drm, dev->of_node);
		if (encoder->possible_crtcs == 0)
			return -EPROBE_DEFER; // CRTC 尚未注册，稍后重试

		// 注册 encoder 辅助函数和初始化
		drm_encoder_helper_add(encoder, &dw_hdmi_rockchip_encoder_helper_funcs);
		drm_simple_encoder_init(drm, encoder, DRM_MODE_ENCODER_TMDS);
	}

	// 设置最大 TMDS 时钟频率（默认 594MHz）
	if (!plat_data->max_tmdsclk)
		hdmi->max_tmdsclk = 594000;
	else
		hdmi->max_tmdsclk = plat_data->max_tmdsclk;

	// 保存 HDMI 特性标志
	hdmi->is_hdmi_qp = plat_data->is_hdmi_qp;
	hdmi->unsupported_yuv_input = plat_data->unsupported_yuv_input;
	hdmi->unsupported_deep_color = plat_data->unsupported_deep_color;

	// 解析设备树中的 HDMI 相关配置
	ret = rockchip_hdmi_parse_dt(hdmi);
	if (ret) {
		DRM_DEV_ERROR(hdmi->dev, "Unable to parse OF data\n");
		return ret;
	}

	// 使能所有 HDMI 相关时钟
	ret = clk_prepare_enable(hdmi->aud_clk);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI aud_clk: %d\n", ret);
		return ret;
	}

	ret = clk_prepare_enable(hdmi->hpd_clk);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI hpd_clk: %d\n", ret);
		return ret;
	}

	ret = clk_prepare_enable(hdmi->hclk_vo1);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI hclk_vo1: %d\n", ret);
		return ret;
	}

	ret = clk_prepare_enable(hdmi->earc_clk);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI earc_clk: %d\n", ret);
		return ret;
	}

	ret = clk_prepare_enable(hdmi->hdmitx_ref);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI hdmitx_ref: %d\n", ret);
		return ret;
	}

	ret = clk_prepare_enable(hdmi->pclk);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI pclk: %d\n", ret);
		return ret;
	}

	// 配置 RK3568 的 DDC（I2C）引脚复用
	if (hdmi->chip_data->ddc_en_reg == RK3568_GRF_VO_CON1) {
		regmap_write(hdmi->regmap, RK3568_GRF_VO_CON1,
			     HIWORD_UPDATE(RK3568_HDMI_SDAIN_MSK |
					   RK3568_HDMI_SCLIN_MSK,
					   RK3568_HDMI_SDAIN_MSK |
					   RK3568_HDMI_SCLIN_MSK));
	}

	// 如果是 RK3588 的 HDMI QP（高质量 PHY）版本，进行特殊寄存器配置
	if (hdmi->is_hdmi_qp) {
		if (!hdmi->id) {
			// 配置 HDMI0 的 GRF 寄存器（引脚复用、HPD 路径、仲裁等）
			val = HIWORD_UPDATE(RK3588_SCLIN_MASK, RK3588_SCLIN_MASK) |
			      HIWORD_UPDATE(RK3588_SDAIN_MASK, RK3588_SDAIN_MASK) |
			      HIWORD_UPDATE(RK3588_MODE_MASK, RK3588_MODE_MASK) |
			      HIWORD_UPDATE(RK3588_I2S_SEL_MASK, RK3588_I2S_SEL_MASK);
			regmap_write(hdmi->vo1_regmap, RK3588_GRF_VO1_CON3, val);

			val = HIWORD_UPDATE(RK3588_SET_HPD_PATH_MASK,
					    RK3588_SET_HPD_PATH_MASK);
			regmap_write(hdmi->regmap, RK3588_GRF_SOC_CON7, val);

			val = HIWORD_UPDATE(RK3588_HDMI0_GRANT_SEL,
					    RK3588_HDMI0_GRANT_SEL);
			regmap_write(hdmi->vo1_regmap, RK3588_GRF_VO1_CON9, val);
		} else {
			// 配置 HDMI1
			val = HIWORD_UPDATE(RK3588_SCLIN_MASK, RK3588_SCLIN_MASK) |
			      HIWORD_UPDATE(RK3588_SDAIN_MASK, RK3588_SDAIN_MASK) |
			      HIWORD_UPDATE(RK3588_MODE_MASK, RK3588_MODE_MASK) |
			      HIWORD_UPDATE(RK3588_I2S_SEL_MASK, RK3588_I2S_SEL_MASK);
			regmap_write(hdmi->vo1_regmap, RK3588_GRF_VO1_CON6, val);

			val = HIWORD_UPDATE(RK3588_SET_HPD_PATH_MASK,
					    RK3588_SET_HPD_PATH_MASK);
			regmap_write(hdmi->regmap, RK3588_GRF_SOC_CON7, val);

			val = HIWORD_UPDATE(RK3588_HDMI1_GRANT_SEL,
					    RK3588_HDMI1_GRANT_SEL);
			regmap_write(hdmi->vo1_regmap, RK3588_GRF_VO1_CON9, val);
		}
		// 初始化 HPD（热插拔检测）工作队列
		init_hpd_work(hdmi);
	}

	// 使能 PHY 参考时钟
	ret = clk_prepare_enable(hdmi->phyref_clk);
	if (ret) {
		DRM_DEV_ERROR(hdmi->dev, "Failed to enable HDMI vpll: %d\n", ret);
		return ret;
	}

	// 使能其他视频相关时钟
	ret = clk_prepare_enable(hdmi->hclk_vio);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI hclk_vio: %d\n", ret);
		return ret;
	}

	ret = clk_prepare_enable(hdmi->hclk_vop);
	if (ret) {
		dev_err(hdmi->dev, "Failed to enable HDMI hclk_vop: %d\n", ret);
		return ret;
	}

	// 为 HDMI QP 配置 HPD 中断
	if (hdmi->is_hdmi_qp) {
		if (!hdmi->id)
			val = HIWORD_UPDATE(RK3588_HDMI0_HPD_INT_MSK, RK3588_HDMI0_HPD_INT_MSK);
		else
			val = HIWORD_UPDATE(RK3588_HDMI1_HPD_INT_MSK, RK3588_HDMI1_HPD_INT_MSK);
		regmap_write(hdmi->regmap, RK3588_GRF_SOC_CON2, val);

		// 获取 HPD 中断号（设备树中第 4 个中断）
		hdmi->hpd_irq = platform_get_irq(pdev, 4);
		if (hdmi->hpd_irq < 0)
			return hdmi->hpd_irq;

		// 注册线程化中断处理函数
		ret = devm_request_threaded_irq(hdmi->dev, hdmi->hpd_irq,
						rockchip_hdmi_hardirq,    // 上半部
						rockchip_hdmi_irq,        // 下半部
						IRQF_SHARED, "dw-hdmi-qp-hpd", hdmi);
		if (ret)
			return ret;
	}

	// 获取 HDMI PHY（可能有两个名字）
	hdmi->phy = devm_phy_optional_get(dev, "hdmi");
	if (IS_ERR(hdmi->phy)) {
		hdmi->phy = devm_phy_optional_get(dev, "hdmi_phy");
		if (IS_ERR(hdmi->phy)) {
			ret = PTR_ERR(hdmi->phy);
			if (ret != -EPROBE_DEFER)
				DRM_DEV_ERROR(hdmi->dev, "failed to get phy\n");
			return ret;
		}
	}

	// 如果是 HDMI QP 版本，绑定 QP 专用驱动
	if (hdmi->is_hdmi_qp) {
		hdmi->hdmi_qp = dw_hdmi_qp_bind(pdev, &hdmi->encoder, plat_data);

		if (IS_ERR(hdmi->hdmi_qp)) {
			ret = PTR_ERR(hdmi->hdmi_qp);
			drm_encoder_cleanup(&hdmi->encoder);
			return ret;
		}

		// 注册子设备（用于 Rockchip DRM 框架管理 connector）
		if (plat_data->bridge) {
			struct drm_connector *connector = NULL;
			struct list_head *connector_list =
				&plat_data->bridge->dev->mode_config.connector_list;

			// 从 bridge 中找到关联的 connector
			list_for_each_entry(connector, connector_list, head)
				if (drm_connector_has_possible_encoder(connector,
							&hdmi->encoder))
					break;

			hdmi->sub_dev.connector = connector;
			hdmi->sub_dev.of_node = dev->of_node;
			rockchip_drm_register_sub_dev(&hdmi->sub_dev);
		} else if (plat_data->connector) {
			hdmi->sub_dev.connector = plat_data->connector;
			hdmi->sub_dev.loader_protect = dw_hdmi_rockchip_encoder_loader_protect;
			if (secondary && device_property_read_bool(secondary->dev, "split-mode"))
				hdmi->sub_dev.of_node = secondary->dev->of_node;
			else
				hdmi->sub_dev.of_node = hdmi->dev->of_node;

			rockchip_drm_register_sub_dev(&hdmi->sub_dev);
		}

		// 在分屏模式下，互相设置 left/right 指针
		if (plat_data->split_mode && secondary) {
			if (device_property_read_bool(dev, "split-mode")) {
				plat_data->right = secondary->hdmi_qp;
				secondary->plat_data->left = hdmi->hdmi_qp;
			} else {
				plat_data->left = secondary->hdmi_qp;
				secondary->plat_data->right = hdmi->hdmi_qp;
			}
		}

		return 0;
	}

	// 非 QP 版本：绑定标准 DW HDMI 驱动
	hdmi->hdmi = dw_hdmi_bind(pdev, &hdmi->encoder, plat_data);

	/*
	 * 如果绑定失败，手动清理 encoder 并关闭已使能的时钟
	 */
	if (IS_ERR(hdmi->hdmi)) {
		ret = PTR_ERR(hdmi->hdmi);
		drm_encoder_cleanup(&hdmi->encoder);
		clk_disable_unprepare(hdmi->aud_clk);
		clk_disable_unprepare(hdmi->phyref_clk);
		clk_disable_unprepare(hdmi->hclk_vop);
		clk_disable_unprepare(hdmi->hpd_clk);
		clk_disable_unprepare(hdmi->hclk_vo1);
		clk_disable_unprepare(hdmi->earc_clk);
		clk_disable_unprepare(hdmi->hdmitx_ref);
		clk_disable_unprepare(hdmi->pclk);
		return ret;
	}

	// 注册标准 HDMI 子设备
	if (plat_data->connector) {
		hdmi->sub_dev.connector = plat_data->connector;
		hdmi->sub_dev.of_node = dev->of_node;
		rockchip_drm_register_sub_dev(&hdmi->sub_dev);
	}

	return 0;
}

/**
 * dw_hdmi_rockchip_unbind - 解绑 HDMI 驱动
 * 在 DRM 设备卸载或模块移除时调用，释放资源
 */
static void dw_hdmi_rockchip_unbind(struct device *dev, struct device *master,
				    void *data)
{
	struct rockchip_hdmi *hdmi = dev_get_drvdata(dev);

	// 如果是 HDMI QP，取消并销毁 HPD 工作队列
	if (hdmi->is_hdmi_qp) {
		cancel_delayed_work(&hdmi->work);
		flush_workqueue(hdmi->workqueue);
		destroy_workqueue(hdmi->workqueue);
	}

	// 注销子设备
	if (hdmi->sub_dev.connector)
		rockchip_drm_unregister_sub_dev(&hdmi->sub_dev);

	// 调用核心驱动的解绑函数
	if (hdmi->is_hdmi_qp)
		dw_hdmi_qp_unbind(hdmi->hdmi_qp);
	else
		dw_hdmi_unbind(hdmi->hdmi);

	// 关闭所有时钟
	clk_disable_unprepare(hdmi->aud_clk);
	clk_disable_unprepare(hdmi->phyref_clk);
	clk_disable_unprepare(hdmi->hclk_vop);
	clk_disable_unprepare(hdmi->hpd_clk);
	clk_disable_unprepare(hdmi->hclk_vo1);
	clk_disable_unprepare(hdmi->earc_clk);
	clk_disable_unprepare(hdmi->hdmitx_ref);
	clk_disable_unprepare(hdmi->pclk);
}
~~~

该函数的重点在于第309行的dw_hdmi_bind函数，`dw_hdmi_bind()`（位于 `drivers/gpu/drm/bridge/synopsys/dw-hdmi.c`）这是 **Synopsys DesignWare HDMI IP 的通用驱动**，它用于初始化HDMI 寄存器、创建drm_bridge、创建 drm_connector，具体内容如下所示：

~~~c
struct dw_hdmi *dw_hdmi_bind(struct platform_device *pdev,
			     struct drm_encoder *encoder,
			     struct dw_hdmi_plat_data *plat_data)
{
	struct dw_hdmi *hdmi;
	int ret;

	hdmi = dw_hdmi_probe(pdev, plat_data);
	if (IS_ERR(hdmi))
		return hdmi;

	ret = drm_bridge_attach(encoder, &hdmi->bridge, NULL, 0);
	if (ret) {
		dw_hdmi_remove(hdmi);
		DRM_ERROR("Failed to initialize bridge with drm\n");
		return ERR_PTR(ret);
	}

	if (!hdmi->next_bridge)
		plat_data->connector = &hdmi->connector;

	return hdmi;
}
~~~

在 `dw_hdmi_bind()` 中，当你调用：

```C
ret = drm_bridge_attach(encoder, &hdmi->bridge, NULL, 0);
```

这会触发 **DW HDMI 驱动内部初始化其 connector**，其中关键一步是设置 connector 的操作函数：

```C
// drivers/gpu/drm/bridge/synopsys/dw-hdmi.c
static const struct drm_connector_funcs dw_hdmi_connector_funcs = {
	.fill_modes = dw_hdmi_connector_get_modes,
	...
};
```

也就是说，**每当 DRM 核心需要获取该 HDMI 接口支持的显示模式时，就会调用 `dw_hdmi_connector_get_modes()`**。

该函数的具体内容如下所示：

~~~c
/**
 * dw_hdmi_connector_get_modes - 获取 HDMI 连接器支持的显示模式
 * @connector: DRM 连接器结构体，代表 HDMI 输出端口
 *
 * 该函数在 DRM 核心请求探测连接器支持的显示模式时被调用（例如用户空间执行 modetest -c）。
 * 主要流程：
 *  1. 尝试通过 DDC（I²C）读取显示器的 EDID 数据；
 *  2. 若成功，解析 EDID 并添加所有支持的视频模式；
 *  3. 若失败，回退到预定义的安全模式（如 640x480）；
 *  4. 同时处理 HDR、色彩格式、CEC 物理地址等 HDMI 特性。
 *
 * 返回值：成功添加的模式数量
 */
static int dw_hdmi_connector_get_modes(struct drm_connector *connector)
{
	// 通过 container_of 从 connector 反向获取 dw_hdmi 主结构体
	struct dw_hdmi *hdmi = container_of(connector, struct dw_hdmi, connector);

	// 指向 HDR 静态元数据结构（用于 HDMI 2.0a+ 的 HDR10 支持）
	struct hdr_static_metadata *metedata =
			&connector->hdr_sink_metadata.hdmi_type1;

	// EDID 数据指针
	struct edid *edid;

	// 用于遍历已探测模式的临时变量
	struct drm_display_mode *mode;

	// 指向连接器的显示能力信息（如色彩格式、YUV420 支持等）
	struct drm_display_info *info = &connector->display_info;

	// 平台私有数据（通常用于芯片特定操作，如 Rockchip 的 PHY 控制）
	void *data = hdmi->plat_data->phy_data;

	int i, ret = 0;

	// 清零 HDR 元数据，确保未设置时为默认值
	memset(metedata, 0, sizeof(*metedata));

	// 尝试从 HDMI DDC 通道读取 EDID
	edid = dw_hdmi_get_edid(hdmi, connector);

	if (edid) {
		int vic = 0;

		// 调试信息：打印显示器物理尺寸（单位：厘米）
		dev_dbg(hdmi->dev, "got edid: width[%d] x height[%d]\n",
			edid->width_cm, edid->height_cm);

		// 将 EDID 数据更新到 DRM connector 的 sysfs 属性中（如 /sys/class/drm/card0-HDMI-A-1/edid）
		drm_connector_update_edid_property(connector, edid);

		// 从 EDID 中提取 CEC 物理地址，并通知 CEC 子系统
		cec_notifier_set_phys_addr_from_edid(hdmi->cec_notifier, edid);

		// 解析 EDID 中的标准/CEA/DMT 模式，并添加到 connector->probed_modes 列表
		ret = drm_add_edid_modes(connector, edid);

		// 若平台支持动态色彩格式检测，则调用回调函数处理 YUV422 格式
		if (hdmi->plat_data->get_color_changed)
			hdmi->plat_data->get_yuv422_format(connector, edid);

		// 若平台支持色彩空间（colorimetry）解析，则调用回调处理
		if (hdmi->plat_data->get_colorimetry)
			hdmi->plat_data->get_colorimetry(data, edid);

		// 遍历所有已探测到的显示模式，修正部分模式的图像宽高比（Picture Aspect Ratio）
		list_for_each_entry(mode, &connector->probed_modes, head) {
			// 获取该模式对应的 CEA VIC（Video Identification Code）
			vic = drm_match_cea_mode(mode);

			// 若 EDID 未指定宽高比，根据 VIC 补充标准值：
			if (mode->picture_aspect_ratio == HDMI_PICTURE_ASPECT_NONE) {
				// VIC 93~95：16:9 宽屏（如 1920x1080）
				if (vic >= 93 && vic <= 95)
					mode->picture_aspect_ratio = HDMI_PICTURE_ASPECT_16_9;
				// VIC 98：超宽屏 256:135（如 3840x2160 UHD）
				else if (vic == 98)
					mode->picture_aspect_ratio = HDMI_PICTURE_ASPECT_256_135;
			}
		}

		// 释放 EDID 内存
		kfree(edid);
	} else {
		// EDID 读取失败（如未接显示器或 DDC 通信异常）
		dev_info(hdmi->dev, "failed to get edid\n");

		// 假设连接的是 HDMI 设备（而非 DVI），并支持音频（保守假设）
		hdmi->support_hdmi = true;
		hdmi->sink_has_audio = true;

		// 回退到预定义的默认安全模式列表（如 640x480, 800x600 等）
		for (i = 0; i < ARRAY_SIZE(dw_hdmi_default_modes); i++) {
			const struct drm_display_mode *ptr = &dw_hdmi_default_modes[i];

			// 复制默认模式到当前 connector
			mode = drm_mode_duplicate(connector->dev, ptr);
			if (mode) {
				// 第一个模式标记为“首选模式”（PREFERRED）
				if (!i)
					mode->type = DRM_MODE_TYPE_PREFERRED;
				// 将模式添加到探测列表中
				drm_mode_probed_add(connector, mode);
				ret++;
			}
		}

		// 清空高级显示能力标志（因为无 EDID，无法确认支持）
		info->edid_hdmi_dc_modes = 0;      // 不支持深色深（Deep Color）
		info->hdmi.y420_dc_modes = 0;      // 不支持 YUV420 深色深
		info->color_formats = 0;           // 不支持任何特殊色彩格式（仅默认 RGB）
	}

	// 更新 HDR 相关的 DRM 属性（如是否支持 HDR10）
	dw_hdmi_update_hdr_property(connector);

	// 检查输出类型是否发生变化（如从 DVI 切换到 HDMI），用于平台特定处理
	dw_hdmi_check_output_type_changed(hdmi);

	// 返回成功探测到的模式数量
	return ret;
}
~~~
