先来看一眼SPI的前两部分内容，遇到有疑问的地方就思考一下。

（1）SPI传输数据一般都采取高位先行，移位寄存器每接收一个时钟脉冲（时钟源由主机的波特率发生器提供），移位寄存器（无论是主机还是从机）就会向左进行移位，主机中被移出的数据通过MOSI进入从机的移位寄存器最右端，从机中被移出的数据通过MOSO进入主机的移位寄存器最右端，以此循环8次，主机和从机实现1个字节的数据交换。

（2）可以规定波特率发生器产生时钟上升沿时所有移位寄存器向左移一位，被移出位置于相应引脚上，波特率发生器产生时钟下降沿时引脚上的位移入到移位寄存器的最低位，也就是下面的模式1/3；也可以规定波特率发生器产生时钟下降沿时所有移位寄存器向左移一位，被移出位置于相应引脚上，波特率发生器产生时钟上升沿时引脚上的位移入到移位寄存器的最低位，也就是下面的模式0/2。

（3）基于数据交换，主机和从机可以实现数据的单向传输，比如主机需要接收从机的数据，那么就“随便”使用一个数据和从机做数据交换即可（一般用0x00或0xFF进行数据交换）。



### SPI 数据交换机制深度解析：从位操作到全双工通信的本质

#### 一、SPI 数据交换的核心逻辑：逐位齿轮传动模型

SPI 的全双工数据交换本质上是**主从设备移位寄存器的同步逐位移位过程**，可类比为两个齿轮的啮合转动：

1. **移位寄存器的结构**：

   - 主机和从机各有一个**8 位（或更多位）串行移位寄存器**，两者通过 MOSI 和 MISO 线形成闭环链路。
   - 例如：主机移位寄存器为`HOST_REG[7:0]`，从机为`SLAVE_REG[7:0]`，初始值分别为`HOST_DATA`和`SLAVE_DATA`。

2. **单时钟周期数据流动**：

   - 当 SCLK 时钟脉冲到来时（上升沿或下降沿，取决于 CPHA 配置）：
     - 主机移位寄存器左移一位，最高位`HOST_REG[7]`通过 MOSI 线输出到从机；
     - 从机移位寄存器同时左移一位，最高位`SLAVE_REG[7]`通过 MISO 线输出到主机；
     - 主机通过 MISO 线接收从机移出的位，存入移位寄存器的最低位`HOST_REG[0]`；
     - 从机通过 MOSI 线接收主机移出的位，存入移位寄存器的最低位`SLAVE_REG[0]`。

3. **8 次移位完成一个字节交换**：

   - 每轮时钟脉冲（共 8 次）后，主机移位寄存器最终存储的是从机原始数据`SLAVE_DATA`，从机则存储主机原始数据`HOST_DATA`。

   - 示例

     ：

     - 主机发送`0xAA`（二进制`10101010`），从机发送`0x55`（`01010101`）；
     - 8 次移位后，主机接收`0x55`，从机接收`0xAA`，完成一次字节交换。

#### 二、时钟边沿与移位方向的时序控制

SPI 的 4 种工作模式（由 CPOL 和 CPHA 组合决定）本质上是定义**移位时机**和**采样时机**的差异：

##### ▌模式 0（CPOL=0，CPHA=0）：

- **时钟空闲电平**：SCLK=0（低电平）。
- **移位边沿**：SCLK 上升沿（0→1）时，主从移位寄存器左移一位，数据从最高位移出到引脚。
- **采样边沿**：SCLK 下降沿（1→0）时，主机从 MISO 读取从机输出的位，从机从 MOSI 读取主机输出的位。

##### ▌模式 1（CPOL=0，CPHA=1）：

- **时钟空闲电平**：SCLK=0。
- **移位边沿**：SCLK 下降沿（1→0）时移位。
- **采样边沿**：SCLK 上升沿（0→1）时采样。

##### ▌模式 2（CPOL=1，CPHA=0）：

- **时钟空闲电平**：SCLK=1（高电平）。
- **移位边沿**：SCLK 下降沿（1→0）时移位。
- **采样边沿**：SCLK 上升沿（0→1）时采样。

##### ▌模式 3（CPOL=1，CPHA=1）：

- **时钟空闲电平**：SCLK=1。
- **移位边沿**：SCLK 上升沿（0→1）时移位。
- **采样边沿**：SCLK 下降沿（1→0）时采样。

#### 三、为什么不会停留在 “两个数据交换”？—— 多字节传输的实现

用户的疑问核心在于：“若每次交换都是两个数据互传，那如何传输更多数据？” 答案在于**SPI 的流式传输特性**：

1. **单字节交换的延续性**：

   - 每次 8 时钟周期完成一个字节交换后，主机可立即发起下一个字节的传输：
     - 主机向移位寄存器写入新的待发送数据（如`0x00`）；
     - 从机移位寄存器中保留上一次交换后的数据（或新数据，取决于从机逻辑）；
     - 下一轮 8 时钟周期后，主机接收从机当前移位寄存器的数据，从机接收主机新写入的数据。

2. **实际应用场景举例**：

   - 主机读取从机数据：

     1. 主机发送`0x00`（“无效数据”），从机返回寄存器中的数据`DATA1`；
     2. 主机接收`DATA1`，同时发送下一个`0x00`，从机返回`DATA2`；
     3. 重复该过程，可连续读取从机多个字节（如传感器数据）。

   - 主机发送命令 + 接收响应

     ：

     1. 主机发送命令字节`CMD`，从机接收后执行操作；
     2. 主机发送`0x00`，从机返回操作结果`RESULT`；
     3. 若结果为多字节，继续发送`0x00`获取后续数据。

3. **移位寄存器的动态更新**：

   - 从机移位寄存器的数据来源并非固定，而是由其内部逻辑决定（如传感器采集的数据、寄存器存储的值）。每次交换时，从机输出的是当前移位寄存器的值，而输入的主机数据可能被用于更新从机内部状态（如命令解析）。

#### 四、波特率与时钟频率的影响

SPI 的时钟由主机波特率发生器产生，其频率决定了数据传输速度：

- **波特率计算**：波特率 = 主机系统时钟 / 分频系数（如 16、32 等，由寄存器配置）。
- 时钟频率与移位的关系：
  - 时钟频率越高，单位时间内移位次数越多，数据传输速度越快（如 1MHz 时钟可每秒传输约 125KB 数据）。
  - 但需注意从机的最大时钟频率限制（如某些外设最高支持 10MHz），否则会因时序不匹配导致数据错误。

#### 五、总结：SPI 数据交换的本质是 “逐位同步流传输”

- **单次交换**：8 个时钟周期完成 1 字节全双工交换，主从移位寄存器数据对调。
- **连续传输**：主机通过不断更新移位寄存器中的数据，可实现任意长度的数据流传输，从机则根据协议响应数据（或忽略主机数据，仅返回自身数据）。
- **核心类比**：SPI 的移位过程类似 “传送带”，每个时钟脉冲推动一位数据前进，主机和从机各自向传送带 “放入” 和 “取出” 数据，从而实现连续的数据流交互，而非局限于两个固定数据的循环。

通过这种机制，SPI 可高效支持传感器数据采集、存储器读写、外设控制等场景，只要主从设备保持时钟模式一致，即可实现稳定的多字节数据交换。

### SPI 数据交换实战解析：以 ADXL345 加速度计为例

#### 一、ADXL345 传感器 SPI 接口特性

ADXL345 是一款三轴加速度计，通过 SPI 接口与主机通信。其 SPI 特性如下37：

1. **通信模式**：
   - 固定使用**SPI 模式 3**（CPOL=1，CPHA=1），即 SCLK 空闲时为高电平，数据在上升沿采样，下降沿移位。
   - 支持 4 线 SPI（MOSI/MISO/SCLK/CS），通过 DATA_FORMAT 寄存器（地址 0x31）的 D6 位配置。
2. **寄存器访问格式**：
   - **单字节读写**：指令格式为`[R/W][ADDR7:ADDR0]`，其中最高位（D7）为 0 表示写操作，为 1 表示读操作。
   - **多字节读写**：需在指令中设置多字节标志位（D6=1），例如读取 0x32~0x37 的 6 字节加速度数据时，指令为`0x32 | 0x80 | 0x40 = 0xD2`3。
3. **数据格式**：
   - 加速度数据为 16 位补码，存储在 DATAX0/DATAX1（X 轴）、DATAY0/DATAY1（Y 轴）、DATAZ0/DATAZ1（Z 轴）寄存器中。
   - 13 位有效精度时，1 LSB 对应 3.9mg，例如原始值`0x03E8`（十进制 1000）表示 3.9g3。

#### 二、SPI 数据交换全流程拆解（以读取加速度数据为例）

假设主机为 STM32 微控制器，从机为 ADXL345，以下为具体步骤：

##### ▌1. 初始化 SPI 通信参数

- 主机配置：
  - 设置 SPI 模式为 Mode3（CPOL=1，CPHA=1）。
  - 波特率设为 2MHz（根据传感器最大支持频率调整）。
  - 片选信号（CS）由 GPIO 控制，初始为高电平。
- 传感器配置：
  - 写入 POWER_CTL 寄存器（0x2D）使能测量模式（0x08）。
  - 配置 DATA_FORMAT 寄存器（0x31）为全分辨率模式（0x08），选择 SPI 模式（D6=0）。

##### ▌2. 单字节寄存器读取：获取设备 ID

```c
// 读取DEVID寄存器（地址0x00）
uint8_t read_register(uint8_t addr) {
  uint8_t tx_buf[2] = {addr | 0x80, 0x00}; // 读指令 + 占位字节
  uint8_t rx_buf[2] = {0};
  HAL_SPI_TransmitReceive(&hspi1, tx_buf, rx_buf, 2, 100);
  return rx_buf[1]; // 实际数据在第二个字节
}

// 验证设备ID（应为0xE5）
uint8_t device_id = read_register(0x00);
```

**关键机制**：

- 主机发送读指令（`0x80 | 0x00`）后，ADXL345 在 MISO 线返回寄存器值。
- 即使主机发送的第二个字节（0x00）是占位符，ADXL345 仍会同步移位并返回数据5。

##### ▌3. 多字节数据读取：获取三轴加速度值

```c
// 读取6字节加速度数据（0x32~0x37）
void read_accel_data(int16_t *accel) {
  uint8_t tx_buf[7] = {0x32 | 0x80 | 0x40}; // 多字节读指令
  uint8_t rx_buf[7] = {0};
  HAL_SPI_TransmitReceive(&hspi1, tx_buf, rx_buf, 7, 100);
  
  // 合并高低字节并转换为补码
  accel[0] = (rx_buf[1] << 8) | rx_buf[2];
  accel[1] = (rx_buf[3] << 8) | rx_buf[4];
  accel[2] = (rx_buf[5] << 8) | rx_buf[6];
}

// 转换为g单位
int16_t raw_data[3];
read_accel_data(raw_data);
float x_g = raw_data[0] * 0.0039f; // 3.9mg/LSB
```

**时序细节**：

- 主机发送多字节读指令（`0xD2`）后，ADXL345 连续输出 6 字节数据，主机通过 7 次移位完成传输（1 指令字节 + 6 数据字节）3。
- 片选信号（CS）在整个传输过程中保持低电平，确保从机持续响应7。

#### 三、SPI 数据交换的本质：动态数据流交互

用户的核心疑问 “为何不会停留在两个数据交换” 可通过 ADXL345 的实际应用解答：

1. **数据来源的动态性**：
   - ADXL345 的移位寄存器数据并非固定，而是实时更新的加速度值。每次交换时，从机输出的是当前寄存器值，而主机发送的指令或占位符可能触发传感器内部状态变化（如启动测量）3。
2. **多字节传输的连续性**：
   - 主机通过单次 SPI 事务读取多个寄存器（如 6 字节加速度数据），避免了逐字节交换的开销。例如，读取 0x32~0x37 时，ADXL345 自动递增地址，主机只需发送一次指令即可获取完整数据帧4。
3. **指令与数据的协同**：
   - 主机发送的指令（如配置寄存器）和数据（如 0x00 占位符）共同驱动传感器行为。例如，写入 POWER_CTL 寄存器（0x2D）的 0x08 可激活测量模式，后续读取操作才能获得有效数据3。

#### 四、实际应用中的关键问题与解决方案

1. **时序匹配与模式配置**：
   - ADXL345 必须与主机使用相同的 SPI 模式（Mode3），否则会因采样边沿错位导致数据错误。例如，若主机误设为 Mode0，数据可能在错误的时钟边沿被采样8。
2. **数据溢出与补码处理**：
   - 16 位补码的取值范围为 - 32768~32767，当加速度超过 ±16g 时会发生溢出。需通过寄存器设置量程（如 ±2g、±4g）并在软件中进行饱和处理3。
3. **多从机系统的片选管理**：
   - 若多个 SPI 设备共享总线，主机需通过独立的 CS 引脚分时选通。例如，读取 ADXL345 时拉低其 CS，读取其他传感器时拉高该引脚并拉低对应 CS9。

#### 五、总结：SPI 在传感器中的数据流本质

ADXL345 的案例揭示了 SPI 数据交换的**动态数据流特性**：



- **单次交换**：8 个时钟周期完成 1 字节全双工交换，主从移位寄存器数据对调。
- **连续传输**：主机通过指令控制从机行为（如配置寄存器、启动测量），并通过多字节读取获取实时数据流。
- **核心类比**：SPI 如同 “数据管道”，主机通过指令打开阀门（CS），时钟推动数据流动，主从设备在管道中同步交换指令与响应，实现连续的数据流交互而非固定数据循环。



通过这种机制，SPI 可高效支持传感器数据采集、工业控制等场景，只要主从设备保持模式一致，即可实现稳定的多字节数据交换。

### I2C 与 SPI 通信协议对比表

| **对比维度**     | **I2C 协议**                                   | **SPI 协议**                                              |
| ---------------- | ---------------------------------------------- | --------------------------------------------------------- |
| **硬件结构**     | 2 根信号线（SDA 数据 + SCL 时钟）              | 4 根信号线（SCLK 时钟 + MOSI 主发 + MISO 从发 + CS 片选） |
| **多设备连接**   | 所有设备共享总线，通过地址寻址                 | 每个从设备独立连接 CS 线，主设备分时选通                  |
| **通信模式**     | 半双工（同一时刻单向传输）                     | 全双工（主从设备同时收发数据）                            |

高速复位与检测机制遵循低速 / 全速的行为模型。当复位完成后，链路必须按照此前使用规则所确定的、合适的信号模式（低速、全速或高速 ）运行，且端口状态寄存器中的速度指示位会准确报告该模式。软件只需触发复位操作，并在收到复位完成通知时读取端口状态寄存器即可。

具备高速能力的设备最初会作为全速设备进行连接。这意味着，对于支持高速的上行端口而言，上拉电阻（Rpu，阻值为 1.5 kΩ±5% ）必须通过一个可由软件控制开启的开关，从 D + 引脚连接至 3.3 V 电源（如图 7 - 1 所示 ）。

在初始连接之后，具备高速能力的收发器会在复位期间执行一种底层协议，以建立高速链路，并在相应的端口状态寄存器中指示高速运行状态。该协议在 7.1.7.5 节中有详细说明 。

高速设备接入系统时，因 D+ 上拉 1.5kΩ 电阻（协议要求的高速标识），最初会被 HOST 识别为 “全速设备” 完成基础枚举。随后，HOST 触发高速协商流程，通过 **Chirp 电流脉冲信号** 与 DEVICE 双向确认高速能力；确认完成后，系统切换到高速模式。

在高速模式下，通信采用 **电流传输模式**（区别于全速 / 低速的电压模式 ），但 **D+ 上的 1.5kΩ 上拉电阻是硬件固定设计，无需物理断开** —— 其仍参与电路，但信号驱动逻辑从 “电压上拉识别” 切换为 “电流脉冲通信”，上拉电阻作为高速协商的基础始终保持连接。
